Lesson 2：Anima 的架构关系

本节记住什么

  • 当前主线围绕 Qwen3 text encoder、AnimaTextConditioner、Cosmos transformer 与 Qwen-Image VAE 这四个公开组件展开。
  • 2B 参数量不只是标签，它和 hidden width、层数、latent token 数、分辨率预算绑在一起。

为什么重要

如果架构层搞错，prompt 机制、冻结策略、数据解释和 LoRA 设计都会连锁错位。

课程位置

这是整个课程的结构骨架。后面所有训练与评估讨论都建立在这里。

用通俗语言讲清主线

公开配置给出的最稳妥关系可以画成：

prompt
  -> Qwen tokenizer / Qwen3 text encoder
  -> AnimaTextConditioner
  -> Cosmos transformer
  -> FlowMatch scheduler
  -> Qwen-Image VAE decode
  -> image

这里最关键的两点是：

第一，公开配置把 Qwen3Model 放在主 text encoder 位置上。这样一来，文本条件、中文 prompt 与冻结策略，都应围绕 Qwen3 及其后续条件模块来理解。

第二，公开配置还显示存在独立的 AnimaTextConditioner。这意味着文本侧不是“一个 encoder 输出完就结束”，而是还有一个单独的条件重整模块，负责把语言侧表示送进视觉主干。

对 2B 参数量更稳妥的理解，不是“它小所以便宜”，而是“它落在一组受到高分辨率 token 成本约束的设计选择上”。当 latent token 数增大时，自注意力成本会按平方增长，因此 2B 很可能对应一个计算预算与表达能力之间的可用平衡点。

已公开确认

  • Qwen3Model 是公开配置中的 text encoder。
  • AnimaTextConditioner 是独立组件。
  • CosmosTransformer3DModel 是视觉主干。
  • Qwen-Image VAE 是公开 VAE 类。

解释性重构

  • text conditioner 很可能是 prompt alignment 的敏感位置。
  • 2B 更像 operating point，而不是任意参数标签。

尚未公开确认

  • text conditioner 内部究竟保留了多少 alignment 知识。
  • 原始训练中是否对 text-side 模块做过部分解冻。
  • VAE 的完整潜空间行为边界。

自测

  1. 为什么 Qwen3Model 的出现，会直接改变你对文本条件路径的理解？
  2. 为什么 text conditioner 的存在，使“冻结文本侧”这个问题变得更细，而不是更简单？
  3. 为什么分辨率一升高，参数量讨论就不再只是 P 的问题，而是 P + N 的问题？
