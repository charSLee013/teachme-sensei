自顶向下的执行路径

prompt / 输入图 -> 命令入口 -> 准备文本条件 -> 加载主模型与图像编解码器 -> 若有参考图，先编码成参考表示 -> 生成时间步与采样计划 -> 在 latent 空间里逐步去噪 -> 把 latent 解码回像素图 -> 输出最终图像
研究含义： 这个仓库的重心是推理编排，不是训练系统。它没有数据集代码、优化器配置、checkpoint 保存策略、评测循环，也没有分布式训练 plumbing。

当前被固定住的东西

Latent 形状假设

  • AutoEncoder.encode() 会把 z_channels=32 的 latent 按 ps=[2,2] 重新打包。
  • 这样一来，重排后的通道数就是 128。
  • CLI 也按 (1, 128, height // 16, width // 16) 分配初始噪声。
  • 输入在编码前会先裁成 16 的倍数。

模型侧期待

  • 这个仓库里的每个模型参数 dataclass 都把 in_channels = 128 写死了。
  • img_in 就是直接按这个契约搭出来的。
  • 所以你一旦改 latent 通道数，面对的是联动模型手术，而不只是单独替换 AE。

模型家族选择会改变什么

家族 为什么可能从这里起步 为什么它又不是“中性起点”
flux.2-klein-base-4b 官方仓库里最小的、还保留 base 路径的训练向起点。 它仍然继承 128 通道 latent 契约和 Qwen3 文本侧。
flux.2-klein-base-9b 更大的研究目标，同时保留 base 模型的灵活性。 硬件成本更高，但并不会自动补上这个仓库缺 trainer 的问题。
flux.2-dev 官方 32B 旗舰，也是最强参考 checkpoint。 硬件要求重，而且文本编码器分支换成了 Mistral。
Distilled klein 非常适合做快速推理实验。 固定 4-step / fixed-guidance 的行为，使它更像推理产物，而不是训练研究首站。

这个仓库没有提供什么

缺失的训练栈

  • 没有 dataset loader
  • 没有 optimizer 或 scheduler 配置
  • 没有 gradient accumulation 或分布式启动器
  • 没有 checkpoint 保存 / 恢复循环
  • 没有验证或 benchmark harness

这对研究意味着什么

  • 你可以用它来找契约和推理耦合点。
  • 但不能把它误认成一个现成的 VAE 或 DiT 训练框架。
  • 大多数严肃训练工作都需要第二个代码库，或一个很重的 fork。
