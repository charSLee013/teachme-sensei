FLUX.2 训练缺口清单

这页不是讲“训练应该怎么做”，而是讲“你不能把什么当成已经有了”。 目的很简单：在规划 VAE 或训练研究之前，先把官方最小推理仓库和真实训练工程分开。

总览大课 课程 0002 课程 0001 Latent 契约图
一句话版本： 这个仓库能告诉你“推理时契约如何连接”，但它不能直接告诉你“训练作业如何启动、如何喂数据、如何保存恢复、如何评估收敛”。

训练工程最常见的五类缺口

缺口 为什么重要 在本仓库里的证据
数据管线 没有数据读入、清洗、采样、批处理，就没有训练入口。 仓库公开出来的主体实现几乎都围绕推理闭环展开，看不到成型的数据读入与批处理层。
优化器与调度器 没有优化器、学习率调度器，就没有参数更新过程。 公开入口和依赖侧都看不到典型的参数更新机制，说明它并不是一个已打开的训练作业。
训练循环 没有 forward-loss-backward-step 的循环，就没有训练系统。 CLI 只做推理：加载模型、采样、解码、保存图片。
checkpoint 管理 没有保存 / 恢复机制，研究无法稳定复现。 仓库只有 Hugging Face 权重下载逻辑，没有训练中间状态保存逻辑。
验证与评测 没有验证集和指标，无法判断 VAE 或训练改动到底有没有变好。 仓库没有 tests、benchmarks、validation harness，只有最小推理脚本和展示文档。

研究上最容易踩的认知坑

  • 坑 1： 把 base 模型误认为“已经附带训练脚本”。实际上它只是更适合作为训练研究起点的 checkpoint 类型。
  • 坑 2： 把 diffusers 文档里的量化 / remote encoder 路线误认为“训练基础设施”。那是部署 workaround，不是 trainer。
  • 坑 3： 看到 autoencoder 代码就默认“VAE 微调很快能落地”。事实上只要 latent contract 一变，联动面就会迅速扩大。

对你当前使命最有用的判断

  • 如果你的目标是“先做准备”，这个仓库最适合拿来确认契约和起点，而不是直接拿来训练。
  • 如果你的目标是“尽快上手训练相关实验”，更现实的路径往往是：保留这里的契约理解，另起一个训练向代码库或大 fork。
  • 如果你想尽量少动联动面，优先考虑从 flux.2-klein-base-4b 出发，而不是先碰 distilled klein。
