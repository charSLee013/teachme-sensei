课程 0002

把训练缺口画出来

这节课的目标不是教你怎么训练 FLUX.2，而是让你不再把“最小推理仓库”误读成“差一点就能训练的仓库”。 这是做 VAE / 训练研究准备时第二个必须过的关。

和使命的关系： 你现在不是要立刻写 trainer，而是要先知道 trainer 缺了哪些骨头。只有这样，后面才不会把时间浪费在“这个仓库里明明根本没有的东西”上。

这节课只学一个技能

学会用“文件树 + 入口脚本 + 运行闭环”三板斧，快速判断一个仓库到底是推理仓库、训练仓库，还是两者兼有。

看文件树

当前仓库公开出来的主体实现几乎都围绕“加载模型、采样、解码、保存图片”这条推理闭环展开，没有成型的 dataset、trainer、checkpoint manager 一类模块。

看入口脚本

CLI 的主线是“加载模型 -> 采样 -> 解码 -> 保存图片”，而不是“取 batch -> 算 loss -> backward -> step”。

看运行闭环

这个仓库的闭环产物是一张生成图，而不是训练 checkpoint、训练日志或验证指标。

实操：用三步排除幻觉

  1. 从仓库根目录往下看，确认有没有和 dataset / trainer / checkpoint 直接相关的模块目录。
  2. 在本页底部源码附录里找到“命令入口”的那组证据，确认它的结束动作是不是“保存图片”。
  3. 再找到“模型注册与权重加载”的那组证据，确认它解决的是“如何拿到并加载权重”，而不是“如何训练并保存新权重”。
检查点： 如果一个仓库没有训练入口、没有训练循环、没有训练状态管理，那它就不是“只差一点”的训练系统。

你现在应该形成的判断

  • flux2 的价值在于帮你确认契约和推理耦合点。
  • 真正的训练工程，至少还需要数据、优化器、训练循环、checkpoint 管理、验证与指标。
  • 对你的使命来说，最合理的动作不是盲写 trainer，而是先选研究起点，再决定要不要另起代码库或重 fork。
