课程 0006

四大核心对象

如果你总是按文件树看仓库，你会很难长期记住它。更稳的方式是：先把系统固定成四个对象，再去看它们如何通过契约协作。

课程位置：这是对象层的总课。读完后你应该能不用文件名，也能解释系统由什么构成。

记住什么

  • Text Encoder 负责把 prompt 变成条件表示。
  • AutoEncoder 负责像素与 latent 表示之间的往返。
  • Sampler 负责组织时间步、token、reference 条件和 denoising loop。
  • Transformer 负责主要的 latent 去噪计算。

对象图

Text Encoder

输入是 prompt，输出是后续 transformer 可消费的条件表示。这里最容易忽视的一点是：dev 与 klein 路线的文本模型并不相同。

AutoEncoder

负责把图片压到 latent 空间，再从 latent 解回图片。它决定了通道数、空间缩放和表示习惯，因此是 VAE 研究的第一战场。

Sampler

负责把“条件”组织成实际的推理过程：token 化、位置 ID、schedule、CFG、多参考图与 KV cache 分流都在这里发生。

Transformer

负责真正的 flow matching 主干推理。这里不仅有模型规模差异，还有双流 / 单流和 KV cache 的结构路径差异。

可见证据：四对象划分是课程重构，但每个对象都直接对应当前仓库中的清晰模块：text_encoder.py、autoencoder.py、sampling.py、model.py。边界：这是为教学做的结构化解释，不是官方 README 的原始章节结构。

为什么这个对象图比文件树更好用

因为研究问题常跨文件

“如果我改 latent 表示会怎样”不是某一个文件的问题，而是 AE、sampler 和 transformer 三个对象的共同问题。

因为产品叙事常跨模块

比如 KV cache，不是只属于模型文件，也会影响 sampler 选择和模型家族理解。

因为教学必须稳定

哪怕未来文件结构调整，这四个对象依然是你解释系统的稳定骨架。

练习与检索

  1. 不用提文件名，复述四个对象的输入、输出和职责。
  2. 说出哪个对象最像“组织层”，哪个对象最像“表示层”，哪个对象最像“主干计算层”。
  3. 判断：如果你要研究 prompt upsampling，首先应该把注意力放在哪个对象上？如果要研究 latent 表示，首先该看哪个对象？

展开实现锚点
