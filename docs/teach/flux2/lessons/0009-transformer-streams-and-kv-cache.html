课程 0009

Transformer、双流与 KV Cache

这一课不试图把整个主干的每一层都讲成论文复现，而是先抓住三个真正会影响系统理解的点：规模参数、双流到合流的结构、以及 reference token 的 KV cache 路径。

课程位置：这是机制层里最重的一课。你已经知道对象与采样层后，才适合进入这里。

记住什么

  • 当前主干不是“纯图像网络”，它从一开始就在组织文本流与图像流。
  • 双流与单流的切换，是理解 reference token、文本条件和图像 latent 如何汇合的关键。
  • KV cache 路径的意义，不是抽象提速，而是“把参考图条件的重复处理挪到第一步”。

你真正需要抓住的三个结构点

结构点 1：规模参数是显式分叉

Flux2Params、Klein9BParams、Klein4BParams 分别定义不同主干规模。这不是外部元数据，而是本地实现显式写进参数类中的事实。

结构点 2：先双流，再合流

主干先让文本和图像在双流块中分别调制，再进入单流块做统一处理。这是“文本条件如何影响图像 latent”的主要结构骨架。

结构点 3：KV cache 把 reference tokens 先提出来

第一步完整跑 reference token 并抽出 KV，后续步数不再重复把 reference token 整个走一遍。

可见证据：当前仓库的主干类里有明确的 `double_blocks`、`single_blocks`、`forward_kv_extract` 和 `forward_kv_cached`。边界：本课重点是解释结构作用，不尝试逐层数学复写全部 attention 细节。

为什么 9B KV 不能被理解成“只是更快”

因为它改的是条件复用方式

真正被缓存的是 reference token 的 KV，而不是整条推理链路里所有内容。

因为它只在特定条件下触发

本地仓库里只有显式支持 `use_kv_cache` 的模型，并且在有 reference tokens 时才会走 cached 路线。

因为它改变了你解释多参考编辑性能的方式

如果你以后要比较 4B、9B、9B KV 在多参考任务上的差异，必须把这条路径单独讲出来。

练习与检索

  1. 不用看代码，解释“先双流，再合流”到底在教学上帮你理解了什么。
  2. 说清 cached 路线里“第一步做了什么，后续步做了什么”。
  3. 判断：如果没有 reference image，`9B KV` 还会体现出同样的结构优势吗？为什么？
