课程 0007

AE 与 latent 表示

如果你的研究目标和 VAE 有关，这一课是整个教学包里最关键的机制课。它会把“AE 不只是压缩一下图片”这句话讲清楚。

课程位置：这是从四对象进入表示层的第一课。它把 0001 的契约警告展开成具体机制。

记住什么

  • AE 决定的不只是重建质量，还决定 downstream 模块要接受怎样的 latent 表示。
  • 当前仓库里的 128 通道不是一个随意数字，而是由 AE 参数与重排规则共同产生的。
  • 如果你修改 latent 表示，最先动手的是 AE，但最先出问题的不一定只在 AE 内部。

AE 在当前仓库里做了三件事

1. 压缩到基础 latent

encoder 输出的是更基础的空间表示，尚未等于后面主干直接消费的最终 latent 形状。

2. 重新打包通道

encode() 会把基础表示按固定 patch 规则重排，这一步把 z_channels=32 变成后续需要的 128 通道。

3. 归一化 / 反归一化

AE 不只是编码和解码，中间还有 BatchNorm 统计参与的 normalize / inv_normalize 约定。

可见证据：本课的关键事实来自 AutoEncoder.encode() 与 decode() 的实现，以及参数中的 z_channels=32 与 ps=[2,2]。边界：“它因此会联动整个系统”属于迁移判断，不是 AE 文件单独直接宣称。

为什么 VAE 改动会外溢

改通道数

如果最终 latent 通道不再是 128，transformer 的输入投影就会首先失去匹配。

改空间缩放

如果像素到 latent 的缩放比例不再与 /16 对齐，CLI 构造噪声与 reference image 路径都会错位。

改归一化统计

哪怕通道数和网格大小不变，latent 分布改变也会影响采样时的稳定性与主干期望。

研究迁移：什么时候可以把改动视为“局部”

  1. 最终 latent 仍保持 128 通道。
  2. 像素到 latent 的总缩放仍与 CLI 假设兼容。
  3. token 化前的网格组织仍能被当前 sampler 正确展平。
一个安全启发：如果你此刻的研究重点是“先验证表示学习方向值不值得”，优先找“不破契约”的改动入口，而不是一上来就重定义 latent 形状。

练习与检索

  1. 不用看源码，说清 128 通道是怎样由更基础 latent 通过重排得到的。
  2. 判断一个改动：如果你只换 AE 的 loss，但最终 latent 形状、缩放和归一化接口都不变，这更像局部改动还是联动改动？
  3. 判断另一个改动：如果你把 latent 通道改成 192，即便重建质量更好，为什么也不能说“我只改了 VAE”？
