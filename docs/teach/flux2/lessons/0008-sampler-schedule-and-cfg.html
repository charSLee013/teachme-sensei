课程 0008

Sampler、schedule 与 CFG

这一课讲清一件事：模型主干不会自己决定如何被调用，真正把条件、步数、位置 ID 和参考图约束组织成一条推理过程的是 sampler 层。

课程位置：这是从“对象图”进入“过程图”的第一课。它解释为什么很多研究改动其实会先撞上采样层约束。

记住什么

  • sampler 不只是一个 for-loop；它定义了文本、图像、reference 和时间步怎么被组织成主干输入。
  • 当前仓库里有三条关键 denoising 路线：denoise、denoise_cached、denoise_cfg。
  • Distilled 和 Base 的差别，在 sampler 层是直接可见的，而不只是产品定位。

采样层的四个职责

组织 token

把图像 latent 与文本条件分别转换成可被主干消费的序列，并为它们构造位置 ID。

构造时间步

get_schedule() 根据步数和图像序列长度生成时间步表，决定 denoising loop 如何前进。

插入 reference 条件

参考图不是直接送给主干，而是先编码、展平、打时间偏移，再并入条件序列。

选择具体 denoise 路线

Base 用 CFG 路线；distilled 用普通 denoise；若模型支持且有 reference tokens，则可走 cached 路线。

可见证据：这些职责都能在 sampling.py 中找到明确函数边界：`prc_img`、`prc_txt`、`get_schedule`、`encode_image_refs`、`denoise*`。边界：这是对职责的课程重组，不是文件原始注释的逐句转写。

为什么 schedule 和 CFG 不能被当成外围细节

因为它们决定“如何调用模型”

同一个主干，如果采样步数和 guidance 组织方式不同，体验出来的模型就不是同一条产品路径。

因为 Base / distilled 的差别在这里显式出现

Base 路线需要 CFG 双路条件；distilled 路线则直接按固定 guidance / steps 跑。

因为 reference image 的组织在这里完成

如果你以后研究多参考编辑，sampler 是第一批必须深入理解的地方。

练习与检索

  1. 说出三条 denoise 路线各自服务哪类模型或条件。
  2. 解释为什么“改 sampler 默认步数”并不是一个纯外部脚本层的小改动。
  3. 复述 reference image 从输入图到 reference tokens 的大致过程。
