课程 0005

CLI 与端到端主流程

这一课只回答一个问题：当前仓库里，用户输入是怎样一路被编排成最终输出图像的。你应该从这里形成系统级时间顺序，而不是先扎进某个模块里。

课程位置：这是机制层的第一课，介于总览入口与四大对象之间。读完后你应该能画出推理闭环。

记住什么

  • 真正把仓库串起来的是 scripts/cli.py，不是某个 model 文件。
  • CLI 的职责不是“顺手跑一下模型”，而是协调模型选择、权重加载、文本编码、参考图编码、采样、解码和保存。
  • 如果你以后要做训练向 fork，CLI 里的很多判断会迁移成更正式的 pipeline 组织层。

主流程时间线

1. 选择模型
从注册表里选 `klein / base / dev` 路线，并据此决定默认 steps、guidance、文本编码器与是否有 KV cache 分支。
2. 加载组件
分别加载 text encoder、flow model、AE。这里已经把“系统”拆成三个不同重量级组件。
3. 解析输入
解析 prompt、尺寸、输入图、match size、upsampling 选项与 seed。
4. 文本编码
把 prompt 变成条件表示。对 Base 路线，还要组织空 prompt 和正向 prompt 的 CFG 双路条件。
5. 参考图编码
若有输入图，先经 AE 编码，再打包成 reference tokens 与对应位置 ID。
6. 构造噪声 latent
按 (1, 128, height // 16, width // 16) 直接分配初始噪声。
7. 进入 denoising loop
根据模型家族选择 `denoise` / `denoise_cached` / `denoise_cfg`。
8. 解码与保存
把最终 latent reshape 回网格，经 AE decode 得到图像，再做输出检查与 EXIF 保存。
可见证据：这是对入口脚本行为的教学重构。直接证据来自 scripts/cli.py 中的组件加载、ctx 构造、噪声分配、denoise 分流与 AE decode。边界：这描述的是当前 CLI 主入口，不是官方全部部署系统。

为什么这一层对研究准备很重要

它告诉你“哪里是组织层”

很多读者一开始会误以为训练向 fork 的主要工作是去改 transformer 主干。实际上，真正先要学会的是：哪些判断属于 orchestration 层，哪些才属于模型层。

它暴露了跨模块依赖

比如 latent 形状不是只在 AE 内部出现。它会同时出现在入口噪声分配、reference image 编码、token 化和主干输入里。

它决定“训练栈缺口”怎么被识别

你一旦能画出当前推理闭环，就会更容易看到真正缺失的是数据、loss、优化器、评估和实验编排，而不是“再多几个 helper function”。

练习与检索

  1. 不用看源码，尝试复述“从 prompt 到 output image”的 8 步时间线。
  2. 指出其中哪一步第一次显式暴露了 latent 的 128 通道契约。
  3. 指出哪一步开始分化出 `distilled / base / KV cached` 三条不同路径。
