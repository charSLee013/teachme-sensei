课程 0010

模型家族、许可与边界

这一课专门解决一种常见混淆：把官方整个 FLUX.2 产品家族、当前开源仓库支持的模型、以及 Hugging Face 上的开放程度混成一团。

课程位置：这是机制层之后的“边界课”。它让你重新校准你以为自己已经懂了的模型家族。

记住什么

  • 本地开源仓库的核心公开推理对象是 klein 家族和 dev，不是完整 FLUX.2 产品家族的全部实现。
  • 4B / 9B / 32B 是官方公开参数名义，但系统里还可能包含独立文本编码器。
  • 许可、gated 状态和“能否公开读完整 model card”是相关但不完全等价的三个问题。

最小可用矩阵

模型公开定位许可 / gated适合作什么
`klein 4B`最快、消费级 GPU、开放权重Apache 2.0 / ungated本地部署、实时应用、低门槛实验

可见证据：这一表综合官方 README、官方 docs、公开 model card 与 Hugging Face API 元数据。边界：gated 模型卡正文当前并非全部公开可读，因此某些细节只能记录为元数据层事实。

为什么“9B”不等于“整个系统只有 9B”

官方博客公开写过：[klein] 9B 是 9B flow model with 8B Qwen3 text embedder。这条信息很重要，因为它告诉你：

  1. 产品命名通常先指主干规模。
  2. 系统里还可能包含独立文本编码器。
  3. 讨论参数量时，必须说清你在说“主干名义规模”还是“更完整的系统组成”。

研究决策上的直接后果

  1. 如果你最怕被许可卡住，先看 4B 路线。
  2. 如果你最怕蒸馏带来的采样限制，先看 Base 路线。
  3. 如果你追求更接近高质量研究路径、又能接受更重硬件和 gated 约束，再考虑 dev。
