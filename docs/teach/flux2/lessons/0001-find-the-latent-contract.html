课程 0001

找到 Latent Contract

如果你想围绕 FLUX.2 做 VAE 或训练研究，第一件事不是先补更多理论，而是先知道这个仓库里哪些部件被悄悄绑在一起了。这节课就是给你这张图。

课程总地图 返回总览大课 术语表 Latent 契约图
和使命的关系： 一个 VAE 实验只有在保持 latent contract 不变时，才算“局部改动”。只要你动了 latent 通道数、空间缩放比例或 token 布局，transformer 和 sampler 就会一起卷进来。

记住什么

  • 128 通道只是表面现象；真正重要的是它背后代表的共享契约。
  • “先看 klein-base-4b”是研究起步建议，不是官方写死的唯一路线。
  • 一旦你改了通道数、缩放倍率或 token 组织，就不要再把问题叫成“只改 VAE”。

为什么重要

这页决定你后面对 VAE 改动风险的判断方式。如果这里没站稳，你会不断把系统级联动误判成局部实验，把研究准备误判成“只差一点就能训练”。

可见证据： 本地实现直接支持三件事：AE 的重排会形成 128 通道、主模型参数把 in_channels=128 设为前提、CLI 会按 (1, 128, height // 16, width // 16) 创建初始噪声。 边界： “因此默认先看 klein-base-4b”属于工程迁移建议，不是官方产品页逐字结论。 置信度： 对共享契约本身为高，对起步模型推荐为中高。

这节课只学一件事

在这个仓库里，autoencoder、sampler 和 transformer 通过一个非常具体的 latent 形状耦合在一起： 128 个通道、与 height // 16 / width // 16 兼容的空间缩放，以及会把输入裁成 16 的倍数的预处理。

事实 1：AE 会把 latent 打包成固定形状

AutoEncoder.encode() 会取 encoder mean，再按 ps=[2,2] 重排。 因为 z_channels=32，重排后正好得到 128 个通道。

事实 2：主模型已经把这个形状当成前提

所有参数 dataclass 都把 in_channels = 128 写死了。transformer 的图像输入投影就是按这个假设搭的。

事实 3：采样路径也已经默认接受这套契约

CLI 会按 (1, 128, height // 16, width // 16) 生成噪声，而预处理会先把图片中心裁到 16 的倍数再编码。

练习：判断一个改动是局部还是联动

  1. 在本页底部源码附录里先找到“AE 如何把基础 latent 重排成最终通道数”的证据，写出数字 128 是怎么来的。
  2. 再找到“主模型把多少通道当成输入前提”的证据，确认 transformer 是否同意这个数字。
  3. 最后找到“入口层如何分配初始噪声”的证据，确认 sampling 侧是不是也同意。
检查点： 这三处只要有一处不再一致，你面对的就不再是“只改 AE”的实验。

快速小测

1. 如果你把 AE 输出从 128 通道改成 192，但别的都不动，最先坏的是哪一侧？

2. 如果通道数还是 128，但 AE 不再匹配 /16 的空间缩放，你必须回看哪一侧？

3. 这个仓库里，哪一类模型最适合作为训练向实验的第一站？

这节课应该改变你脑中的三件事

  • 不要把这个仓库描述成训练代码库。它是一个带强 latent 形状假设的最小推理仓库。
  • 做 VAE 工作前，先问两个问题：“通道数还保持 128 吗？”、“/16 网格契约还完整吗？”
  • 任何一个问题答案是“否”，都要按 AE、sampler、transformer 接口联动来规划。
