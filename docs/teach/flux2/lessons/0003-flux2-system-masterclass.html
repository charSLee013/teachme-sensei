一、先抓住一句话：这是什么仓库

如果你现在只记三句话，先记这三句

  • 第一句：这不是训练仓库，而是把公开权重跑起来的最小推理仓库。
  • 第二句：它最值得你研究的，不是“训练技巧”，而是整个系统共同依赖的 latent contract。
  • 第三句：如果你现在是为 VAE / 训练研究做准备，默认先从不破契约的入口开始，通常先看 klein-base-4b，不要一上来就造 trainer。

先别急着看公式。你第一步只需要记住一句话： 官方 `flux2` 仓库是一个最小推理仓库。 它负责把已经训练好的 FLUX.2 开放权重跑起来，完成图像生成、单参考编辑和多参考编辑。 它擅长的是“把 checkpoint 接成一个可运行系统”，而不是“把训练流程完整公开给你”。

它能帮你做什么

  • 确认真实入口在哪里
  • 看懂 autoencoder、sampler、transformer、text encoder 如何接起来
  • 判断 latent contract 到底固定了哪些条件
  • 分清 `klein`、`base`、`dev` 的研究含义

它不能直接替你做什么

  • 不会替你提供 dataset 管线
  • 不会替你提供 optimizer / scheduler / loss loop
  • 不会替你提供 checkpoint 保存与恢复系统
  • 不会自动变成可训练的 VAE / DiT 研究框架
你现在只要会做的判断： 如果一个仓库的主闭环是“加载权重 -> 采样 -> 解码 -> 保存图片”，那它的重心就是推理，不是训练。
为什么这件事会直接影响你的研究路线： 因为一旦你接受“它不是训练仓库”，你就不会再幻想只靠在这里补几段脚本就能自然长出完整 trainer。 你会更现实地把这里当成契约地图，然后决定哪些研究能在这张地图里先做，哪些研究必须去另起训练向代码库。

二、系统总览：四大部件是怎么协作的

你可以先把 FLUX.2 想成四个必须同时站住的部件： 文本侧负责把 prompt 变成模型能用的上下文， transformer负责在 latent 空间里逐步更新图像表示， sampler负责决定每一步怎么往前走， autoencoder负责把像素世界和 latent 世界接起来。

共同工作区：latent 空间
文本编码器 把 prompt 压成模型可用的上下文条件。
FLUX.2 Transformer 在 latent 空间里逐步把噪声推向目标图像。
Sampler 决定时间步、guidance 和每一步怎么往前走。
Autoencoder 连接像素世界与 latent 世界，负责编码与解码。

怎么看这张图

这只是一个帮助你建立空间感的局部 3D 模块。真正的学习主线仍然在正文。你完全可以先忽略它，等读完整节再回来确认关系。

部件 它负责什么 如果你动它，会影响哪类研究判断
Text encoder 把 prompt 变成上下文表示；不同模型家族用的文本编码器并不一样。 影响文本侧契约与 prompt 理解质量，不一定会动 latent 几何。
Transformer 在 latent 空间里逐步把噪声推向目标图像。 一旦 latent 通道数或 token 组织改了，它通常会被迫联动。
Sampler 定义每一步更新顺序、时间步、CFG 或 distilled guidance 的行为。 影响推理路径，也影响你对“哪个模型更像训练起点”的判断。
Autoencoder 负责像素与 latent 的往返映射。 是 VAE 研究最自然的切口，但只有不破坏契约时才可能局部推进。

三、模型家族：为什么 `base` 和 distilled 不能混着理解

初学时最容易犯的错，是把 `klein`、`base`、`dev` 当成“只是大小不同的 checkpoint”。 这会直接导致研究路线判断失真。你真正要看的不是参数量本身，而是： 它是更像推理产物，还是更像训练起点？

家族 直觉定位 推理特征 对训练研究意味着什么
distilled klein 实时、轻、为“很快跑出来”服务 固定 4-step，固定 guidance 行为 非常适合看推理成品，但不是最干净的训练思维起点
klein base 保留 base 路径的较小模型家族 50-step，约束更少 更像训练准备入口，尤其是 `klein-base-4b`
flux.2-dev 官方 32B 旗舰参考 质量高、硬件重、文本侧不同 是强参考，不是低摩擦首站
停一下，自己复述： 如果你的目标是“先为训练研究做准备，而不是先做最快推理”，那你脑中更自然的默认起点应该是谁？ 这节课希望你答出：`klein-base-4b` 往往是更稳的第一站，因为它既保留了 base 路径，又把体量压在更能承受的范围里。
可见证据： 当前仓库注册表把 Base 路线和 distilled 路线的 sampler 合约明确分开；官方 docs 和公开 model card 也把 Base 变体定位为 fine-tuning、LoRA、研究和自定义 pipeline。 边界： “更稳的第一站”是面向当前教学任务的工程迁移判断，不代表其他路线必然错误。 置信度： 中高。

四、为什么 latent contract 是整页最重要的概念

当你说“我想研究 VAE”时，你真正要问的不是“能不能改 autoencoder”，而是： 我改动之后，还保不保持整个系统对 latent 的共同约定？ 这就是 latent contract。

通道数契约

当前系统把 latent 当成 128 通道来理解。只要这个数字变了，transformer 的输入假设就会跟着出事。

空间缩放契约

当前 latent 网格和输入宽高之间有稳定的缩放关系。你若打破它，sampler 的噪声形状也必须跟着变。

token 组织契约

参考图 token、文本 token、图像 token 的组织方式不是随便排的。它们决定了编辑、多参考、KV cache 等路径能否成立。

第一次读到这里，先停一下。 你此刻只需要带走三件事：
  • 只要保持 128 通道、网格关系和 token 组织不动，你更可能还在局部改动范围里。
  • 一旦动到这三层契约，问题就不再只是“改一下 AE”。
  • 如果你还没法顺口复述这三层契约，就不要急着进入训练向 fork。
第二遍再读：token 组织例子、最短术语和三个公式

一个具体例子：为什么 token 组织契约不是抽象口号

以“单参考图编辑”为例，系统至少要同时处理三类东西： 文本条件、参考图表示、待更新的目标图像 latent。 它们不是随便堆在一起，而是按约定的角色和顺序进入同一条推理路径。

第 1 步：文本先给方向

prompt 先告诉系统你想往什么方向改图，它提供的是“语义目标”。

第 2 步：参考图给约束

参考图编码后的 token 提供“你要保留或借用什么”的图像条件，它不是普通噪声。

第 3 步：目标 latent 才是被更新对象

真正在去噪过程中被一步步改写的是目标图像 latent；如果你把前两类 token 的位置、数量或缓存方式打乱，编辑路径就可能直接失效。

所以“token 组织契约”真正保护的，不只是一个形状，而是 谁在提供条件、谁在提供参考、谁才是被更新对象。 这也是为什么它会影响多参考编辑和 KV cache 路径。

别被这些词先吓到。 当前阶段你只需要抓住下面这 4 句最短解释：
  • CFG：一种把结果往条件方向拉近的推理机制。
  • KV cache：把已经算过的注意力结果缓存起来，减少重复计算。
  • distilled guidance：蒸馏模型里被一并固化进去的 guidance 行为。
  • Qwen3 / Mistral：这里只是不同模型家族所依赖的文本侧路线，不是你此刻必须深挖的主线。

公式 1：为什么会出现 128 通道

\[ z = \mathrm{rearrange}\left(\mu,\; c \times (i \cdot p_i) \times (j \cdot p_j) \rightarrow (c \cdot p_i \cdot p_j) \times i \times j\right) \]

白话解释：autoencoder 会先得到一个 latent mean，然后把空间维的一部分重新打包进通道维。 在当前实现里，基础 latent 通道是 32，而重排因子是 \(2 \times 2\)，所以最后进入主系统的通道数是 \(32 \times 2 \times 2 = 128\)。

这对研究的含义是：如果你只改 AE 内部细节，但最后仍然吐出同样的 128 通道与相同网格关系，你的改动更可能停留在局部；一旦这个数字变了，就进入联动改造。

公式 2：采样每一步在做什么

\[ x_{t_{\text{prev}}} = x_{t_{\text{curr}}} + (t_{\text{prev}} - t_{\text{curr}})\,\hat{\epsilon}_\theta(x, t, c) \]

白话解释：sampler 并不是“一次生成整张图”，而是在 latent 空间里按时间步一点点更新图像表示。 所谓 `num_steps`，就是你给这种更新多少次机会。

这对研究的含义是：distilled 模型把这个路径压得更短、更固定，所以它更像“为推理打磨后的成品”；base 模型保留的自由度更像训练思维。

公式 3：CFG 的直觉

\[ \hat{\epsilon}_{\text{cfg}} = \hat{\epsilon}_{u} + s \cdot (\hat{\epsilon}_{c} - \hat{\epsilon}_{u}) \]

这里的 \(\hat{\epsilon}_u\) 是无条件预测，\(\hat{\epsilon}_c\) 是有条件预测，\(s\) 是 guidance 强度。 你不用先背它的推导，只需要记住： 这代表一种“把模型往条件方向拉近”的机制。

这也解释了为什么“guidance 固定住了没有”会成为区分 base 与 distilled 的重要线索。

五、为什么这个仓库不能直接当训练仓库

讲到这里，你应该已经能看懂系统的主要部件了。下一步最重要的不是更深入推理，而是防止自己产生幻觉： “既然这里有 model、sampler、autoencoder，那是不是离 trainer 就差一点？” 不是。

训练系统必须有的东西 为什么它重要 当前官方仓库的状态
数据管线 没有数据读入、清洗、批处理，就没有训练入口。 没有提供公开的数据训练入口。
优化器 / scheduler 没有参数更新机制，训练不会发生。 当前重心是加载现成权重，并未暴露训练循环。
loss loop 没有 forward-loss-backward-step 的持续过程，就不是 trainer。 当前闭环是推理、采样、解码、保存图片。
checkpoint 管理 研究需要保存、恢复、比较不同阶段。 这里只有下载与加载公开权重的路径。
验证与评测 没有指标，就无法判断改动究竟变好还是变坏。 仓库没有公开训练验证框架。
你现在只要会说这句话就够了： “这个仓库非常适合确认契约与推理耦合点，但如果我要做真正训练研究，我还需要另一个训练向代码库，或者一个很重的 fork。”
