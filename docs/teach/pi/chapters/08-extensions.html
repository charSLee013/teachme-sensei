第 8 章

扩展系统 — 插件架构详解

理解扩展的发现、加载、运行全流程

Main Line

扩展系统是 Pi 把核心保持精简的主要手段。它把“给系统加能力”的需求从核心包中抽离出来，交给 hook、工具注册和 UI 扩展机制。

Remember Now

不要把扩展理解成补丁脚本。它们是有生命周期、命令、工具和 UI 注入点的正式运行时参与者。

本章是最强大的功能章节。Pi 的扩展系统让它从一个编码助手变成了一个可扩展的平台。现有仓库和用户目录机制展示了这个系统能承载工具、命令、事件和 UI 扩展。

8.1 扩展是什么？

Extension 是一个 TypeScript 模块，可以：

  • 🔗 订阅生命周期事件：agent_start、turn_end、message_update...
  • 🔧 注册自定义工具：让 LLM 可以调用你的函数
  • ⌨️ 注册命令：/mycommand 斜杠命令
  • 🖥️ 注册 UI 组件：自定义编辑器、widget、消息渲染器
  • 📡 拦截请求：修改发送给 LLM 的 payload

类比：Pi 的扩展就像 VS Code 的扩展、浏览器的插件 — 它们在不修改核心代码的情况下增强功能。

8.2 扩展发现与加载

发现机制

// core/extensions/loader.ts
export async function discoverAndLoadExtensions(): Promise<LoadExtensionsResult> {
  const dirs = [
    getAgentDir() + "/extensions",    // ~/.pi/extensions/
    projectDir + "/.pi/extensions",   // 项目级扩展
  ];

  for (const dir of dirs) {
    if (!existsSync(dir)) continue;
    const files = readdirSync(dir);
    for (const file of files) {
      if (file.endsWith(".ts") || file.endsWith(".js")) {
        await loadExtensionFromFile(join(dir, file));
      }
    }
  }
}

jiti 运行时 TypeScript 加载

import { createJiti } from "jiti/static";

const jiti = createJiti(import.meta.url, {
  // 别名映射：扩展可以用 @earendil-works/pi-xxx 导入
  aliases: {
    "@earendil-works/pi-ai": piAiEntry,
    "@earendil-works/pi-agent-core": piAgentCoreEntry,
    "@earendil-works/pi-tui": piTuiEntry,
    "@earendil-works/pi-coding-agent": piCodingAgentEntry,
  },
  // Bun 二进制模式的虚拟模块
  virtualModules: VIRTUAL_MODULES,
});

// 加载扩展模块
const extensionFactory = jiti(filePath).default;
jiti 的作用：让扩展可以用 TypeScript 编写，运行时自动编译。不需要预编译步骤。同时支持别名映射，让扩展用 @earendil-works/pi-xxx 导入 Pi 的包。

VIRTUAL_MODULES — Bun 二进制模式的技巧

const VIRTUAL_MODULES: Record<string, unknown> = {
  "typebox": _bundledTypebox,
  "@earendil-works/pi-agent-core": _bundledPiAgentCore,
  "@earendil-works/pi-tui": _bundledPiTui,
  "@earendil-works/pi-ai": _bundledPiAiCompat,
  "@earendil-works/pi-coding-agent": _bundledPiCodingAgent,
  // ... 旧包名别名
};

当 Pi 编译为 Bun 二进制时，扩展无法通过 node_modules 解析依赖。VIRTUAL_MODULES 让 jiti 在运行时注入预打包的模块。

8.3 ExtensionFactory 模式

// 扩展工厂函数
export type ExtensionFactory = (context: ExtensionContext) => Extension | Promise<Extension>;

// Extension 定义
export interface Extension {
  name: string;
  description?: string;

  // 钩子函数
  onAgentStart?: (event: AgentStartEvent) => Promise<void>;
  onAgentEnd?: (event: AgentEndEvent) => Promise<void>;
  onTurnEnd?: (event: TurnEndEvent) => Promise<void>;
  onBashToolCall?: (event: BashToolCallEvent) => Promise<void>;
  onEditToolCall?: (event: EditToolCallEvent) => Promise<void>;
  onMessageUpdate?: (event: MessageUpdateEvent) => Promise<void>;
  onInput?: (event: InputEvent) => Promise<InputEventResult | void>;
  onBeforeProviderRequest?: (event: BeforeProviderRequestEvent) => Promise<BeforeProviderRequestEventResult>;

  // 注册自定义内容
  tools?: RegisteredTool[];
  commands?: RegisteredCommand[];
  keybindings?: AppKeybinding[];
  widgets?: ExtensionWidgetOptions[];
  messageRenderers?: MessageRenderer[];
}

ExtensionContext — 工厂接收的上下文

export interface ExtensionContext {
  // 扩展自身的配置
  config?: Record<string, unknown>;

  // 运行时 API（通过 ExtensionRuntime 暴露）
  runtime: ExtensionRuntime;

  // 日志
  log: (message: string) => void;
  error: (message: string) => void;
}

8.4 事件系统 — 核心！

Pi 的扩展系统围绕一个丰富的事件系统构建。以下是完整的事件流：

事件分类

类别 事件 触发时机
Agent 生命周期 agent_start, agent_end 整个会话开始/结束
Turn 生命周期 turn_start, turn_end 每一轮（assistant response + tools）
消息事件 message_start, message_update, message_end 消息流式传输过程中
工具事件 bash_tool_call, edit_tool_call, *_tool_result 工具调用前后
会话事件 session_compact, session_fork, session_switch 会话状态变化时
输入事件 input 用户输入时（可拦截/修改）
Provider 事件 before_provider_request, after_provider_response LLM 请求前后（可修改 payload）

8.5 示例扩展走读

hello.ts — 最小扩展

// examples/extensions/hello.ts
import type { ExtensionFactory } from "@earendil-works/pi-coding-agent";

const helloExtension: ExtensionFactory = (context) => {
  return {
    name: "hello",
    description: "A simple hello world extension",

    async onAgentStart(event) {
      context.log("Agent started!");
    },

    async onBashToolCall(event) {
      context.log(`Tool called: ${event.toolName}`);
    },
  };
};

export default helloExtension;

git-checkpoint.ts — 中等复杂度

// examples/extensions/git-checkpoint.ts
// 自动在每次 turn 结束时创建 git checkpoint

const gitCheckpointExtension: ExtensionFactory = (context) => {
  return {
    name: "git-checkpoint",
    description: "Auto-create git checkpoints after each turn",

    async onTurnEnd(event) {
      const { message, toolResults } = event;

      // 如果有文件修改，创建 git commit
      const hasFileChanges = toolResults.some(
        r => r.toolName === "write" || r.toolName === "edit"
      );

      if (hasFileChanges) {
        const exec = context.runtime.exec;
        await exec("git add -A");
        await exec(`git commit -m "checkpoint: turn ${message.timestamp}"`);
      }
    },
  };
};

8.6 自定义工具注册

import { defineTool, Type } from "@earendil-works/pi-coding-agent";

const myToolsExtension: ExtensionFactory = (context) => {
  return {
    name: "my-tools",
    tools: [
      defineTool({
        name: "calculator",
        description: "Evaluate a math expression",
        parameters: Type.Object({
          expression: Type.String({ description: "Math expression, e.g. '2 + 2'" }),
        }),
        async execute(toolCallId, params) {
          try {
            const result = eval(params.expression); // 注意：生产环境不要这样！
            return {
              content: [{ type: "text", text: `Result: ${result}` }],
              details: { expression: params.expression, result },
            };
          } catch (e) {
            return {
              content: [{ type: "text", text: `Error: ${e.message}` }],
              details: { error: e.message },
            };
          }
        },
      }),
    ],
  };
};

8.8 小测验

Q1: 扩展不能在以下哪个时机执行代码？

A. Agent 启动时
B. 每个工具调用后
C. Pi 编译为二进制时
D. 用户输入时
✅ 正确答案：C。扩展在 Pi 运行时加载，不能在编译时执行。

Q2: jiti 在扩展系统中的作用是什么？

A. 打包扩展为二进制
B. 运行时编译 TypeScript 扩展
C. 验证扩展的类型安全
D. 发布扩展
✅ 正确答案：B。jiti 是一个运行时 TypeScript 编译器，让扩展不需要预编译。

Q3: VIRTUAL_MODULES 解决了什么问题？

A. 让扩展可以互相依赖
B. 让 Bun 二进制中的扩展能解析 Pi 的包
C. 让扩展可以动态加载
D. 让扩展可以热重载
✅ 正确答案：B。Bun 二进制没有 node_modules，VIRTUAL_MODULES 在运行时注入预打包的模块。

Q4: before_provider_request 钩子可以做什么？

A. 取消 agent 执行
B. 修改发送给 LLM 的 payload
C. 修改 LLM 的响应
D. 切换 LLM 提供商
✅ 正确答案：B。before_provider_request 在 LLM 请求发送前触发，可以检查或修改 payload。

Q5: 扩展可以注册以下哪项？

A. 自定义工具
B. 斜杠命令
C. UI Widget
D. 以上全部
✅ 正确答案：D。扩展可以注册工具、命令、Widget、消息渲染器、键绑定等。

8.9 练习

写一个自定义扩展，注册一个 /hello 命令：

// ~/.pi/extensions/hello.ts
import type { ExtensionFactory } from "@earendil-works/pi-coding-agent";

const factory: ExtensionFactory = (ctx) => ({
  name: "hello-ext",
  commands: [
    {
      name: "/hello",
      description: "Say hello",
      execute: async (cmdCtx) => {
        cmdCtx.notify("Hello from extension! 👋");
      },
    },
  ],

  async onAgentStart() {
    ctx.log("Hello extension loaded!");
  },
});

export default factory;

保存后运行 pi，输入 /hello 测试。
