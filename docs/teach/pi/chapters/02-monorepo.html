第 2 章

Monorepo 架构 — 5 个包如何协作

理解整体项目结构和包之间的依赖关系

Main Line

本章要把“Pi 是一个工具”拆成“Pi 是五个包如何协作”。只有把 workspace 边界和依赖方向看清，后面各章的实现细节才不会漂浮。

Remember Now

这套仓库当前用的是 npm workspaces。遇到旧的 pnpm 或 Turbo 心智时，要回到根 `package.json`、`package-lock.json` 和脚本定义重新核对。

2.1 为什么是 Monorepo？

Pi 使用 npm workspaces 管理 monorepo。这种架构的优势：

  • 共享类型：所有包共享 TypeScript 类型定义，避免重复定义
  • 协调发布：所有包共享一个版本号（lockstep versioning），简化跨包兼容性
  • 统一 CI：一个仓库 = 一套 CI/CD 配置
  • 原子变更：跨包修改可以在一个 PR 中完成

2.4 npm workspaces 配置

根目录的 package.json 定义了 workspaces：

{
  "name": "pi-monorepo",
  "workspaces": [
    "packages/*",
    "packages/coding-agent/examples/extensions/with-deps",
    "packages/coding-agent/examples/extensions/custom-provider-anthropic",
    "packages/coding-agent/examples/extensions/custom-provider-gitlab-duo",
    "packages/coding-agent/examples/extensions/sandbox",
    "packages/coding-agent/examples/extensions/gondolin"
  ]
}

packages/* 自动包含 5 个主要包。额外的 6 个 workspace 是带依赖的示例扩展，它们作为独立的 npm 包来演示扩展的依赖管理机制。

2.5 TypeScript 配置

Pi 使用两层 TypeScript 配置：

tsconfig.base.json — 共享编译器选项

{
  "compilerOptions": {
    "target": "ES2022",
    "module": "Node16",
    "strict": true,
    "erasableSyntaxOnly": true,
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true,
    "allowImportingTsExtensions": true,
    "rewriteRelativeImportExtensions": true
  }
}
erasableSyntaxOnly：这是 Pi 的关键约束！它强制使用"可擦除"的 TypeScript 语法 — 编译后不产生额外的 JS 代码。禁止使用 enum、namespace、import =、参数属性等需要 JS emit 的语法。这保证了代码可以在 Node strip-only 模式下运行。

tsconfig.json — 路径映射

{
  "extends": "./tsconfig.base.json",
  "compilerOptions": {
    "paths": {
      "@earendil-works/pi-ai": ["./packages/ai/src/index.ts"],
      "@earendil-works/pi-agent-core": ["./packages/agent/src/index.ts"],
      "@earendil-works/pi-coding-agent": ["./packages/coding-agent/src/index.ts"],
      "@earendil-works/pi-tui": ["./packages/tui/src/index.ts"],
      "@earendil-works/pi-orchestrator": ["./packages/orchestrator/src/index.ts"]
    }
  }
}

路径映射让所有包在开发时可以直接引用彼此的源码（而不是 dist），实现热更新和类型检查。

2.6 构建顺序

Pi 的构建脚本按依赖顺序编译：

"build": "cd packages/tui && npm run build && cd ../ai && npm run build && cd ../agent && npm run build && cd ../coding-agent && npm run build && cd ../orchestrator && npm run build"

为什么是这个顺序？因为：

  1. tui — 无依赖，最先构建
  2. ai — 无依赖（仅外部 SDK），第二个构建
  3. agent — 依赖 ai，第三个构建
  4. coding-agent — 依赖 ai、agent、tui，第四个构建
  5. orchestrator — 依赖 coding-agent，最后构建

2.7 锁步版本策略

所有包共享一个版本号（当前 v0.80.3）。发布时：

# 升级所有包的 patch 版本
npm run version:patch

# 升级所有包的 minor 版本
npm run version:minor

scripts/sync-versions.js 确保所有 package.json 中的版本号一致。这简化了跨包兼容性管理 — 你永远不需要担心 coding-agent@0.80 和 agent-core@0.79 是否兼容。

2.9 小测验

Q1: 为什么 pi-agent-core 必须在 pi-ai 之后构建？

A. pi-ai 依赖 pi-agent-core
B. pi-agent-core 依赖 pi-ai 的类型和 API
C. 没有特定顺序，这只是惯例
D. pi-agent-core 的代码生成器需要 pi-ai
✅ 正确答案：B。pi-agent-core 导入 pi-ai 的类型和函数，所以必须先编译 pi-ai。

Q2: erasableSyntaxOnly 禁止什么？

A. 使用 interface
B. 使用 enum 和 namespace
C. 使用 type 关键字
D. 使用泛型
✅ 正确答案：B。erasableSyntaxOnly 禁止需要 JS emit 的语法（如 enum、namespace、import =、参数属性），强制纯类型级别的 TypeScript。

Q3: 锁步版本策略的好处是什么？

A. 减少 npm 发布次数
B. 简化跨包兼容性管理
C. 减小包体积
D. 提高构建速度
✅ 正确答案：B。所有包共享一个版本，你永远不需要担心包之间的版本兼容问题。

Q4: 根 package.json 中 include 字段的作用是？

A. 指定 TypeScript 类型检查的范围
B. 指定 npm 发布时包含的文件
C. 指定工作区
D. 指定依赖
✅ 正确答案：A。include 指定哪些文件参与 TypeScript 类型检查。
