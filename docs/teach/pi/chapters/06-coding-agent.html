Chapter 6: pi-coding-agent — 主 CLI 架构

本章深入 Pi 命令行工具的核心——pi-coding-agent 包。这是用户直接交互的入口，也是整个系统中最复杂的模块。我们将从 CLI 启动流程开始，逐步拆解三种运行模式、核心会话抽象、运行时编排、配置系统、模型解析和认证存储。

Main Line

`pi-coding-agent` 是 Pi 的组装层。它不重新发明模型、agent loop 或 TUI，而是把这些子系统按一次真实 CLI 运行的顺序串起来。

Remember Now

如果你只想记住一件事，那就是用户看到的 Pi 行为大多发生在这个包里，因为这里决定了模式、会话、设置、工具和运行时如何组合。

Info

pi-coding-agent 是 Pi 的"外壳"——它负责解析 CLI 参数、加载配置、选择模型、管理会话文件，并把所有组件组装成一个可运行的 agent。理解这个包就理解了 Pi 的用户态行为。

6.1 CLI 启动流程

当你运行 pi "hello" 时，操作系统执行的是 packages/coding-agent/src/cli.ts。这是整个应用的入口点：

// src/cli.ts — CLI 入口
import { APP_NAME } from "./config.ts";
import { configureHttpDispatcher } from "./core/http-dispatcher.ts";
import { main } from "./main.ts";

process.title = APP_NAME;
process.env.PI_CODING_AGENT = "true";
process.emitWarning = (() => {}) as typeof process.emitWarning;

// 在 provider SDK 发送请求前配置 undici 全局 dispatcher
configureHttpDispatcher();

main(process.argv.slice(2));

这个入口做了四件事：

  1. 设置进程标题为 pi（影响 ps 输出）
  2. 设置环境变量 PI_CODING_AGENT=true（用于检测是否在 agent 内部运行）
  3. 抑制 Node.js 的 process warning（避免干扰输出）
  4. 调用 main() 启动应用

真正的逻辑在 main.ts 中。它的核心流程如下：

// 启动流程概览（src/main.ts）
export async function main(args: string[], options?: MainOptions) {
    resetTimings();

    // 1. 处理离线模式
    const offlineMode = args.includes("--offline") || isTruthyEnvFlag(process.env.PI_OFFLINE);

    // 2. 创建 bootstrap settings manager（用于读取 sessionDir）
    const bootstrapSettingsManager = SettingsManager.create(cwd, agentDir, { projectTrusted: false });

    // 3. 处理 package/config 子命令（pi update, pi config ...）
    if (await handlePackageCommand(args, ...)) { process.exit(exitCode); }
    if (await handleConfigCommand(args, ...)) { return; }

    // 4. 解析 CLI 参数
    const parsed = parseArgs(args);

    // 5. 确定运行模式（interactive / print / rpc）
    let appMode = resolveAppMode(parsed, process.stdin.isTTY, process.stdout.isTTY);

    // 6. 运行数据迁移
    const { migratedAuthProviders, deprecationWarnings } = runMigrations(cwd);

    // 7. 创建 SettingsManager
    const startupSettingsManager = SettingsManager.create(cwd, agentDir);

    // 8. 首次运行设置（主题选择、分析 opt-in）
    if (appMode === "interactive" && shouldRunFirstTimeSetup()) {
        await showFirstTimeSetup(startupSettingsManager);
    }

    // 9. 创建 SessionManager
    let sessionManager = await createSessionManager(parsed, cwd, sessionDir, startupSettingsManager);

    // 10. 创建 AuthStorage
    const authStorage = AuthStorage.create();

    // 11. 创建运行时（AgentSessionRuntime）
    const runtime = await createAgentSessionRuntime(createRuntime, { cwd, agentDir, sessionManager });

    // 12. 根据模式运行
    if (appMode === "rpc") {
        await runRpcMode(runtime);
    } else if (appMode === "interactive") {
        const interactiveMode = new InteractiveMode(runtime, { ... });
        await interactiveMode.run();
    } else {
        const exitCode = await runPrintMode(runtime, { mode: toPrintOutputMode(appMode), ... });
    }
}

Tip

main() 的设计遵循"先收集依赖，再执行逻辑"的模式。它先创建所有必需的服务（SettingsManager、AuthStorage、ModelRegistry），然后才创建 AgentSessionRuntime，最后根据模式启动对应的 I/O 层。

6.2 三种运行模式

Pi 支持三种运行模式，由 resolveAppMode() 函数决定：

// src/main.ts — 模式判断逻辑
function resolveAppMode(parsed: Args, stdinIsTTY: boolean, stdoutIsTTY: boolean): AppMode {
    if (parsed.mode === "rpc") {
        return "rpc";
    }
    if (parsed.mode === "json") {
        return "json";
    }
    if (parsed.print || !stdinIsTTY || !stdoutIsTTY) {
        return "print";
    }
    return "interactive";
}

模式判断规则：

条件 模式 说明
--mode rpc rpc JSON-RPC over stdio，用于外部程序嵌入
--mode json json JSON 事件流输出到 stdout
-p 或 stdin/stdout 非 TTY print 单次执行，输出结果后退出
默认（TTY + 无 -p） interactive 交互式 REPL，使用 pi-tui 渲染终端 UI

Interactive 模式

Interactive 模式是默认模式，当你直接在终端中运行 pi 时进入。它使用 pi-tui 库渲染终端用户界面：

// src/modes/interactive/interactive-mode.ts
export class InteractiveMode {
    private runtime: AgentSessionRuntime;
    private tui: TUI;
    // ...

    async run(): Promise<void> {
        await this.init();
        // 启动 TUI 事件循环
        // 处理用户输入、渲染消息流、显示工具调用结果
    }
}

Interactive 模式的特点：

  • 使用 pi-tui 渲染富文本终端 UI（Markdown 渲染、语法高亮、图片显示）
  • 支持多行编辑、历史记录、自动补全
  • 支持会话切换（/resume、/fork、/tree）
  • 支持模型切换（Ctrl+P）、thinking level 调整
  • 支持扩展提供的 UI 组件和命令

Print 模式

6.3 AgentSession：核心抽象

AgentSession 是 pi-coding-agent 中最核心的类——它是所有运行模式共享的会话抽象。它封装了 agent 状态管理、事件订阅、会话持久化、压缩和模型切换。

Info

AgentSession 本身不处理 I/O。它只管理 agent 状态和事件。三种运行模式各自添加自己的 I/O 层：Interactive 用 pi-tui，Print 用 stdout，RPC 用 JSON-RPC。

类结构概览

export class AgentSession {

事件订阅机制

AgentSession 的事件系统连接了底层 Agent 和上层 UI：

// 订阅事件——返回取消订阅函数
subscribe(listener: AgentSessionEventListener): () => void {
    this._eventListeners.push(listener);
    return () => {
        const index = this._eventListeners.indexOf(listener);
        if (index !== -1) {
            this._eventListeners.splice(index, 1);
        }
    };
}

// 内部事件处理器——持久化 + 扩展 + 通知监听器
private _handleAgentEvent = async (event: AgentEvent): Promise<void> {
    // 1. 从队列中移除已交付的消息
    if (event.type === "message_start" && event.message.role === "user") {
        // 从 steering/followUp 队列中移除
    }

    // 2. 通知扩展
    await this._emitExtensionEvent(event);

    // 3. 通知所有监听器
    this._emit(event);

    // 4. 持久化消息到 session 文件
    if (event.type === "message_end") {
        if (event.message.role === "user" || event.message.role === "assistant" || event.message.role === "toolResult") {
            this.sessionManager.appendMessage(event.message);
        }
        // 跟踪 assistant 消息用于自动压缩
        if (event.message.role === "assistant") {
            this._lastAssistantMessage = event.message;
        }
    }
};

prompt() 方法

prompt() 是发送用户消息的核心方法。它处理了命令拦截、模板扩展、流式排队、模型验证和压缩检查：

async prompt(text: string, options?: PromptOptions): Promise<void> {
    const expandPromptTemplates = options?.expandPromptTemplates ?? true;

    // 1. 处理扩展命令（如 /session, /tree, /help）
    if (expandPromptTemplates && text.startsWith("/")) {
        const handled = await this._tryExecuteExtensionCommand(text);
        if (handled) return;
    }

    // 2. 触发 input 事件（扩展可以拦截或转换输入）
    if (this._extensionRunner.hasHandlers("input")) {
        const inputResult = await this._extensionRunner.emitInput(currentText, ...);
        if (inputResult.action === "handled") return;
        if (inputResult.action === "transform") { currentText = inputResult.text; }
    }

    // 3. 展开技能命令和 prompt 模板
    let expandedText = currentText;
    if (expandPromptTemplates) {
        expandedText = this._expandSkillCommand(expandedText);
        expandedText = expandPromptTemplate(expandedText, [...this.promptTemplates]);
    }

    // 4. 如果正在流式响应，排队消息
    if (this.isStreaming) {
        if (!options?.streamingBehavior) throw new Error("需要指定 streamingBehavior");
        if (options.streamingBehavior === "followUp") {
            await this._queueFollowUp(expandedText, currentImages);
        } else {
            await this._queueSteer(expandedText, currentImages);
        }
        return;
    }

    // 5. 验证模型和 API key
    if (!this.model) throw new Error(formatNoModelSelectedMessage());
    if (!this._modelRegistry.hasConfiguredAuth(this.model)) { throw ... }

    // 6. 检查是否需要压缩
    const lastAssistant = this._findLastAssistantMessage();
    if (lastAssistant) { await this._checkCompaction(lastAssistant, false); }

    // 7. 构建消息数组
    messages = [];
    messages.push({ role: "user", content: [...], timestamp: Date.now() });
    // 注入 nextTurn 消息
    for (const msg of this._pendingNextTurnMessages) { messages.push(msg); }

    // 8. 触发 before_agent_start 扩展事件
    const result = await this._extensionRunner.emitBeforeAgentStart(...);

    // 9. 运行 agent
    await this._runAgentPrompt(messages);
}

steer() 和 followUp()

当 agent 正在流式响应时，用户发送的新消息需要排队。Pi 支持两种排队策略：

// steer——中断当前响应，立即处理新消息
async steer(text: string, images?: ImageContent[]): Promise<void> {
    let expandedText = this._expandSkillCommand(text);
    expandedText = expandPromptTemplate(expandedText, [...this.promptTemplates]);
    await this._queueSteer(expandedText, images);
}

// followUp——等待当前响应完成后处理
async followUp(text: string, images?: ImageContent[]): Promise<void> {
    let expandedText = this._expandSkillCommand(text);
    expandedText = expandPromptTemplate(expandedText, [...this.promptTemplates]);
    await this._queueFollowUp(expandedText, images);
}

private async _queueSteer(text: string, images?: ImageContent[]): Promise<void> {
    this._steeringMessages.push(text);
    this._emitQueueUpdate();
    this.agent.steer({ role: "user", content: [...], timestamp: Date.now() });
}

private async _queueFollowUp(text: string, images?: ImageContent[]): Promise<void> {
    this._followUpMessages.push(text);
    this._emitQueueUpdate();
    this.agent.followUp({ role: "user", content: [...], timestamp: Date.now() });
}
Tip

steer 和 followUp 的区别在于响应优先级。steer 会中断 agent 当前的工具调用链（类似"插嘴"），followUp 则等 agent 完成当前轮次后再处理（类似"排队"）。用户可以通过设置 steeringMode 和 followUpMode 控制一次处理多少消息。

Compaction（压缩）

当会话上下文接近模型窗口限制时，AgentSession 自动触发压缩——将历史消息总结为摘要，释放 token 空间：

// 手动压缩
async compact(customInstructions?: string): Promise<CompactionResult> {
    this._disconnectFromAgent();       // 暂停事件处理
    await this.abort();                // 中止当前操作
    this._compactionAbortController = new AbortController();
    this._emit({ type: "compaction_start", reason: "manual" });

    try {
        // 1. 准备压缩数据

        // 2. 让扩展有机会提供自定义压缩
        if (this._extensionRunner.hasHandlers("session_before_compact")) {
            // 扩展可以取消或提供自定义压缩结果
        }

        // 3. 生成摘要
        const result = await compact(preparation, this.model, apiKey, headers, ...);

        // 4. 保存压缩结果到 session
        this.sessionManager.appendCompaction(summary, firstKeptEntryId, tokensBefore, details);

        // 5. 更新 agent 消息列表
        this.agent.state.messages = sessionContext.messages;

    } finally {
        this._compactionAbortController = undefined;
        this._reconnectToAgent();      // 恢复事件处理

}

自动压缩在每次 agent 响应完成后检查两种情况：

  • Overflow：LLM 返回上下文溢出错误 → 压缩并自动重试
  • Threshold：上下文超过阈值 → 压缩，不重试（用户继续）

AgentSessionRuntime 的会话切换方法是理解 Pi 多会话管理的关键：

// 切换到另一个会话（resume）
async switchSession(sessionPath: string, options?: {
    cwdOverride?: string;
    withSession?: (ctx: ReplacedSessionContext) => Promise<void>;
}): Promise<{ cancelled: boolean }> {
    // 1. 触发 session_before_switch 事件（扩展可以取消）
    const beforeResult = await this.emitBeforeSwitch("resume", sessionPath);
    if (beforeResult.cancelled) return { cancelled: true };

    // 2. 打开目标 session
    const sessionManager = SessionManager.open(sessionPath, undefined, options?.cwdOverride);
    assertSessionCwdExists(sessionManager, this.cwd);

    // 3. 销毁当前运行时
    await this.teardownCurrent("resume", sessionManager.getSessionFile());

    // 4. 创建新运行时
    this.apply(await this.createRuntime({
        cwd: sessionManager.getCwd(),
        agentDir: this.services.agentDir,
        sessionManager,
        sessionStartEvent: { type: "session_start", reason: "resume", previousSessionFile },
    }));

    // 5. 通知 rebind 和 withSession
    await this.finishSessionReplacement(options?.withSession);
    return { cancelled: false };
}

// 创建新会话
async newSession(options?: {
    parentSession?: string;
    setup?: (sessionManager: SessionManager) => Promise<void>;

Warning

会话替换是一个原子操作——要么完全成功，要么完全失败。如果新运行时创建失败，旧会话已经被销毁，错误会传播给调用者处理。调用者负责用户可见的错误处理。

function deepMergeSettings(base: Settings, overrides: Settings): Settings {
    const result: Settings = { ...base };
    for (const key of Object.keys(overrides)) {
        // 嵌套对象递归合并，其他类型直接覆盖
        if (typeof overrideValue === "object" && !Array.isArray(overrideValue)) {
            result[key] = { ...baseValue, ...overrideValue };
        } else {
            result[key] = overrideValue;
        }
    }
    return result;
}

export async function resolveModelScope(
    patterns: string[],
    modelRegistry: ModelRegistry
): Promise<ScopedModel[]> {
    const availableModels = await modelRegistry.getAvailable();
    const scopedModels: ScopedModel[] = [];

    for (const pattern of patterns) {
        // 检查是否包含 glob 字符
        if (pattern.includes("*") || pattern.includes("?")) {
            // 提取可选的 thinking level 后缀（如 "provider/*:high"）
            const colonIdx = pattern.lastIndexOf(":");
            let thinkingLevel: ThinkingLevel | undefined;
            if (colonIdx !== -1) {
                thinkingLevel = isValidThinkingLevel(suffix) ? suffix : undefined;
            }

            // 匹配 "provider/modelId" 或 "modelId"
            const matchingModels = availableModels.filter((m) => {
                const fullId = `${m.provider}/${m.id}`;
                return minimatch(fullId, globPattern, { nocase: true })
                    || minimatch(m.id, globPattern, { nocase: true });
            });
            // ...
        } else {
            // 精确匹配
            const model = findExactModelReferenceMatch(pattern, availableModels);
            // ...
        }
    }
    return scopedModels;
}


        
CLI 模型解析

// src/core/model-resolver.ts
export function resolveCliModel(options: {
    cliProvider?: string;
    cliModel?: string;
    cliThinking?: ThinkingLevel;
    modelRegistry: ModelRegistry;
}): {
    model?: Model<Api>;
    thinkingLevel?: ThinkingLevel;
    warning?: string;
    error?: string;
} {
    const { cliProvider, cliModel, modelRegistry } = options;

    if (cliProvider && !cliModel) {
        // 只有 provider，使用默认模型
        const defaultId = defaultModelPerProvider[cliProvider as KnownProvider];
        if (defaultId) {
            model = modelRegistry.find(cliProvider, defaultId);
        }
    }

    if (cliModel?.includes("/")) {
        // 完整引用：provider/modelId
        model = findExactModelReferenceMatch(cliModel, availableModels);
    } else if (cliModel) {
        // 模糊匹配
        model = findBestMatchWithFallback(cliModel, availableModels, cliProvider);
    }

    return { model, thinkingLevel, warning, error };
}

    private authPath: string;

        let release = this.acquireLockSyncWithRetry(this.authPath);
        const current = readFileSync(this.authPath, "utf-8");

        release();

Pi documentation (read only when the user asks about pi itself):
- Main documentation: ${readmePath}
- Additional docs: ${docsPath}
- Examples: ${examplesPath}`;

    // 追加项目上下文、技能、日期和 cwd
    if (contextFiles.length > 0) { /* ... */ }
    if (skills.length > 0) { prompt += formatSkillsForPrompt(skills); }
    prompt += `\nCurrent date: ${date}`;
    prompt += `\nCurrent working directory: ${promptCwd}`;

    return prompt;
}


        

            
Tip

            
系统提示的构建是"增量式"的——先构建基础模板，然后根据可用工具追加指导原则，最后追加项目上下文和技能。扩展可以通过 appendSystemPrompt 选项注入额外内容。

        


        
        
6.9 SessionManager：会话持久化

        
        
SessionManager 管理会话的 JSONL 文件——每个会话是一个 JSONL 文件，每行是一个 session entry（消息、压缩、模型变更等）。


        
// src/core/session-manager.ts
export const CURRENT_SESSION_VERSION = 3;

export class SessionManager {
    // 创建新会话
    static create(cwd: string, sessionDir?: string, options?: NewSessionOptions): SessionManager;

    // 打开已有会话
    static open(path: string, sessionDir?: string, cwdOverride?: string): SessionManager;

    // 继续最近的会话
    static continueRecent(cwd: string, sessionDir?: string): SessionManager;

    // 内存中会话（--no-session）
    static inMemory(cwd: string, options?: NewSessionOptions): SessionManager;

    // 列出会话
    static async list(cwd: string, sessionDir?: string): Promise<SessionEntrySummary[]>;
    static async listAll(sessionDir?: string): Promise<SessionEntrySummary[]>;

    // 分叉会话
    createBranchedSession(entryId: string): string | undefined;

    // 追加消息
    appendMessage(message: AgentMessage): void;
    appendCompaction(summary: string, firstKeptEntryId: string, tokensBefore: number, ...): void;
    appendModelChange(provider: string, modelId: string): void;
    appendThinkingLevelChange(level: string): void;

    // 构建会话上下文（用于发送给 LLM）
    buildSessionContext(): SessionContext;

    getSessionFile(): string | undefined;
    getSessionId(): string;
    getCwd(): string;
    getBranch(): SessionEntry[];
    isPersisted(): boolean;
}

6.11 Quiz

        
        

            
1. resolveAppMode() 在什么情况下返回 "print" 模式？

            
    
                
  • 当用户输入了空消息时
    • 
                
  • 当使用了 -p 标志、或 stdin/stdout 不是 TTY 时
    • 
                
  • 当使用了 --mode print 标志时
    • 
                
  • 当没有可用模型时
    • 
            

            
resolveAppMode() 的判断逻辑：parsed.print || !stdinIsTTY || !stdoutIsTTY → "print"。--mode 标志只接受 text、json、rpc 三个值，没有 print。

        


        

            
2. AgentSession.prompt() 方法在发送消息前不会做以下哪项检查？

            
    
                
  • 检查是否已选择模型
    • 
                
  • 检查 API key 是否可用
    • 
                
  • 检查消息长度是否超过 token 限制
    • 
                
  • 检查是否需要压缩上下文
    • 
            

            
prompt() 会检查模型选择、API key 和压缩需求，但不会预先检查消息长度——这是 LLM 调用后通过 overflow 错误处理的。如果上下文溢出，会自动触发压缩并重试。

        


        

            
3. AgentSessionRuntime.switchSession() 的正确执行顺序是什么？

            
    
                
  • 创建新运行时 → 销毁旧运行时 → 触发事件
    • 
                
  • 触发 before_switch 事件 → 销毁旧运行时 → 创建新运行时 → 通知 rebind
    • 
                
  • 销毁旧运行时 → 触发事件 → 创建新运行时
    • 
                
  • 创建新运行时 → 触发事件 → 销毁旧运行时
    • 
            

            
正确的顺序是：(1) 触发 session_before_switch 事件（扩展可以取消），(2) 销毁旧运行时（teardownCurrent），(3) 创建新运行时（createRuntime），(4) 通知 rebind 和 withSession 回调。

        


        

            
4. AuthStorage 中 API key 的查找优先级（从高到低）是什么？

            
    
                
  • environment → stored → runtime → fallback
    • 
                
  • runtime → stored → environment → fallback
    • 
                
  • stored → runtime → environment → fallback
    • 
                
  • runtime → environment → stored → fallback
    • 
            

            
优先级是：(1) 运行时覆盖（setRuntimeApiKey / --api-key），(2) auth.json 中存储的 key，(3) 环境变量，(4) models.json 中定义的命令输出（fallback）。

        


        

            
5. 以下关于 SettingsManager 的说法哪个是正确的？

            
    
                
  • 项目设置完全覆盖全局设置
    • 
                
  • 项目设置和全局设置深度合并，项目设置优先
    • 
                
  • 全局设置总是优先于项目设置
    • 
                
  • 项目设置和全局设置互不影响，各自独立生效
    • 
            

            
deepMergeSettings() 实现了深度合并：对于嵌套对象（如 compaction、retry），两层设置递归合并；对于其他类型，项目设置直接覆盖全局设置。
