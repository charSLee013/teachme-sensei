pi-ai — 统一 LLM API 层

理解多提供商抽象的设计，学会添加新提供商

Main Line

pi-ai 的任务不是做“最聪明”的模型层，而是把上游分裂的 provider、API、认证和流式协议压成一个上层可以消费的统一表面。

Remember Now

35 个 provider id 并不等于 35 套完全不同的系统。真正要记住的是三层结构：统一 types、provider 工厂和底层 API 实现。

本章是最核心的技术章节。pi-ai 是整个 Pi 的基础层，所有 LLM 调用都经过这里。理解它，就理解了 Pi 如何与 35 个 LLM 提供商通信。

3.1 为什么需要统一 API？

Pi 支持 35 个 LLM 提供商，每个都有自己的：

  • 不同的 SDK（@anthropic-ai/sdk、openai、@google/genai...）
  • 不同的 API 协议（Completions / Responses / Messages / Converse...）
  • 不同的认证方式（API Key、OAuth、AWS SigV4、GCP ADC...）
  • 不同的流式传输格式（SSE、WebSocket、JSON lines...）
  • 不同的推理/thinking 参数格式

如果没有统一 API 层，上层代码需要为每个提供商写不同的处理逻辑。pi-ai 通过三层架构解决了这个问题。

export interface Model<TApi extends Api> {
  id: string;
  name: string;

  provider: ProviderId;
  baseUrl: string;
  reasoning: boolean;   // 是否支持推理

  contextWindow: number;

AssistantMessageEvent — 流式事件协议

export type AssistantMessageEvent =
  | { type: "start"; partial: AssistantMessage }
  | { type: "text_start"; contentIndex: number; partial: AssistantMessage }
  | { type: "text_delta"; contentIndex: number; delta: string; partial: AssistantMessage }
  | { type: "text_end"; contentIndex: number; content: string; partial: AssistantMessage }
  | { type: "thinking_start"; contentIndex: number; partial: AssistantMessage }
  | { type: "thinking_delta"; contentIndex: number; delta: string; partial: AssistantMessage }
  | { type: "toolcall_start"; contentIndex: number; partial: AssistantMessage }
  | { type: "toolcall_delta"; contentIndex: number; delta: string; partial: AssistantMessage }
  | { type: "toolcall_end"; contentIndex: number; toolCall: ToolCall; partial: AssistantMessage }
  | { type: "done"; reason: StopReason; message: AssistantMessage }
  | { type: "error"; reason: "aborted" | "error"; error: AssistantMessage };

这是一个判别式联合（Discriminated Union）。每个事件都有 type 字段作为判别器，partial 字段携带当前累积的部分消息。

3.4 Provider 工厂模式 — models.ts 走读

packages/ai/src/models.ts 定义了 Provider 的构建和管理：

Provider 接口

export interface Provider<TApi extends Api = Api> {
  readonly id: string;
  readonly name: string;
  readonly auth: ProviderAuth;
  getModels(): readonly Model<TApi>[];
  refreshModels?(): Promise<void>;
  stream<T extends TApi>(
    model: Model<T>,
    context: Context,
    options?: ApiStreamOptions<T>,
  ): AssistantMessageEventStream;
  streamSimple(model: Model<TApi>, context: Context, options?: SimpleStreamOptions): AssistantMessageEventStream;
}

createProvider() 构建器

export function createProvider<TApi extends Api = Api>(
  input: CreateProviderOptions<TApi>
): Provider<TApi> {
  // 支持两种模式：
  // 1. 单个 api 实现（所有模型共用）
  // 2. api 映射（按 model.api 分发到不同实现）
  const single = typeof input.api === "function" ? input.api : undefined;
  const byApi = single ? undefined : input.api as Record<string, ProviderStreams>;

  const dispatch = (model, run) => {
    const streams = single ?? byApi?.[model.api];
    if (!streams) throw new Error(`No API for "${model.api}"`);
    return run(streams);
  };

  return {
    id: input.id,
    auth: input.auth,
    getModels: () => models,
    stream: (model, ctx, opts) => dispatch(model, s => s.stream(model, ctx, opts)),
    streamSimple: (model, ctx, opts) => dispatch(model, s => s.streamSimple(model, ctx, opts)),
  };
}

// OAuth 认证

3.7 Faux Provider — 测试的假提供商

// packages/ai/src/providers/faux.ts
export function fauxProvider(): Provider<"openai-responses"> {
  return createProvider({
    id: "faux",
    name: "Faux (Testing)",
    auth: fauxAuth,
    models: [
      {
        id: "faux-model",
        name: "Faux Model",
        api: "openai-responses",
        provider: "faux",
        baseUrl: "http://localhost:0",
        reasoning: false,
        input: ["text"],

        contextWindow: 128000,
        maxTokens: 4096,
      },
    ],
    api: fauxFns,  // 返回预设响应，不发起真实请求
  });
}

Faux provider 让测试不需要真实 API key 或网络请求，是 Pi 测试套件的基础。

3.8 小测验

Q1: pi-ai 的三层架构中，哪一层负责实际发起 HTTP 请求？

A. types.ts
B. providers/
C. api/
D. models.ts
✅ 正确答案：C。api/ 目录包含每个 API 协议的实际 SDK/HTTP 调用实现。

Q2: Model<TApi> 中的泛型参数 TApi 有什么作用？

A. 指定模型的价格
B. 将模型与特定的 API 协议关联，实现类型安全
C. 指定模型的上下文窗口大小
D. 指定模型的认证方式
✅ 正确答案：B。TApi 将模型与 API 协议关联，使得 stream() 的选项类型会根据模型自动推断。

Q3: createProvider() 支持哪两种 API 分发模式？

A. HTTP 和 WebSocket
B. 单个实现（所有模型共用）和按 model.api 分发
C. 同步和异步
D. REST 和 gRPC
✅ 正确答案：B。api 字段可以是单个 ProviderStreams（所有模型共用），也可以是 Record<TApi, ProviderStreams>（按 model.api 分发）。

Q4: AssistantMessageEvent 中的 partial 字段有什么作用？

A. 表示消息是否完整
B. 携带当前累积的部分消息，用于 UI 增量渲染
C. 表示消息是否被截断
D. 表示消息的优先级
✅ 正确答案：B。每个事件都携带 partial，让 UI 可以在每个 delta 到达时立即更新显示。

Q5: Faux provider 的主要用途是？

A. 测试 — 不需要真实 API key
B. 开发 — 本地 LLM 代理
C. 生产 — 备用提供商
D. 调试 — 网络请求记录
✅ 正确答案：A。Faux provider 返回预设响应，用于 Pi 的测试套件。
