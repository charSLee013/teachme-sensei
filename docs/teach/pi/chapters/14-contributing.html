Chapter 14: 贡献指南与开发实践

本章基于项目当前的贡献规则、开发命令和仓库工作约束，介绍如何为 Pi Agent Harness 贡献代码。

Main Line

这章的核心不是“怎么提交一个 Node 项目 PR”，而是学会遵守 Pi 当前仓库的真实门槛、命令和 contributor gate。

Remember Now

最容易犯错的地方有两个：把 `lgtm` 误读成合并票数，以及把泛化的 npm/pnpm 容器和发布经验误套到当前仓库。

14.1 Philosophy（核心哲学）

Pi's core is minimal. 如果一个功能不属于核心，它应该做成 extension。会让核心膨胀的 PR 很可能会被拒绝。

核心的设计目标是最小化 + 可扩展性——让外部通过扩展来影响和操控系统。即使是给 extension 加 hook point，也需要充分讨论，避免引入不可维护的复杂交互。

Extension-First

想加新功能？先问自己：能不能做成 .pi/extensions/ 下的扩展？多数非核心能力都应该先走扩展路径，而非直接改核心。

14.2 The One Rule（唯一规则）

You must understand your code. 如果你无法解释你的改动做了什么、它与系统其他部分如何交互，你的 PR 将被关闭。

  • 使用 AI 写代码是可以的。
  • 提交者必须能解释自己提交的 AI 生成内容；无法解释的内容不会被接受。
  • 如果你用 agent，请从 pi 根目录运行，让它先加载项目本地规则。

项目规则要点

  • 代码质量：不用 any、内联辅助函数、erasable TS syntax（禁 enum/namespace/parameter properties）。
  • 命令：check 不 build（npm run check，不是 npm run build）。
  • 测试：用 ./test.sh（不直接调用 vitest，避免 e2e 触发 API key 相关流程）。
  • Git：只提交自己负责的文件，永远不要用 git add -A / .。
  • 依赖：直接外部依赖必须锁精确版本，安装时使用 --ignore-scripts。

14.3 Contribution Flow

Contribution Gate（新贡献者门控）

所有来自新 contributor 的 issue 和 PR 默认 auto-close。 这是贡献门禁的一部分：维护者会按质量标准复核，并决定哪些内容值得重开和继续处理。

维护者回复 含义
lgtmi 你未来的 issue 不再被 auto-close（但不包含 PR）
lgtm 你未来的 issue 和 PR 都不再被 auto-close
注意

周五到周日提交的 issue 可能延后到工作日审查；紧急问题可转到 Discord 反馈。

Issue 质量要求

  • 必须使用 GitHub issue 模板。
  • issue 应简短、具体，并在有限篇幅内说明问题、影响和必要上下文。
  • issue 文本应反映提交者自己的理解；如果使用 AI 起草，请追加明确的 AI 标注。
  • 清晰描述 bug 或请求，解释为什么重要，如果你想自己实现请说明。

Before Submitting a PR

npm run check
./test.sh

两者必须都通过。不要编辑 CHANGELOG.md——changelog 条目由维护者添加。

Blocking（违规封禁）

如果反复忽略本指南，或持续提交自动化低质量 issue，项目可能永久屏蔽相关 GitHub 账号。

14.4 FAQ

为什么 auto-close？Pi 收到的 issue 数量超过维护者能实时审查的量，其中一部分并不满足质量要求。默认自动关闭机制的目标，是把维护时间优先留给具体、可复现、符合指南的报告。

为什么有些 issue 没回复？回复也是维护工作。低信号、不清晰、重复、不按指南的 issue 可能没讨论就关闭了——这省出时间给可复现的 bug、深思熟虑的请求。

为什么不全部用 AI triage？AI 可以帮助去重、总结和发现缺失信息，但最终维护判断仍由人工完成，因为误判和误导的排查成本较高。

这对 contributor 有敌意吗？不是。规则的目标是保护维护容量，并减少低质量报告对审查队列的干扰；符合指南的具体、可复现问题仍然欢迎。

在哪了解路线图？Earendil 用 RFC 讨论大变更。部分公开：rfc.earendil.com。

常用命令

# 安装依赖
npm install --ignore-scripts

# 构建所有包（需要时）
npm run build

# 检查（check = biome + TypeScript package checks + shrinkwrap + tsgo --noEmit）
npm run check

# test.sh（无 API key 的本地测试）
./test.sh

# 版本递增 + 发布（维护者权限）
PI_ALLOW_LOCKFILE_CHANGE=1 npm_config_min_release_age=0 npm run release:patch
PI_ALLOW_LOCKFILE_CHANGE=1 npm_config_min_release_age=0 npm run release:minor

Git 规范

  • 分支命名：feat/、fix/、docs/。
  • PR 目标分支是 main。
  • 提交前必须通过 npm run check && ./test.sh。

14.5 Testing

test.sh

项目使用统一的测试脚本 test.sh。它不是测试框架本身，而是一层保护壳：先把 auth.json 临时移走，清除 provider 相关环境变量，设置 PI_NO_LOCAL_LLM=1，最后再调用 npm test。这样 无需真实 API key 即可跑本地默认测试路径。

#!/usr/bin/env bash
set -e

# 备份 auth.json
AUTH_FILE="$HOME/.pi/agent/auth.json"
AUTH_BACKUP="$HOME/.pi/agent/auth.json.bak"
trap '[[ -f "$AUTH_BACKUP" ]] && mv "$AUTH_BACKUP" "$AUTH_FILE"' EXIT

# 跳过本地 LLM 测试
export PI_NO_LOCAL_LLM=1

# 清空所有 API key
unset ANTHROPIC_API_KEY
unset OPENAI_API_KEY
unset GEMINI_API_KEY
# ...（共 30+ 个环境变量）

npm test

回归测试

每个 bug fix 必须附带回归测试——一个在修复前会失败、修复后通过的测试用例。回归测试放在 packages/coding-agent/test/suite/regressions/ 目录下：

// packages/coding-agent/test/suite/regressions/3302-find-path-glob.test.ts
import { describe, it, expect } from 'vitest';
// 真实的测试请参考已有的 regression 文件作为模板

describe('regression: empty-system-prompt', () => {
    it('should not crash on edge case', async () => {
        // 使用 test.sh 运行此测试，无需真实 API key
    });
});

注：具体的 AgentSession API 请参考 packages/coding-agent/src/core/agent-session.ts。本示例为占位。

Faux Provider

Faux Provider helpers（packages/ai/src/providers/faux.ts）是 Pi 的测试利器——它返回预设的 AssistantMessage 或 toolCall 对象，不需要真实 API key。项目测试规则要求非 e2e 测试使用 faux provider。

请在具体测试中参考 fauxAssistantMessage 和 fauxToolCall 的导出 API。

14.6 Release Process

Lockstep Versioning

monorepo 中所有包使用统一版本号。发布时，所有包的版本同步递增：

// 所有包的 package.json 共享相同版本
{
    "name": "pi-agent-core",
    "version": "1.2.3"
}
{
    "name": "pi-tui",
    "version": "1.2.3"
}

本地发布预检

发布前的本地预检以仓库规则为准：先构建一个未发布的本地 release，再在仓库外验证 Node 和 Bun 产物是否可启动、可列模型、可进入交互模式。

npm run release:local -- --out /tmp/pi-local-release --force
cd /tmp

/tmp/pi-local-release/node/pi --help
/tmp/pi-local-release/node/pi --version
/tmp/pi-local-release/node/pi --list-models
/tmp/pi-local-release/node/pi -p "Say exactly: ok"

/tmp/pi-local-release/bun/pi --help
/tmp/pi-local-release/bun/pi --version
/tmp/pi-local-release/bun/pi --list-models
/tmp/pi-local-release/bun/pi -p "Say exactly: ok"

release:patch / release:minor

# patch: fixes + additions
PI_ALLOW_LOCKFILE_CHANGE=1 npm_config_min_release_age=0 npm run release:patch

# minor: breaking changes
PI_ALLOW_LOCKFILE_CHANGE=1 npm_config_min_release_age=0 npm run release:minor

发布脚本会统一更新所有包版本、运行 npm run check、提交 release commit、打 tag，并由 CI 继续完成 npm 发布。这里的关键词是 lockstep versioning，而不是手工逐包 version bump。

CI Publish

GitHub Actions 在 tag push 时自动发布到 npm registry。只有维护者有发布权限。

14.7 Security

Pi 采用多层安全策略：

Supply Chain Hardening

  • 所有依赖锁版本（package-lock.json 和 npm-shrinkwrap.json 提交到仓库）。
  • CI 与定时工作流会运行 npm audit --omit=dev 和签名检查。
  • 根 .npmrc 当前强调的是 save-exact=true 与 min-release-age=2，用于限制依赖漂移和同日新发包风险。

Containerization

Pi 不提供官方 Docker 镜像——用户需自行容器化以隔离 agent 的文件系统访问。一个典型的生产容器化示例：

# 用户自建 Dockerfile（非官方）
FROM node:20-alpine
WORKDIR /workspace
COPY package.json package-lock.json ./
COPY packages/coding-agent/dist/ ./dist/
RUN npm ci --ignore-scripts --omit=dev
USER 1000:1000
ENTRYPOINT ["node", "dist/cli.js"]

No Built-in Permissions

Pi 不在工具系统内建权限检查——这是有意的设计决策。原因：

  • 权限模型因部署环境而异（CLI vs IDE vs CI）。
  • 在核心中硬编码权限会限制灵活性。
  • 权限应该由外部层（容器、OS、用户配置）实施。
Warning

如果你在不受信环境中运行 Pi（例如处理不受信的代码），务必在容器或沙箱中运行，并显式配置 disabledTools 和 allowedPaths。

14.8 Changelog Conventions

Changelog 遵循 Keep a Changelog 格式：

## [Unreleased]

### Added
- New SDK event `session:end` for graceful shutdown hooks.

### Changed
- Improved error messages for tool call failures.

### Fixed
- Fixed #1234: crash on empty system prompt.

### Security
- Updated `anthropic-sdk` to patch a dependency vulnerability.

## [1.2.3] - 2025-06-15
...

分类说明：

  • Added：新功能
  • Changed：现有功能的变更
  • Removed：已移除或即将移除的功能
  • Fixed：bug 修复
  • Security：安全相关

14.10 Quiz

1. 新 contributor 在什么条件下才应提交 PR？
  • CI 能跑通就可以直接开 PR
  • 先在 issue 上获得维护者的 lgtm 批准
  • 先获得两个 maintainer review approval
  • 只要拿到 lgtmi 就可以
lgtm 在这里是贡献门控，而不是“合并前需要几个票”。lgtmi 只放开未来 issue，不放开 PR。
2. faux provider 的主要用途是什么？
  • 提高 LLM 推理速度
  • 缓存 API 响应以节省费用
  • 在测试中模拟 LLM 响应，无需真实 API key
  • 提供多模型切换能力
Faux Provider 返回预设响应，让测试不依赖外部 API，保证确定性和可重复性。
3. 为什么 Pi 不在工具系统内建权限检查？
  • 因为实现起来太复杂
  • 权限模型因部署环境而异，应由外部层（容器、OS、配置）实施
  • 因为 Pi 不支持文件系统操作
  • 因为 TypeScript 的类型系统已经足够安全
Pi 有意将权限检查交给外部层——容器、操作系统或用户配置。这样同一个工具系统可以在 CLI、IDE、CI 等不同环境中灵活使用。
4. 以下哪个 changelog 分类用于描述"已移除的功能"？
  • Removed
  • Fixed
  • Changed
  • Security
Removed 表示功能已被移除。Fixed 是 bug 修复，Changed 是行为变更，Security 是安全相关修复。
