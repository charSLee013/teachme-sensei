Chapter 10: 压缩系统 — 管理 LLM 上下文窗口

LLM 的上下文窗口有限（128K-200K tokens），而一次编程对话很容易超出这个限制。Pi 的 压缩系统 通过 LLM 生成摘要来压缩历史对话，使会话可以无限延续。

Main Line

压缩系统解决的不是“怎样删旧消息”，而是“怎样在丢掉细节前保住继续工作的最小上下文”。

Remember Now

Compaction 是用摘要替代早期历史，而不是硬截断。只要你忘了这点，就会误解为什么 Pi 还要维护结构化摘要格式和切断点规则。

10.1 为什么需要压缩

一次典型编程对话包含大量 token：系统提示、工具定义、文件内容、工具结果。一轮简单的"读文件-改文件-验证"就可能消耗 10K-30K tokens。当累计上下文逼近窗口上限时，必须压缩历史。

压缩 vs 截断

截断直接丢弃最早的消息，会丢失重要上下文。压缩用 LLM 生成结构化摘要，保留关键信息（目标、约束、进度、决策），只丢弃冗余细节。

10.2 自动 vs 手动压缩

Pi 支持两种触发方式：

  • 自动：每次模型响应后检查 token 用量，超过阈值时自动触发。由 shouldCompact() 决定。
  • 手动：用户输入 /compact 命令，或扩展调用 prepareCompaction() 自定义压缩策略。
export function shouldCompact(
    contextTokens: number,
    contextWindow: number,
    settings: CompactionSettings,
): boolean {
    if (!settings.enabled) return false;
    return contextTokens > contextWindow - settings.reserveTokens;
}

例如 contextWindow=128000，reserveTokens=16384，则当 contextTokens 超过 111616 时触发压缩。

10.3 压缩流程

核心入口：compact()

export async function compact(
    preparation: CompactionPreparation,
    model: Model<any>,
    apiKey: string | undefined,
    headers?: Record<string, string>,
    customInstructions?: string,
    signal?: AbortSignal,
    thinkingLevel?: ThinkingLevel,
    streamFn?: StreamFn,
    env?: Record<string, string>,
): Promise<CompactionResult> {
    const { firstKeptEntryId, messagesToSummarize, turnPrefixMessages,
            isSplitTurn, tokensBefore, previousSummary, fileOps, settings } = preparation;

    let summary: string;
    if (isSplitTurn && turnPrefixMessages.length > 0) {
        // 并行生成历史摘要和 turn 前缀摘要
        const [historyResult, turnPrefixResult] = await Promise.all([...]);
        summary = `${historyResult}\n\n---\n\n**Turn Context (split turn):**\n\n${turnPrefixResult}`;
    } else {
        summary = await generateSummary(messagesToSummarize, model, ...);
    }

    // 附加文件操作信息
    const { readFiles, modifiedFiles } = computeFileLists(fileOps);
    summary += formatFileOperations(readFiles, modifiedFiles);

    return { summary, firstKeptEntryId, tokensBefore, details: { readFiles, modifiedFiles } };
}

10.4 文件操作追踪

压缩时会追踪哪些文件被读取/修改，附加到摘要末尾，让模型知道哪些文件已被处理：

export function extractFileOpsFromMessage(message: AgentMessage, fileOps: FileOperations): void {
    if (message.role !== "assistant") return;
    for (const block of message.content) {
        if (block.type !== "toolCall") continue;
        const path = block.arguments?.path;
        if (!path) continue;
        switch (block.name) {
            case "read":    fileOps.read.add(path); break;
            case "write":   fileOps.written.add(path); break;
            case "edit":    fileOps.edited.add(path); break;
        }
    }
}

export function formatFileOperations(readFiles: string[], modifiedFiles: string[]): string {
    const sections: string[] = [];
    if (readFiles.length > 0)
        sections.push(`<read-files>\n${readFiles.join("\n")}\n</read-files>`);
    if (modifiedFiles.length > 0)
        sections.push(`<modified-files>\n${modifiedFiles.join("\n")}\n</modified-files>`);
    return sections.length > 0 ? `\n\n${sections.join("\n\n")}` : "";
}

Branch vs Compaction

Compaction 是"向前看"——压缩旧消息为新消息腾空间。Branch Summary 是"回头看"——总结被放弃的路径以便将来返回时恢复上下文。两者都用 LLM 生成摘要，但触发时机和目的不同。

10.6 Token 估算

Pi 使用 字符数/4 启发式 估算 token，因为精确计数需要模型特定的 tokenizer：

export function estimateTokens(message: AgentMessage): number {
    let chars = 0;
    switch (message.role) {
        case "user":
            chars = estimateTextAndImageContentChars(message.content);
            return Math.ceil(chars / 4);
        case "assistant":
            for (const block of message.content) {
                if (block.type === "text") chars += block.text.length;
                else if (block.type === "thinking") chars += block.thinking.length;
                else if (block.type === "toolCall")
                    chars += block.name.length + JSON.stringify(block.arguments).length;
            }
            return Math.ceil(chars / 4);
        case "bashExecution":
            chars = message.command.length + message.output.length;
            return Math.ceil(chars / 4);
        // ...
    }
}

10.7 CompactionSettings

export interface CompactionSettings {
    enabled: boolean;
    reserveTokens: number;    // 为响应预留的 token 数
    keepRecentTokens: number; // 压缩时保留最近多少 token 的消息
}

export const DEFAULT_COMPACTION_SETTINGS: CompactionSettings = {
    enabled: true,
    reserveTokens: 16384,
    keepRecentTokens: 20000,
};

findCutPoint() 从最新消息反向累加，直到达到 keepRecentTokens，然后在该位置切断。切断点必须是 user/assistant 消息（不能在 toolResult 中间切断）。

10.9 Quiz

1. 为什么 Pi 使用 chars/4 而不是精确 tokenizer 来估算 token？
  • 不同模型的 tokenizer 不同，chars/4 是跨模型的保守估算
  • chars/4 比 tokenizer 更准确
  • tokenizer 计算太慢，影响性能
  • LLM API 不支持返回 token 数量
chars/4 是粗略但通用的估算。Pi 在有可能时（模型返回 usage）使用精确数据，否则回退到启发式。
2. findCutPoint() 在什么位置切断消息列表？
  • 总是在第一条 user 消息处
  • 从最新反向累加到 keepRecentTokens，在 user/assistant 消息处切断
  • 在任意消息处，只要达到 token 上限
  • 在 compaction entry 之后的第一条消息处
切断点必须是 user 或 assistant 消息（不能切断 toolResult），否则会破坏 tool call/result 的配对。
3. 压缩摘要的格式要求是什么？
  • 自由格式，任意长度
  • 结构化格式，包含 Goal、Progress、Key Decisions 等固定章节
  • 只包含修改过的文件列表
  • JSON 格式，包含 messages 数组
SUMMARIZATION_PROMPT 要求 LLM 输出固定格式的 markdown 章节，确保摘要包含所有关键上下文。
4. Branch Summary 和 Compaction 的主要区别是什么？
  • Branch Summary 更快，Compaction 更慢
  • Compaction 为腾空间向前压缩；Branch Summary 总结被放弃的路径
  • Compaction 只能手动触发，Branch Summary 自动触发
  • Branch Summary 不调用 LLM
两者都用 LLM 生成摘要。Compaction 在上下文超限时触发，Branch Summary 在用户导航到树的另一分支时触发。
