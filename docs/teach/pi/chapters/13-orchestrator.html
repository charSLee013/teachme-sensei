Chapter 13: 编排器 — 多代理协调（实验性）

编排器（Orchestrator）是 Pi 的多代理协调系统——它让多个 agent 协同完成复杂任务。本章介绍其架构、IPC 协议和设计挑战。

Main Line

这一章的目标是建立多代理架构直觉，而不是教你依赖一个稳定产品表面。当前最可靠的收获是消息边界、角色分工和失败模式。

Remember Now

编排器的难点不在“多开几个 agent”，而在上下文隔离、消息路由、可恢复性和错误传播。

Experimental

编排器目前处于实验性阶段。API 可能随时变更，不建议用于生产环境。本章旨在帮助理解多代理系统的设计思路，而非提供稳定使用指南。

13.1 What is the Orchestrator?

编排器是一个多 agent 协调框架。它允许你：

  • 启动多个专门的 agent（coder、reviewer、tester、planner）。
  • 让它们通过消息传递协作完成任务。
  • 由一个 supervisor 协调整体流程。

典型工作流：planner 分解任务 → coder 实现 → reviewer 审查 → tester 验证 → supervisor 汇总。

13.2 Architecture

编排器由四个核心文件组成：

文件 职责
serve.ts 启动编排服务器，监听客户端连接，管理 agent 生命周期
supervisor.ts 中央协调 agent——分配任务、判断完成条件、汇总结果
handler.ts 消息路由——根据消息类型将请求分发给对应的 worker agent
storage.ts 持久化层——保存任务状态、消息历史和 agent 上下文

handler.ts

// 消息路由器——根据 action 分发
export class MessageHandler {
    async handle(msg: Envelope): Promise<void> {
        switch (msg.action) {
            case 'task.assign':  return this.onTaskAssign(msg);
            case 'task.result':  return this.onTaskResult(msg);
            case 'task.review':  return this.onTaskReview(msg);
            case 'agent.status':  return this.onAgentStatus(msg);
            default:              return this.onUnknown(msg);
        }
    }
}

// 任务与消息持久化
export interface StorageBackend {

    loadTask(id: string): Promise<Task | null>;
    appendMessage(taskId: string, msg: Message): Promise<void>;
    getHistory(taskId: string): Promise<Message[]>;
}

13.3 IPC Protocol

编排器使用三层 IPC 协议实现客户端-服务端通信：

层 文件格式 用途
client/ client.ts 客户端 SDK——发送任务、接收结果
server/ server.ts 服务端——接收请求、管理 agent 池
protocol/ protocol.ts 共享消息类型、序列化/反序列化

协议消息格式：

// protocol.ts — 消息信封
interface Envelope {
    id: string;          // 消息唯一 ID
    action: Action;      // 'task.assign' | 'task.result' | ...
    sender: string;      // 发送者 agent ID
    recipient: string;   // 接收者 agent ID 或 'broadcast'
    payload: unknown;    // 消息体
    timestamp: number;   // 发送时间戳
    correlation?: string;// 关联的父消息 ID
}
Info

IPC 层支持多种传输：Unix socket（本地）、TCP（远程）、in-memory（测试）。协议本身与传输无关。

13.4 Radius — 网络注册机制

radius.ts 实现了编排器的网络注册与心跳机制。它让编排器实例能够向远程服务 https://radius.pi.dev/ 注册自己的 presence（在线状态），并周期性发送心跳以维持注册有效性。

导出/概念 说明
RadiusPresence Presence 状态数据结构——记录实例 ID、URL、token、平台信息
RadiusHttpError HTTP 错误封装——携带 status code（用于退避重试决策）
radiusAuthStorage 使用 AuthStorage 存储 radius 的 OAuth token
退避重试 基于 HEARTBEAT_BACKOFF_BASE_MS=1000，最大 HEARTBEAT_BACKOFF_MAX_MS=30000，NOT_FOUND_RETRY_THRESHOLD=3 次 404 后停止重试
说明

Radius 是网络层面的服务发现机制，与教学计划早期草稿中提到的"agent 可见性范围"概念无关。实际源码通过向远程 HTTP endpoint 注册来实现实例发现，而不控制 agent 之间的上下文共享。

13.6 Design Challenges

多代理系统面临若干独特的设计挑战。编排器的实验性很大程度上源于这些问题尚未有共识解法：

任务分解

如何把一个模糊的用户请求分解为可独立执行的子任务？过度分解导致协调开销，分解不足则失去多代理的意义。

上下文隔离 vs 共享

每个 agent 应该看到多少信息？太多上下文导致 token 浪费和注意力分散；太少则无法做出正确决策。当前编排器实验性地通过 supervisor 手动编排消息流来控制上下文边界——尚未有成熟的自动化方案。

错误传播

一个 agent 的错误如何被检测和恢复？编排器实现了重试和返工循环，但判断"何时放弃"仍然困难。

死锁与活锁

当多个 agent 互相等待对方的输出时，可能陷入死锁。编排器使用超时和 supervisor 干预来缓解。

一致性

多个 agent 并行修改文件系统时，如何保证一致性？当前方案是文件级锁 + supervisor 串行化写操作。

Discussion

多代理系统是否真的比单 agent + 多轮对话更有效？这是当前 AI 工具领域的核心辩论。编排器的实验性设计让 Pi 社区可以探索不同策略，收集实证数据。

13.7 Quiz

1. radius.ts 在编排器中的实际职责是什么？
  • 控制哪些 agent 能看到其他 agent 的输出
  • 向远程注册服务汇报编排器实例的在线状态和心跳
  • 对 agent 之间的消息进行路由和转发
  • 管理 agent 池的创建和销毁
radius.ts 使用 HTTP 请求向 radius.pi.dev 注册本地机器并发送心跳。它不控制 agent 之间的消息可见性。
2. handler.ts 的核心职责是什么？
  • 执行 LLM 推理
  • 根据消息 action 字段将请求路由到对应的处理函数
  • 管理文件系统操作
  • 生成用户界面
handler.ts 是消息路由器——它根据 Envelope 的 action 字段（如 task.assign、task.result）将消息分发给对应的处理逻辑。
3. 以下哪项不是编排器面临的设计挑战？
  • 任务分解的粒度
  • 错误传播与恢复
  • 死锁与活锁
  • 单 agent 的 token 限制
单 agent 的 token 限制是 LLM 本身的约束，不是多代理编排特有的挑战。编排器面临的独特问题包括任务分解、上下文隔离、错误传播和死锁。
