pi-agent-core — Agent 运行时

深入理解 Agent 的核心循环、消息系统、工具执行与事件机制

Main Line

本章真正要学的是 agent 如何维持一个可继续、可中断、可插工具的控制循环，而不是背诵每个事件名。

Remember Now

Agent 本身不替模型做决策。它负责保存状态、组织消息、执行工具和发事件，决策点仍然落在模型响应和钩子回调上。

Chapter 4: pi-agent-core — Agent 运行时

1. Agent 的核心循环

Agent 的本质是一个循环：接收用户消息，调用 LLM 获取响应，如果响应中包含工具调用则执行工具，将结果反馈给 LLM，然后重复——直到 LLM 给出最终回答。

核心概念 Agent 循环的关键设计：LLM 是决策者。Agent 本身不决定下一步做什么——它将上下文交给 LLM，LLM 决定是输出文本还是调用工具。这种设计使得 Agent 的行为完全由 prompt + tools 驱动。

2. 代码走读：agent-loop.ts

packages/agent/src/agent-loop.ts 是 Agent 运行时的核心文件。它导出了三个关键函数，构成循环的入口和主逻辑。

2.1 agentLoop() — 入口点

agent-loop.ts 
export function agentLoop(
  prompts: AgentMessage[],
  context: AgentContext,
  config: AgentLoopConfig,
  signal?: AbortSignal,
  streamFn?: StreamFn,
): EventStream<AgentEvent, AgentMessage[]> {
  const stream = createAgentStream();

  void runAgentLoop(
    prompts,
    context,
    config,
    async (event) => { stream.push(event); },
    signal,
    streamFn,
  ).then((messages) => {
    stream.end(messages);
  });

  return stream;
}

agentLoop 是非阻塞的：它创建一个 EventStream，异步启动 runAgentLoop，然后立即返回流。调用方通过订阅事件流来获取实时更新。

2.2 runAgentLoop() — 核心循环

agent-loop.ts 
export async function runAgentLoop(
  prompts: AgentMessage[],
  context: AgentContext,
  config: AgentLoopConfig,
  emit: AgentEventSink,
  signal?: AbortSignal,
  streamFn?: StreamFn,
): Promise<AgentMessage[]> {
  const newMessages: AgentMessage[] = [...prompts];
  const currentContext: AgentContext = {
    ...context,
    messages: [...context.messages, ...prompts],
  };

  await emit({ type: "agent_start" });
  await emit({ type: "turn_start" });
  // ... emit message_start/end for each prompt

  await runLoop(currentContext, newMessages, config, signal, emit, streamFn);
  return newMessages;
}

2.3 agentLoopContinue() — 重试/继续

agent-loop.ts 
export function agentLoopContinue(
  context: AgentContext,
  config: AgentLoopConfig,
  signal?: AbortSignal,
  streamFn?: StreamFn,
): EventStream<AgentEvent, AgentMessage[]> {
  if (context.messages.length === 0) {
    throw new Error("Cannot continue: no messages in context");
  }
  if (context.messages[context.messages.length - 1].role === "assistant") {
    throw new Error("Cannot continue from message role: assistant");
  }

  const stream = createAgentStream();
  void runAgentLoopContinue(...);
  return stream;
}
约束 agentLoopContinue 要求上下文最后一条消息必须是 user 或 toolResult。这是 LLM provider 的硬性要求——不能从 assistant 消息之后继续。

2.4 runLoop() — 真正的循环逻辑

这是整个运行时的心脏。它包含内外两个循环：

  • 外层循环：处理 follow-up 消息（agent 本该停止时，有新消息到达）
  • 内层循环：处理 steering 消息 + tool calls
agent-loop.ts — runLoop (简化) 
async function runLoop(...) {
  let currentContext = initialContext;
  let config = initialConfig;
  let firstTurn = true;
  let pendingMessages = (await config.getSteeringMessages?.()) || [];

  while (true) {                       // outer: follow-up loop
    let hasMoreToolCalls = true;

    while (hasMoreToolCalls || pendingMessages.length > 0) {  // inner: tool loop
      if (!firstTurn) await emit({ type: "turn_start" });

      // 1. Process pending steering messages
      for (const message of pendingMessages) {
        currentContext.messages.push(message);
        newMessages.push(message);
      }

      // 2. Stream assistant response from LLM
      const message = await streamAssistantResponse(...);

      // 3. Check for error/abort
      if (message.stopReason === "error" || message.stopReason === "aborted") {
        await emit({ type: "agent_end", messages: newMessages });
        return;
      }

      // 4. Execute tool calls
      const toolCalls = message.content.filter(c => c.type === "toolCall");
      if (toolCalls.length > 0) {
        const batch = await executeToolCalls(...);
        hasMoreToolCalls = !batch.terminate;
        // ... push results
      }

      await emit({ type: "turn_end", message, toolResults });

      // 5. prepareNextTurn — dynamic model/context switching
      const nextTurn = await config.prepareNextTurn?.(...);
      if (nextTurn) {
        currentContext = nextTurn.context ?? currentContext;
        config = { ...config, model: nextTurn.model ?? config.model, ... };
      }

      // 6. shouldStopAfterTurn — graceful stop
      if (await config.shouldStopAfterTurn?.(...)) {
        await emit({ type: "agent_end", messages: newMessages });
        return;
      }

      // 7. Poll for new steering messages
      pendingMessages = (await config.getSteeringMessages?.()) || [];
    }

    // 8. Agent would stop — check for follow-up messages
    const followUpMessages = (await config.getFollowUpMessages?.()) || [];
    if (followUpMessages.length > 0) {
      pendingMessages = followUpMessages;
      continue;  // back to outer loop
    }

    break;  // truly done
  }

  await emit({ type: "agent_end", messages: newMessages });
}
设计洞察 注意 prepareNextTurn 在 shouldStopAfterTurn 之前调用。这意味着你可以在改变模型配置之后、决定是否停止之前，基于新的状态做出判断。这个顺序是有意为之的。

3. Agent 类

Agent 类（packages/agent/src/agent.ts）是有状态的高层封装。它包装了低层的 agentLoop / agentLoopContinue，管理 transcript、事件分发和消息队列。

3.2 AgentState — 可观察状态

3.3 createMutableAgentState() — 内部可变状态

agent.ts — createMutableAgentState 
function createMutableAgentState(initialState?: Partial<...>): MutableAgentState {
  let tools = initialState?.tools?.slice() ?? [];
  let messages = initialState?.messages?.slice() ?? [];

  return {
    systemPrompt: initialState?.systemPrompt ?? "",
    model: initialState?.model ?? DEFAULT_MODEL,
    thinkingLevel: initialState?.thinkingLevel ?? "off",

    get tools() { return tools; },
    set tools(nextTools: AgentTool<any>[]) { tools = nextTools.slice(); },

    get messages() { return messages; },
    set messages(nextMessages: AgentMessage[]) { messages = nextMessages.slice(); },

    isStreaming: false,
    streamingMessage: undefined,
    pendingToolCalls: new Set<string>(),
    errorMessage: undefined,
  };
}

3.4 Agent 的公共 API

方法说明
prompt(input)发送新消息启动循环。接受 string、AgentMessage 或 AgentMessage[]
continue()从当前 transcript 继续。最后一条消息必须是 user 或 toolResult
steer(message)注入 steering 消息（在当前 turn 结束后）
followUp(message)注入 follow-up 消息（在 agent 本应停止后）
subscribe(listener)订阅 AgentEvent。返回取消订阅函数
abort()中止当前运行
waitForIdle()等待当前运行及所有 listener 完成
reset()清除 transcript 和队列
并发保护 prompt() 和 continue() 不允许并行调用。如果 Agent 正在运行，它们会抛出错误。正确做法是使用 steer() / followUp() 队列消息，或等待 waitForIdle()。

4. AgentMessage vs Message — 消息类型系统

pi-agent-core 使用两套消息类型：

  • Message：LLM provider 理解的标准消息（user, assistant, toolResult）
  • AgentMessage：扩展消息，包含 LLM 消息 + 自定义应用消息

4.1 AgentMessage 的定义

types.ts 
export interface CustomAgentMessages {
  // Empty by default - apps extend via declaration merging
}

export type AgentMessage = Message | CustomAgentMessages[keyof CustomAgentMessages];

通过 TypeScript 的 declaration merging，应用可以扩展自定义消息类型：

declare module "@earendil-works/pi-agent-core" {

  }
}

// 现在 AgentMessage 自动包含 ArtifactMessage 和 NotificationMessage

4.2 convertToLlm() — AgentMessage → Message 转换

Agent 内部全程使用 AgentMessage[]。只有在调用 LLM 的边界才需要转换为 Message[]。这是 convertToLlm 的职责：

agent-loop.ts — streamAssistantResponse 
async function streamAssistantResponse(...): Promise<AssistantMessage> {
  // 1. transformContext 操作的是 AgentMessage[]
  let messages = context.messages;
  if (config.transformContext) {
    messages = await config.transformContext(messages, signal);
  }

  // 2. convertToLlm 操作的是 AgentMessage[] → Message[]
  const llmMessages = await config.convertToLlm(messages);

  // 3. 构建 LLM context
  const llmContext: Context = {
    systemPrompt: context.systemPrompt,
    messages: llmMessages,
    tools: context.tools,
  };

  const response = await streamFunction(config.model, llmContext, ...);
  // ... handle streaming events
}

4.3 默认转换行为

agent.ts — defaultConvertToLlm 
function defaultConvertToLlm(messages: AgentMessage[]): Message[] {
  return messages.filter(
    (message) => message.role === "user"
      || message.role === "assistant"
      || message.role === "toolResult",
  );
}

默认实现简单地过滤掉非 LLM 消息。自定义消息类型如果 role 不是上述三者之一，会被自动排除。你也可以提供自定义 convertToLlm 来将自定义消息转换为 LLM 可理解的形式。

4.4 transformContext() — 上下文窗口管理

transformContext 在 convertToLlm 之前运行，操作的是 AgentMessage[]。适用于：

  • 裁剪旧消息以适配 context window
  • 从外部源注入上下文
  • 压缩历史对话
两个转换的顺序 transformContext(AgentMessage[]) → AgentMessage[] 先执行，然后 convertToLlm(AgentMessage[]) → Message[] 执行。这种分层设计让你可以在 AgentMessage 层面操作（包括自定义消息），再转换为 LLM 格式。

5. 工具执行模式

当 LLM 返回的 assistant 消息包含多个 tool call 时，Agent 支持两种执行策略。

5.1 sequential vs parallel

types.ts — ToolExecutionMode 
export type ToolExecutionMode = "sequential" | "parallel";

5.2 执行决策逻辑

agent-loop.ts — executeToolCalls 
async function executeToolCalls(...): Promise<ExecutedToolCallBatch> {
  const toolCalls = assistantMessage.content.filter(c => c.type === "toolCall");

  const hasSequentialToolCall = toolCalls.some(
    (tc) => currentContext.tools?.find((t) => t.name === tc.name)
      ?.executionMode === "sequential",
  );

  if (config.toolExecution === "sequential" || hasSequentialToolCall) {
    return executeToolCallsSequential(...);
  }
  return executeToolCallsParallel(...);
}

决策规则：

  1. 如果全局 toolExecution === "sequential"，所有工具顺序执行
  2. 如果某个工具声明了 executionMode: "sequential"，降级为顺序执行
  3. 否则使用并行执行

5.3 beforeToolCall / afterToolCall 钩子

每个工具调用都有两个生命周期钩子：

钩子时机能力
beforeToolCall 参数验证后，执行前 可阻止执行（{ block: true }）
afterToolCall 执行完成后 可覆盖 result.content / details / isError / terminate
types.ts — AfterToolCallResult 
export interface AfterToolCallResult {
  content?: (TextContent | ImageContent)[];  // 替换全部 content
  details?: unknown;                         // 替换全部 details
  isError?: boolean;                         // 覆盖错误标志
  terminate?: boolean;                       // 覆盖终止提示
}

6. 高级控制钩子

Agent 循环提供了四个强大的控制钩子，让你在运行时动态影响 Agent 行为。

6.1 shouldStopAfterTurn — 优雅停止

types.ts — ShouldStopAfterTurnContext 
export interface ShouldStopAfterTurnContext {
  message: AssistantMessage;
  toolResults: ToolResultMessage[];
  context: AgentContext;
  newMessages: AgentMessage[];
}

// 在 AgentLoopConfig 中
shouldStopAfterTurn?: (context: ShouldStopAfterTurnContext)
  => boolean | Promise<boolean>;

每轮结束后调用。返回 true 则 Agent 停止。这发生在 turn_end 之后、polling steering/follow-up 之前。

典型用途 检测 context window 即将溢出、用户取消了任务、或达到了最大轮次限制。

6.2 prepareNextTurn — 动态切换模型和上下文

types.ts — AgentLoopTurnUpdate 
export interface AgentLoopTurnUpdate {
  context?: AgentContext;       // 替换整个上下文
  model?: Model<any>;            // 切换模型
  thinkingLevel?: ThinkingLevel; // 切换推理级别
}

// 在 AgentLoopConfig 中
prepareNextTurn?: (context: PrepareNextTurnContext)
  => AgentLoopTurnUpdate | undefined | Promise<...>;

在 shouldStopAfterTurn 之前调用。返回 undefined 表示保持当前配置不变。

6.3 getSteeringMessages — 中途注入

在每轮 tool calls 执行完毕后调用。返回的消息会在下一轮 LLM 调用之前注入到上下文中。适用于"边跑边指导"的场景。

6.4 getFollowUpMessages — 后续消息

在 Agent 本应停止时调用（没有 tool calls、没有 steering 消息）。返回的消息会让 Agent 继续运行。适用于 post-completion follow-up。

6.5 QueueMode — 队列模式

types.ts — QueueMode 
export type QueueMode = "all" | "one-at-a-time";

控制消息队列的排水方式：

  • "all"：排水点到达时，一次性注入所有队列中的消息
  • "one-at-a-time"：每次只注入最老的一条，其余等待下一个排水点
agent.ts — PendingMessageQueue 
class PendingMessageQueue {
  private messages: AgentMessage[] = [];
  public mode: QueueMode;

  drain(): AgentMessage[] {
    if (this.mode === "all") {
      const drained = this.messages.slice();
      this.messages = [];
      return drained;
    }
    // one-at-a-time: 只取第一条
    const first = this.messages[0];
    this.messages = this.messages.slice(1);
    return [first];
  }
}

7. 事件系统

Agent 通过事件流向外部传递实时状态更新。事件是按顺序 emit 的，每个事件都携带足够的信息供 UI 渲染。

7.1 AgentEvent 联合类型

types.ts — AgentEvent 
export type AgentEvent =
  // Agent lifecycle
  | { type: "agent_start" }
  | { type: "agent_end"; messages: AgentMessage[] }

  // Turn lifecycle
  | { type: "turn_start" }
  | { type: "turn_end"; message: AgentMessage; toolResults: ToolResultMessage[] }

  // Message lifecycle
  | { type: "message_start"; message: AgentMessage }
  | { type: "message_update"; message: AgentMessage;
      assistantMessageEvent: AssistantMessageEvent }
  | { type: "message_end"; message: AgentMessage }

  // Tool execution lifecycle
  | { type: "tool_execution_start"; toolCallId: string;
      toolName: string; args: any }
  | { type: "tool_execution_update"; toolCallId: string;
      toolName: string; args: any; partialResult: any }
  | { type: "tool_execution_end"; toolCallId: string;
      toolName: string; result: any; isError: boolean };

7.2 一个 Turn 的事件时间线

message_update 只在流式传输时出现 对于 assistant 消息，message_start → 多个 message_update → message_end 构成流式传输的完整序列。user 和 toolResult 消息只产生 message_start 和 message_end。

8. Thinking Levels — 推理深度

Agent 支持六个级别的推理深度，对应不同 LLM 的 "thinking" / "reasoning" 能力。

types.ts — ThinkingLevel 
export type ThinkingLevel =
  | "off" | "minimal" | "low" | "medium" | "high" | "xhigh";
Level说明典型用途
off不使用推理简单任务，速度优先
minimal最小推理几乎不需要思考
low少量推理简单推理任务
medium中等推理平衡速度和质量
high深度推理复杂推理、多步规划
xhigh极限推理仅部分模型家族支持
模型差异 不同 provider 对 thinking level 的映射不同。例如 Claude 的 "high" 可能对应不同的 token budget than GPT 的 "high"。使用 ThinkingBudgets 可以精确控制 token 数量。xhigh 仅被部分模型家族支持，使用前应检查模型元数据。

在 Agent 中设置 thinking level：

const agent = new Agent({
  initialState: {
    thinkingLevel: "high",  // 默认 "off"
    thinkingBudgets: {
      high: 16000,   // 为 high level 指定 token budget
      medium: 8000,
    },
  },
});

通过 prepareNextTurn 可以在运行时动态切换推理深度：

const agent = new Agent({
  prepareNextTurn: async (context) => {
    // 如果工具调用失败，切换到更深的推理
    const hasErrors = context.toolResults.some(r => r.isError);
    if (hasErrors) {
      return { thinkingLevel: "high" };
    }
  },
});

9. 练习：创建最小 Agent

动手实践

下面是一个完整的、可运行的最小 Agent 示例。它使用一个 faux provider（模拟 LLM），执行一轮包含工具调用的对话。

minimal-agent.ts (完整可运行示例) 
import { Agent } from "@earendil-works/pi-agent-core";
import type { AgentEvent, AgentMessage } from "@earendil-works/pi-agent-core";

// 1. 定义一个简单的工具
const weatherTool = {
  name: "get_weather",
  label: "Get Weather",
  description: "Get the current weather for a city",
  parameters: {
    type: "object",
    properties: {
      city: { type: "string", description: "City name" },
    },
    required: ["city"],
  },
  async execute(_id: string, params: { city: string }) {
    // 模拟 API 调用
    return {
      content: [{ type: "text", text: `Weather in ${params.city}: 22°C, sunny` }],
      details: { temp: 22, condition: "sunny" },
    };
  },
};

// 2. 创建 Agent 实例
const agent = new Agent({
  initialState: {
    systemPrompt: "You are a helpful assistant.",
    model: {
      id: "faux-model",
      name: "Faux Model",
      api: "openai" as const,
      provider: "faux",
      baseUrl: "",
      reasoning: false,
      input: ["text"],
      cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
      contextWindow: 4096,
      maxTokens: 1024,
    },
    tools: [weatherTool],  // 教学示例中省略了完整的 AgentTool 类型标注
  },
  // 3. 使用 faux stream function 模拟 LLM 响应
  streamFn: async (model, context) => {
    // 模拟 LLM 返回一个 tool call
    return {
      async *[Symbol.asyncIterator]() {
        yield {
          type: "start",
          partial: {
            role: "assistant",
            content: [],
            stopReason: "toolUse",
          },
        };
        yield {
          type: "toolcall_start",
          toolCallId: "call_1",
          toolName: "get_weather",
          partial: {
            role: "assistant",
            content: [{
              type: "toolCall",
              id: "call_1",
              name: "get_weather",
              arguments: { city: "Tokyo" },
            }],
            stopReason: "toolUse",
          },
        };
        yield {
          type: "toolcall_end",
          toolCallId: "call_1",
          toolName: "get_weather",
          partial: {
            role: "assistant",
            content: [{
              type: "toolCall",
              id: "call_1",
              name: "get_weather",
              arguments: { city: "Tokyo" },
            }],
            stopReason: "toolUse",
          },
        };
      },
      async result() {
        return {
          role: "assistant",
          content: [{
            type: "toolCall",
            id: "call_1",
            name: "get_weather",
            arguments: { city: "Tokyo" },
          }],
          stopReason: "toolUse",
        };
      },
    }; // 教学示例中省略了完整的 AssistantMessageStream 类型标注
  },
});

// 4. 订阅事件
agent.subscribe((event: AgentEvent) => {
  console.log(`[${event.type}]`);
});

// 5. 发送 prompt 并等待完成

// 6. 查看结果

// Output: Messages: 3 (user + assistant + toolResult)
