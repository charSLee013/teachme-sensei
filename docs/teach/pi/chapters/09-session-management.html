Chapter 9: 会话管理 — 持久化、分支与导航

Pi 的会话系统把每次对话持久化为一个 append-only JSONL 文件，支持树状分支、标签标记和版本迁移。本章深入解析 SessionManager 的核心数据结构与 API。

Main Line

会话系统是 Pi 把“一个瞬时问答”变成“可恢复、可分支、可审计的长期工作流”的关键层。

Remember Now

这里最重要的不是 JSONL 这个文件格式本身，而是 append-only 和树状分支语义：旧记录不改，新的路径从 leaf 往外长。

9.1 为什么需要会话管理

一个 coding agent 的对话可能持续数小时甚至数天。会话管理解决了三个核心问题：

  • 持久化：崩溃恢复、跨天续聊。Pi 退出后重启可用 --session <id> 或 --continue 恢复。
  • 分支：用户想回到某条消息重新提问，而不丢失已有的对话历史。
  • 可观测性：所有交互以结构化 JSONL 存储，方便导出、分析和调试。
Info

会话文件默认存放在 ~/.pi/agent/sessions/<encoded-cwd>/ 下，文件名格式为 <timestamp>_<sessionId>.jsonl。每个项目目录对应一个独立的 session 子目录。

9.2 JSONL 存储格式

每行一个 JSON 对象。第一行永远是 SessionHeader，后续行是各种 SessionEntry。

// 一个典型的 session.jsonl 文件（每行一个 JSON）
{"type":"session","version":3,"id":"0192ef8e-...","timestamp":"2025-11-01T10:00:00.000Z","cwd":"/Users/alice/project"}
{"type":"message","id":"a1b2c3d4","parentId":null,"timestamp":"2025-11-01T10:00:01.000Z","message":{"role":"user","content":[{"type":"text","text":"帮我写一个排序函数"}]}}
{"type":"message","id":"e5f6g7h8","parentId":"a1b2c3d4","timestamp":"2025-11-01T10:00:05.000Z","message":{"role":"assistant","content":[{"type":"text","text":"好的，我来写一个快速排序："},{"type":"toolCall","name":"write","arguments":{"path":"sort.ts","content":"..."}}]}}
{"type":"message","id":"i9j0k1l2","parentId":"e5f6g7h8","timestamp":"2025-11-01T10:00:06.000Z","message":{"role":"toolResult","content":[{"type":"text","text":"File written"}],"toolCallId":"..."}}
{"type":"message","id":"m3n4o5p6","parentId":"i9j0k1l2","timestamp":"2025-11-01T10:00:07.000Z","message":{"role":"assistant","content":[{"type":"text","text":"已写入 sort.ts"}]}}
{"type":"compaction","id":"q7r8s9t0","parentId":"m3n4o5p6","timestamp":"2025-11-01T10:05:00.000Z","summary":"## Goal\n...","firstKeptEntryId":"e5f6g7h8","tokensBefore":18500}
Tip

JSONL 而非 JSON 数组：append-only 写入不需要解析整个文件，崩溃时不会损坏已有数据。读取时逐行 JSON.parse 即可。

9.3 SessionHeader 结构

每个会话文件的第一行是 header，包含会话的元数据：

export const CURRENT_SESSION_VERSION = 3;

export interface SessionHeader {
    type: "session";
    version?: number;       // v1 sessions 没有此字段
    id: string;             // UUID v7
    timestamp: string;      // ISO 8601
    cwd: string;            // 工作目录
    parentSession?: string; // fork 来源的 session 路径
}
  • version：当前固定为 3。旧版本会在加载时自动迁移（见 9.7 版本迁移）。
  • id：使用 UUID v7，按时间排序。
  • cwd：会话绑定的工作目录，影响 session 的查找和恢复。
  • parentSession：如果该会话是从另一个会话 fork 出来的，这里记录源文件路径。

9.4 SessionEntry 类型层次

每个 entry 都有 id、parentId、timestamp 三个基础字段。id 是 8 字符短 UUID（碰撞检测），parentId 指向上一个 entry，形成树结构。

export interface SessionEntryBase {
    type: string;
    id: string;
    parentId: string | null;
    timestamp: string;
}

共有 9 种 entry 类型：

// 1. 用户/助手消息
export interface SessionMessageEntry extends SessionEntryBase {
    type: "message";
    message: AgentMessage;
}

// 2. 思考级别变更
export interface ThinkingLevelChangeEntry extends SessionEntryBase {
    type: "thinking_level_change";
    thinkingLevel: string;
}

// 3. 模型变更
export interface ModelChangeEntry extends SessionEntryBase {
    type: "model_change";
    provider: string;
    modelId: string;
}

// 4. 压缩摘要
export interface CompactionEntry<T = unknown> extends SessionEntryBase {
    type: "compaction";
    summary: string;
    firstKeptEntryId: string;
    tokensBefore: number;
    details?: T;
    fromHook?: boolean;
}

// 5. 分支摘要
export interface BranchSummaryEntry<T = unknown> extends SessionEntryBase {
    type: "branch_summary";
    fromId: string;
    summary: string;
    details?: T;
    fromHook?: boolean;
}

// 6. 扩展自定义数据（不进入 LLM 上下文）
export interface CustomEntry<T = unknown> extends SessionEntryBase {
    type: "custom";
    customType: string;
    data?: T;
}

// 7. 标签
export interface LabelEntry extends SessionEntryBase {
    type: "label";
    targetId: string;
    label: string | undefined;
}

// 8. 会话元数据（如显示名称）
export interface SessionInfoEntry extends SessionEntryBase {
    type: "session_info";
    name?: string;
}

// 9. 扩展自定义消息（进入 LLM 上下文）
export interface CustomMessageEntry<T = unknown> extends SessionEntryBase {
    type: "custom_message";
    customType: string;
    content: string | (TextContent | ImageContent)[];
    details?: T;
    display: boolean;
}

// 联合类型
export type SessionEntry =
    | SessionMessageEntry
    | ThinkingLevelChangeEntry
    | ModelChangeEntry
    | CompactionEntry
    | BranchSummaryEntry
    | CustomEntry
    | CustomMessageEntry
    | LabelEntry
    | SessionInfoEntry;
CustomEntry vs CustomMessageEntry

CustomEntry 仅用于扩展持久化自己的状态，不参与 LLM 上下文。CustomMessageEntry 的内容会被 buildSessionContext() 转成 user 消息发给 LLM。用 display: false 可隐藏渲染。

9.5 会话树结构

每个 entry 通过 parentId 指向父节点，形成一棵树。leaf 指针 指向当前对话位置。

树结构带来的能力：

  • 分支（branch）：把 leaf 移到任意历史 entry，下一次 append 就会长出新的分支，旧分支不受影响。
  • 提取（createBranchedSession）：把一条路径导出为独立的 session 文件。
  • 遍历（getTree）：返回完整树，用于 TUI 中的树选择器。

SessionManager 是会话系统的核心类，提供读写、树遍历和工厂方法。

9.7 版本迁移（CURRENT_SESSION_VERSION = 3）

会话文件支持向前兼容。加载时检测到旧版本会自动迁移并覆写文件：

/** v1 -> v2: 添加 id/parentId 树结构 */
function migrateV1ToV2(entries: FileEntry[]): void {
    let prevId: string | null = null;
    for (const entry of entries) {
        if (entry.type === "session") { entry.version = 2; continue; }
        entry.id = generateId(ids);
        entry.parentId = prevId;
        prevId = entry.id;
        // 将 firstKeptEntryIndex 转换为 firstKeptEntryId
        if (entry.type === "compaction") {
            const comp = entry as CompactionEntry & { firstKeptEntryIndex?: number };
            if (typeof comp.firstKeptEntryIndex === "number") {
                const targetEntry = entries[comp.firstKeptEntryIndex];
                if (targetEntry) comp.firstKeptEntryId = targetEntry.id;
                delete comp.firstKeptEntryIndex;
            }
        }
    }
}

/** v2 -> v3: 将 hookMessage 重命名为 custom */
function migrateV2ToV3(entries: FileEntry[]): void {
    for (const entry of entries) {
        if (entry.type === "session") { entry.version = 3; continue; }
        if (entry.type === "message") {
            const msg = entry as SessionMessageEntry;
            if (msg.message.role === "hookMessage") {
                msg.message.role = "custom";
            }
        }
    }
}

function migrateToCurrentVersion(entries: FileEntry[]): boolean {
    const header = entries.find((e) => e.type === "session") as SessionHeader;
    const version = header?.version ?? 1;
    if (version >= CURRENT_SESSION_VERSION) return false;
    if (version < 2) migrateV1ToV2(entries);
    if (version < 3) migrateV2ToV3(entries);
    return true;
}
迁移是原地修改

迁移直接修改 entries 数组中的对象，然后调用 _rewriteFile() 覆写整个 JSONL 文件。因为 JSONL 是 append-only 格式，迁移是唯一需要覆写文件的场景。

9.8 练习：手动创建会话文件

Exercise

创建一个最小的合法 session.jsonl 文件，包含：

  1. 一个 v3 版本的 SessionHeader
  2. 一条 user 消息（"hello"）
  3. 一条 assistant 回复（"Hi there!"）
  4. 一个 compaction entry（summary 自拟）

然后用 SessionManager.open() 加载它，调用 buildSessionContext() 验证消息列表是否正确。

提示：id 字段可以用 crypto.randomUUID().slice(0, 8) 生成，parentId 按顺序指向上一个 entry。

9.9 Quiz

1. 会话文件为什么使用 JSONL 而不是 JSON 数组？
  • JSONL 解析更快
  • 支持 append-only 写入，崩溃时不损坏已有数据
  • JSONL 文件体积更小
  • JSON 数组不支持 tree 结构
JSONL 每行独立，追加写入时不需要解析整个文件。崩溃时只有最后一行可能损坏，其余行保持完整。
2. buildSessionContext() 遇到 compaction entry 时如何处理？
  • 跳过 compaction 之前的所有消息
  • 先放摘要，再放 firstKeptEntryId 之后保留的消息，最后放压缩后新消息
  • 把摘要追加到消息列表末尾
  • 直接返回原始消息，不做任何处理
compaction 的核心语义是：用摘要替代被压缩的消息，但保留最近的消息。所以顺序是摘要 → 保留消息 → 新消息。
3. CustomEntry 和 CustomMessageEntry 的关键区别是什么？
  • CustomEntry 只能存字符串，CustomMessageEntry 可以存图片
  • CustomMessageEntry 参与 LLM 上下文，CustomEntry 不参与
  • CustomEntry 可以删除，CustomMessageEntry 不能删除
  • 没有区别，只是历史遗留的两套接口
CustomEntry 用于扩展持久化内部状态（不进入 LLM），CustomMessageEntry 的内容会被转成 user 消息发给 LLM。
4. 当前 CURRENT_SESSION_VERSION 的值是？
  • 1
  • 2
  • 3
  • 4
当前版本为 3。v2 引入了 id/parentId 树结构，v3 将 hookMessage 角色重命名为 custom。
5. branch() 操作会修改已有 entries 吗？
  • 会，删除分支后的所有 entry
  • 不会，只移动 leaf 指针，下次 append 创建新分支
  • 会，给旧 entry 打上 "archived" 标记
  • 会，把所有 parentId 重定向到新的 leaf
SessionManager 严格 append-only。branch() 只改变 leafId 指针，已有 entries 永远不被修改。
