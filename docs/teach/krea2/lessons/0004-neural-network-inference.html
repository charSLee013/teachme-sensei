Lesson 04 / Foundation

神经网络从训练到推理

Krea 2 仓库不包含训练循环，但你必须理解训练留下了什么，推理时又做了什么，才能读懂 checkpoint 和模型输出。

前置：随机性产物：train/inference 对照表实验：tiny network

Remember now

训练是在更新参数以减少损失；推理是在固定参数下执行函数。推理代码的每一层都应能回答：参数来自哪里？输入是什么？输出如何被下一步消费？

直觉：学习规则与使用规则

学生做题和考试可以类比训练与推理。训练阶段反复看题、比较答案、修改解题规则；考试阶段规则被冻结，只把新题输入规则。Krea 2 的 safetensors 是冻结后的规则，不是“已经生成的一张图片”。

training: data -> model -> loss -> backward -> update parameters inference: prompt/noise -> frozen model -> prediction -> next state training changes theta; inference changes x

公式：参数固定的函数

训练阶段可抽象为：

θ_{k+1} = θ_k - η ∇_θ L(f_θ(x), y)

推理阶段只执行：

y = f_θ(x, c, t)

在 Krea 2 中，θ 是 MMDiT checkpoint 参数，x 是当前 image token，c 是文本 hidden states，t 是当前时间。采样循环改变 x，采样期间保持 θ。

代码：冻结模型

# inference.py
mmdit = mmdit.to(device=device, dtype=dtype).eval().requires_grad_(False)
ae = ae.to(device=device, dtype=dtype).eval().requires_grad_(False)
encoder = encoder.to(device=device, dtype=dtype).eval().requires_grad_(False)

# sampling.py
@torch.no_grad()
def sample(...):
    ...

eval() 和 requires_grad_(False) 分别影响模块行为与参数梯度；@torch.no_grad() 避免构建训练用的 autograd graph。它们不是“让模型更聪明”的开关。

实验：tiny network 的训练/推理对照

model = torch.nn.Linear(2, 1)
before = model.weight.detach().clone()
prediction = model(torch.tensor([[1., 2.]]))
assert torch.equal(before, model.weight)

# 只有 optimizer.step() 才会改变参数
loss = (prediction - 3).square().mean()
loss.backward()
torch.optim.SGD(model.parameters(), lr=.1).step()
assert not torch.equal(before, model.weight)

这个实验建立“推理不更新参数”的可观察证据。之后看到 load_state_dict(..., strict=True)，你就知道它是在检查固定规则是否完整装配。

Evidence and scope

Source：inference.py 的 meta/load/move、sampling.sample。

Teaching use：区分 checkpoint、模型状态、采样状态和梯度状态。

Scope：本课不重建 Krea 2 的训练数据和训练损失；这些是后续研究问题的背景，而非当前仓库事实。

Retrieval practice

  1. 为什么当前 sampler 更新的是 img ，同时保持模型参数固定？
  2. eval()、requires_grad_(False)、no_grad() 分别解决什么问题？
  3. 如果 checkpoint 少一个 key，为什么应该尽早失败？
分层答案

标准答案：采样求解的是固定 velocity field 下的状态轨迹，参数是 checkpoint 中冻结的 theta。eval() 切换 dropout/batchnorm 等模块行为，requires_grad 关闭参数梯度，no_grad 关闭计算图记录。缺 key 会让某些层使用未加载或不存在的参数，后续输出失去可解释性，strict load 能把问题定位在装配阶段。

常见误解：仓库可以只公开推理代码；外部 checkpoint 仍然承载训练阶段得到的参数。
