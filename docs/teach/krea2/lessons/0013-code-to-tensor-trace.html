Lesson 13 / Code Reconstruction

从代码到张量状态

本课训练研究者最重要的动作：在每个关键调用点停下来，写出变量的 shape、语义、来源和下一位消费者。

前置：完整 pipeline产物：trace sheet实验：tiny + hooks

Remember now

一行代码只有放回数据流才有意义。不要问“这行为什么这样写”而不问“它保护了哪一个接口不变量”。

直觉：给状态贴标签

调试时不要只打印数值。像机场行李一样给每个状态贴标签：来源、目的地、尺寸、重量、温度和是否可继续运输。对张量来说，就是 producer、consumer、shape、dtype、device、mask validity。

公式：接口不变量

把一个模块接口写成谓词：

Invariant(x) = shape(x) ∧ dtype(x) ∧ device(x) ∧ semantic(x)

例如 MMDiT 输入要求 image token 最后一维为 channels*patch²=64，context 最后一维为 txtdim=2560，position 最后一维为 3，mask 与 combined sequence 长度相等。测试通过明确断言保护这些不变量。

代码：最小 trace helper

def describe(name, x):
    print({
        "name": name,
        "shape": tuple(x.shape),
        "dtype": str(x.dtype),
        "device": str(x.device),
        "finite": bool(torch.isfinite(x).all()),
    })

describe("noise", noise)
describe("txt", txt)
describe("img_tokens", x)

在真实运行中，不要长期向日志输出整个 tensor。记录摘要、统计量、seed 和 step index 就足够；需要复核时保存小规模 trace artifact。

实验：故意制造三个错误

  1. 把 patch 从 2 改成 3，让最后一维与 first Linear 不匹配。
  2. 把 mask 放到 CPU、其他张量放到 CUDA。
  3. 把 float32 VAE 和 bfloat16 latent 混用。

对每个错误记录：错误在哪一层首次出现、错误信息是否足够、应该把 validation 放在哪里。研究论文的工程附录可以把这张表变成可复现性证据。

Evidence and scope

Source：sampler 状态准备、MMDiT forward。

Teaching use：将代码阅读转化为可执行的接口不变量。

Scope：trace helper 是教学工具，不是对生产日志系统的完整设计。

Retrieval practice

  1. 为什么“shape 对了”仍可能“语义错了”？
  2. 如果只记录异常消息，不记录 seed 和参数，为什么实验不可复核？
  3. 哪个错误应该在模型运行前被拦截？
分层答案

标准答案：同样的 shape 可能有不同 axis 语义，例如把 H/W 交换或把 text/image 顺序错置。没有 seed、checkpoint、steps、CFG、mu、dtype 和环境，别人无法重建状态。device、dtype、宽高和 negative prompt 数量等输入契约应在入口验证，让错误在进入深层 kernel 前出现。

常见误解：可诊断性取决于错误能否在靠近问题且保留上下文的位置出现；traceback 长度只是辅助信息。
