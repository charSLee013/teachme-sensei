Lesson 25 / Review

模拟审稿人与 objection map

审稿检查问题、证据、实现和结论之间的断点，并将质疑转成可验证项。

前置：limitations产物：objection map练习：证据定位

Remember now

先分类 objection：有效性、对照、公平性、统计、可复现性、范围、写作清晰度。分类后才能决定补实验、补解释还是缩小 claim。

直觉：审稿人是在模拟误用

一个读者可能拿你的方法放到不同分辨率、seed、硬件或 prompt 上。如果论文没有说边界，他会自然地把局部结果理解成普遍结论。好的审稿训练是在论文发布前模拟这种误读。

Objection审查的断点可能修复
只测了一个 seed统计不确定性增加 seeds、报告方差
Raw/Turbo 不公平steps/CFG/mu 或硬件不同说明协议、增加 matched comparison
指标不代表质量claim 超出 metric scope增加人工分析、缩小 claim
无法复现revision/环境/失败缺失补 manifest 和 artifact

公式： objection -> evidence -> action

每条质疑写成三元组：

O_i = (claim challenged, evidence needed, repair action)

若没有证据可以补，就不能用语气反驳；可以选择承认限制、修正结论或删除主张。

代码：建立 objection records

objections = [{
    "id": "R1",
    "claim": "Turbo is more efficient",
    "challenge": "efficiency ignores quality and memory",
    "evidence_needed": ["quality", "alignment", "latency", "memory"],
    "repair": "add matched table and scope language",
}]

把 objection 当结构化数据，避免 rebuttal 只剩情绪。论文作者应该能指出结果文件、表格行和正文句子。

实验：三角色审稿

  1. 作者：写出三条最强 claim。
  2. 怀疑者：只允许引用论文现有内容提出质疑。
  3. 编辑：判断质疑需要补实验、补定义还是缩小结论。

每轮交换角色。高价值质疑要指出可验证的缺口，并说明所需证据。

Evidence and scope

Source：paper checklist、实验记录和本课程固定毕业问题。

Teaching use：把审稿意见转成可执行的 evidence/action 对。

Scope：模拟审稿不能预测真实审稿人，只能提升论文的自审能力。

Retrieval practice

  1. 什么情况下应该补实验，什么情况下应该删 claim？
  2. 为什么“我们同意审稿人”不是完整回应？
  3. 如何区分代码 bug 与研究结果负面？
分层答案

标准答案：若质疑是关键且可控变量，应补实验；若资源/数据无法覆盖或 claim 本身超范围，应承认并缩小/删除。回应必须说明改了什么、证据在哪里、结论是否变化。代码 bug 可在修复后重跑；负面结果是修复实现后仍然观察到的现象。

常见误解：审稿意见的处理目标是让论文更准确；逐条记录接受、澄清、补证或收窄主张的决定。
