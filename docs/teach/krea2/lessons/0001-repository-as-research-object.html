把 GitHub 仓库读成研究对象

研究从准确识别对象、接口、假设和证据开始。

前置：会读 Python 文件产物：一张系统对象图建议：先不运行大模型

Remember now

仓库由一组承担不同角色的模块组成。先问它接收什么、改变什么、交给谁，以及这些行为成立的条件；随后再读具体实现。

Purpose

本课建立研究阅读的共同语言。Krea 2 仓库只有少量 Python 文件，但它压缩了一个完整生成系统：文本进入 Qwen3-VL，噪声进入 MMDiT，MMDiT 反复预测速度，VAE 将 latent 解码成图像。后面每个公式和实验都必须回到这张地图。

直觉：把仓库看成一条有交接单的流水线

文字不是直接变成图片。它先被 Qwen3-VL 变成可供主干网络读取的条件；噪声是 latent 空间里的起点；MMDiT 在多个时间点回答“当前状态应该往哪里移动”；VAE 最后把 latent 还原到像素。每次交接都要带着输入输出、shape、dtype、device 和 mask 规则，任何一个交接单缺失，后面的“模型效果”就可能只是错误被掩盖。

prompt -> text hidden states + mask noise -> image latent -> image tokens + positions \\ / -> MMDiT velocity -> Euler updates -> VAE decode -> PNG

公式：先写组合关系，再写内部细节

令 c=E(prompt) 为文本条件，X_1 为初始噪声 token，v_theta 为 MMDiT 的速度预测，时间网格为 t_0=1 > ... > t_K=0。当前 sampler 的抽象可以写成：

X_{k+1}=X_k+(t_{k+1}-t_k)v_theta(X_k,t_k,c)

最后经过 unpatchify 和 decoder：image=D(unpatchify(X_K))。这个写法的价值是把“模型一次 forward”和“采样多次更新”区分开；后面查代码时，每个符号都要找到一个 producer 和一个 consumer。

代码：从入口顺着调用关系走

先看资源装配，再看状态循环。inference.py:_pipeline 创建 autoencoder、conditioner 和 MMDiT；sampling.py:sample 负责 prompt 编码、noise、patchify、时间循环和 decode；mmdit.py:SingleStreamDiT 才是被每个时间点调用的主干。可以先用这个最小 trace 记录“谁调用谁”，不要一开始就跳进 28 个 block：

entry: inference.main
  -> _pipeline(checkpoint)
  -> sample(dit, ae, encoder, prompts)
       -> encoder(prompts)
       -> prepare(noise, txtlen, patch, mask)
       -> model(img, context, t, pos, mask)  # repeated
       -> ae.decode(latent)

这段 trace 没有声称模型已经生成了好图；它只确认了研究对象的边界和观察入口。

实验：用静态证据和 tiny contract 验证理解

本课先验证你能从入口函数和关键符号重建数据流。使用可逆的 toy contract 检查每一次“状态交接”，完整 checkpoint 留到具备对应硬件和记录协议时运行：

from pathlib import Path

source = Path("sampling.py").read_text()
assert "def prepare" in source
assert "def timesteps" in source
assert "ae.decode" in source

stages = ["prompt", "noise", "image tokens", "velocity", "decoded image"]
assert stages == ["prompt", "noise", "image tokens", "velocity", "decoded image"]

第二步打开 第 2 课 的 round-trip 实验，给 noise、txt、pos、mask 各写一个 shape contract。若你只能说“这是一个大模型”，却说不出一个 tensor 的 producer 和 consumer，实验没有通过。

对象地图

inference.py _pipeline: 资源与模型装配 main: CLI 参数与输出文件 encoder.py Qwen3VLConditioner: prompt -> hidden states + mask sampling.py prepare: latent -> image tokens + position + mask timesteps: resolution -> shifted time grid sample: encode -> Euler/CFG -> decode mmdit.py attention/RoPE/norm: 基础算子 TextFusionTransformer: 多层文本融合 SingleStreamDiT: 文本 token 与图像 token 的联合主干 autoencoder.py QwenAutoencoder: normalized latent -> pixels

研究者的四个问题

  1. 接口：每个模块的输入输出 shape、dtype、device 是什么？
  2. 表示：文本、latent、image token 和 position 分别代表什么？
  3. 过程：一次 forward 与多步 sampling 的状态如何变化？
  4. 证据：哪些结论来自代码，哪些来自官方模型配置，哪些必须由实验验证？

分层答案

提示：按 text、latent、velocity、decode 的顺序回忆。

标准答案：文本编码 -> 采样初始噪声 -> 将 latent patchify 并建立位置/mask -> MMDiT 在多个时间步预测速度并 Euler 更新 -> unpatchify 后交给 VAE decode，再转换成 PIL 图像。改 sampler 可以改时间表、CFG 和 latent 更新，但不会直接改变 Qwen 的 hidden-state 选择或 MMDiT 参数。参数量只描述容量，不说明数据流、训练目标、输入约束、评测方式和失败模式。

常见误解：VAE 不是负责“理解文字”；它只负责 latent 与像素之间的转换。
