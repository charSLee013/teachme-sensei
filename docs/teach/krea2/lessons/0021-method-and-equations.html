Lesson 21 / Paper Writing

Method 与公式写作

Method 提供足够的表示、过程和约束，让读者判断实验测量的对象；源码行号用于追溯实现。

前置：pipeline trace产物：method section练习：公式-代码对照

Remember now

每个公式都要回答“这个量从哪里来、如何更新、何时停止、怎样被测量”。漂亮的符号如果没有代码和实验锚点，只会增加歧义。

直觉：给别人一张可复核的地图

Method 让读者在没有阅读全部仓库的情况下理解系统，但不隐藏关键条件。先给组件和数据流，再给每个机制的最小公式，最后给实现选择和复杂度。

Text encoder E(prompt) -> c, mask Latent initialization -> x_1 ~ N(0,I) Packing P(x_1) -> image tokens + positions Velocity model v_theta -> cond / uncond predictions Guidance G -> guided velocity Euler integration -> x_0 VAE decode D -> image

公式：从变量定义开始

先定义：

  • c ∈ R^{B×L_t×12×2560}：选定 Qwen hidden states。
  • x_k ∈ R^{B×L_i×64}：packed image tokens。
  • t_k：从 1 到 0 的 resolution-aware time grid。
  • v_c, v_u：条件和无条件 velocity。

再写算法：

v_k = v_c + g(v_c-v_u)

x_{k+1} = x_k + (t_{k+1}-t_k)v_k

最后说明 x_0 如何 unpatchify、反归一化并送入 VAE。

代码：公式旁边放实现锚点

# Equation: v_k = v_c + g(v_c - v_u)
cond = model(img=img, context=txt, t=t, pos=pos, mask=mask)
uncond = model(img=img, context=untxt, t=t, pos=unpos, mask=unmask)
v = cond + guidance * (cond - uncond)

# Equation: x_{k+1} = x_k + Delta_t * v_k
img = img + (tprev - tcurr) * v

方法章节不要只写“we use Euler sampling”；要告诉读者时间顺序、CFG convention、steps、mu 来源和 padding 规则。

实验：公式审计

  1. 从 Method 每个变量找到一个本地代码定义。
  2. 从代码每个关键分支找到 Method 的一句说明。
  3. 用 tiny example 手算一次 Euler update。
  4. 标记尚未被实验验证的假设。

若公式、代码、实验三者不能互相指向，Method 尚未完成。

Evidence and scope

Source：sampling schedule and update、text conditioning、decode。

Teaching use：演示如何把实际实现压缩成可复核的数学描述。

Scope：公式描述当前 inference implementation，不等同于训练目标或论文中未公开的模型内部细节。

Retrieval practice

  1. 为什么 Method 必须说明 CFG 的具体 convention？
  2. 为什么把 all constants 塞进一个公式会降低可复核性？
  3. 一条公式找不到代码锚点时，应该怎么办？
分层答案

标准答案：不同库对 guidance scale 的定义可能相差 1；不写 convention 会造成数字不可比。变量、配置和算法步骤分开才能知道每项如何变化。找不到代码锚点时标为外部假设或未确认，并回到 source 或 experiment 继续验证；自行补全会把假设写成事实。

常见误解：公式越复杂越像研究；清楚、可计算、可验证才是目标。
