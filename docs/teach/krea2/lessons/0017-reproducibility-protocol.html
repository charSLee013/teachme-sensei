Lesson 17 / Reproducibility

可复现 inference protocol

可复现协议需要记录资源来源、环境、运行步骤和失败情况；seed 只是其中一个字段。

前置：metrics产物：protocol + manifest实验：复跑一致性

Remember now

一份结果只有在别人能重建其条件时才是研究证据。版本、权重 revision、prompt、seed、resolution、steps、CFG、mu、dtype、设备和输出后处理都属于结果。

直觉：菜谱不只写“煮熟”

“用 Krea 2 生成一张图”像菜谱里的“煮熟”。可复现实验需要写食材版本、设备、温度、时间和判断标准。尤其是 Krea 2 的 compile、CUDNN backend 和外部 Hugging Face 模型，会把环境差异带进输出。

artifact manifest code commit package lock + Python checkpoint path + SHA256 + revision tokenizer/VAE revision prompt set + seed set width/height/steps/CFG/mu/dtype/device metrics version + raw outputs + failures

公式：结果是条件函数

将结果写成：

R = F(code, weights, environment, prompt, seed, sampler_config)

复现目标包括让 R 相同，并让函数参数透明。跨硬件可能无法 bitwise 相同，因此需要定义可接受差异，例如指标区间、图像统计或人工判断协议。

代码：运行配置必须可序列化

manifest = {
    "git_commit": "...",
    "checkpoint": {"name": "oss_turbo", "sha256": "..."},
    "python": "3.12.x",
    "torch": "...",
    "cuda": "...",
    "prompt_file": "prompts.jsonl",
    "seeds": [0, 1, 2, 3, 4],
    "width": 1024, "height": 1024,
    "steps": 8, "cfg": 0.0, "mu": 1.15,
    "dtype": "bfloat16",
}

当前仓库没有完整 protocol writer；课程中的这段代码是 transfer artifact，不能伪装成项目已有功能。

实验：同机复跑与跨条件复跑

  1. 同一环境、同一 seed 重跑两次，检查输出 hash 和指标。
  2. 只改 seed，检查平均值/方差。
  3. 只改 Torch 或 GPU，记录 bitwise 差异和质量差异。
  4. 故意删除一个 revision 字段，要求 reviewer 判断是否还能复现。

实验结束后写一段范围声明：“本 protocol 支持在 X 环境下复现 Y；不保证在 Z 上 bitwise identical。”

Evidence and scope

Source：pyproject.toml/uv.lock、checkpoint 与加载、seed。

Teaching use：从项目的真实版本与硬件风险抽象出研究 protocol。

Scope：可复现协议要随 checkpoint、软件和硬件版本更新；它不是永久不变的保证。

Retrieval practice

  1. 为什么 checkpoint 身份还需要文件哈希与来源 revision？
  2. bitwise reproducibility 和 scientific reproducibility 有什么区别？
  3. 哪些失败必须写进 manifest？
分层答案

标准答案：同名文件可能被替换，必须记录 hash/revision。bitwise 是逐元素完全相同，scientific reproducibility 允许硬件浮点差异但要求结论在预定义范围内保持。OOM、compile error、NaN、坏图、缺失 prompt 和指标异常都应记录，因为它们改变结果范围。

常见误解：复现不是隐藏失败后展示成功样本。
