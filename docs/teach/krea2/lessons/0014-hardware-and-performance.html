编译、attention kernel、显存与性能

同一个公式可以在不同设备上表现为不同的运行结果。研究者必须把硬件前提写进实验，并将环境错误与模型错误分别诊断。

Remember now

数学公式给出可计算关系；当前实现还要求 CUDNN attention，并在若干方法上使用 TorchInductor/Triton compile。运行条件因此受硬件和软件环境约束。

直觉：同一条路有不同收费站

模型 forward 是路线，dtype、memory、kernel backend 和 compiler 是收费站。CPU 可以理解公式，却没有 CUDNN attention；没有 C compiler，第一次 torch.compile 就会在模型计算前失败；2k resolution 又会显著增加 token 和 mask 成本。

source code -> PyTorch graph -> torch.compile -> Triton/C++ kernels -> CUDA device | | | | syntax OK shape OK compiler required backend required | memory + timing

公式：显存如何随序列增长

联合序列长度近似为 L = text_len + image_tokens。如果显式构造 (B,1,L,L) mask，mask 元素数按 O(L²) 增长。1024 图像大约有 4096 image tokens，2k 图像约有 16384 image tokens，注意力计算和 mask 都会明显变重。

参数存储只是显存的一部分：memory = weights + activations + attention workspace + temporary tensors。

Retrieval practice

  1. 为什么 CPU 上 attention 公式能算但项目仍可能失败？
  2. 为什么第一次推理不适合直接作为 latency benchmark？
  3. 2k 图像为什么不是 1k 图像的“宽高各加倍”这么简单？
分层答案

标准答案：项目强制指定 CUDNN backend，并依赖 CUDA/compile 环境。第一次包含图编译和 kernel cache。2k 的空间边长加倍后，latent 面积约变四倍，image token 和部分 attention 成本按面积/序列平方增长，不能用线性直觉估计。

常见误解：“OOM”不是模型质量结论；它首先是资源配置和规模条件的观察。
