Reference / Architecture

Krea 2 Architecture Atlas

一页查清 Krea 2 推理时每个模块的输入、输出、shape、代码位置和不可随意改动的约束。

先记住

一次生成由 encoder、latent autoencoder、MMDiT 和 sampler 通过明确接口协作完成。读代码时先检查模块接口，再进入模块实现。

完整数据流

prompt / negative prompt | v Qwen3VLConditioner -> txt: (B, 512, 12, 2560), mask: (B, 512) | noise: (B, 16, H/8, W/8) -> patchify(p=2) -> img: (B, N, 64) | N=(H/16)*(W/16) +---- pos: (B, 512+N, 3), mask: (B, 512+N) ----+ v SingleStreamDiT / MMDiT -> velocity: (B, N, 64) | Euler path, unpatchify, Qwen VAE decode v RGB image: (B, 3, H, W)

模块 contract

模块输入输出代码与关键约束
Qwen3VLConditionerPython stringstxt (B,512,12,2560)、bool maskencoder.py；固定模板切片、最长 512 token、选 12 个 hidden layers。
QwenImageVAERGB image 或 latent16-channel latent，compression=8autoencoder.py；采样阶段只调用 decode。
prepare / patchify(B,16,H/8,W/8)(B,N,64)、3-axis position、masksampling.py；宽高必须对齐到 8*2=16。
TextFusionTransformertext hidden states + text mask投影前的 text representationmmdit.py；先在文本支路融合，再与 image token 合并。
SingleStreamDiTimage tokens、text context、time、pos、maskimage-token velocitymmdit.py；padding 后仍只切回原 image token 段。
Euler sampler初始 noise、velocity、时间网格最终 latent 和 imagesampling.py；每个区间做 x_next=x+dt*v，CFG 可多跑一次主干。

Raw 与 Turbo 的实验含义

checkpoint官方定位起始协议不能直接混用的解释
Raw未蒸馏基础模型，适合微调与后训练52 steps、CFG 3.5 是 README 示例；课程把 steps/CFG/mu 作为变量测量。不能把 Raw 的最佳 steps 当成 Turbo 的默认事实。
Turbo8-step distilled checkpoint，面向快速生成8 steps、CFG 0、mu=1.15、较高分辨率是 README 示例。CFG=0 和固定 mu 是蒸馏推理协议，不是所有 flow model 的普遍定律。

表格中的“官方定位”来自仓库 README；“课程把哪些变量放进实验”是教学设计，最终结论必须由实际 checkpoint 的记录支持。

读代码时的五个断点

  1. encoder 输出的最后两个轴为何分别表示“layer 选择”和“hidden feature”？
  2. 为什么 txtlen 由实际 context 决定，image 输出却从 combined sequence 的 txtlen 处切回？
  3. 为什么 model 输出 velocity，图像又在哪个阶段形成？
  4. 为什么 sampler 的 mu 同时牵涉分辨率、checkpoint 和时间网格？
  5. 为什么改变宽高可能改变 token 数、编译 shape、attention 成本和最终画面比例？
