JSON Caption 协议

Ideogram 4 的提示设计重点是把画面条件结构化：整体描述、风格、媒介、颜色、背景、对象、文字和可选位置约束被组织成 JSON caption。magic prompt 在 CLI 默认路径中承担把自然语言提示转换成结构化 caption 的编译器角色。

课程主线 Main Line

本章主线是：这个仓库把 prompt 当成结构化条件协议来对待，而不是随手写一句自然语言。字段的拆分方式，决定了画面语义、风格、颜色、元素和版式控制是怎样进入模型的。

现在记住 Remember Now

现在记住 Remember Now

仓库把 JSON caption 视为高控制度的提示协议，并配有校验器来检查字段结构、顺序、颜色和可选 bbox 等约束。

这为什么重要。 这解释了为什么课程后面谈到的布局、配色、文字元素和 magic prompt 都不是零散技巧，而是同一套结构化提示思路的不同侧面。

为什么可以相信这一点。 提示词文档、校验器行为和 CLI 的默认 prompt 路径指向的是同一个事实：系统期待的是结构化条件，而不是完全自由的散文式提示。

学习边界。 本页和其他 prompt 范式的比较只用于帮助读者理解控制方式差异，不把这种比较写成对私有训练过程或竞品内部机制的断言。

字段语义

字段表示的条件为什么重要
`high_level_description`全局语义意图。给模型一个简洁的整图目标。
`style_description.aesthetics`视觉氛围与风格形容词。把风格从对象内容里拆开表达。
`lighting`光照条件。独立控制场景的受光方式。
`photo` / `art_style`摄影或非摄影渲染模式。在校验器中约束互斥的风格分支。
`color_palette`全局或局部色彩方案。把调色控制从隐含 prose 里显式拿出来。
`background`场景背景环境。把环境语义从前景元素中拆开。
`elements[]`对象或文字实例，可选 bbox。提供元素级的构图与排版目标。

提示范式比较

范式控制形式优势代价
普通提示文本自然语言句子。写作负担低。构图、配色和排版控制仍然是隐含的。
正向提示 + 负向提示正向/反向自然语言描述。能压制不想要的概念。结构仍然偏弱，很多时候带有启发式成分。
外部控制模块prompt 加额外图像或控制条件。能提供很强的空间控制。控制信息被分散到多种工具和素材里。
Ideogram JSON caption用结构化字段描述风格、元素、文字、配色和可选 bbox。把构图和设计控制集中进一套文本协议。编写成本更高，校验器和 Magic Prompt 变得更重要。

Magic Prompt 作为编译器

在 CLI 默认路径里，普通自然语言 prompt 不会直接送进模型，而是先经由 magic prompt 后端扩展成压缩后的 JSON caption。把它理解成“把口语意图编译成结构化条件”的转换器，会比把它看成单纯润色器更准确。

请注意

如果关闭 Magic Prompt，调用者就要自己给出能通过校验器的 prompt 结构；否则系统只能退回到告警模式或直接报错。
