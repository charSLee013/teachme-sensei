Token 打包机制

Ideogram 4 的推理输入不是单一文本向量。推理 pipeline 会构造一条打包序列：左侧是 Qwen3-VL 文本 token 对应的条件特征，右侧是图像 latent token。并行的位置、分段和角色轨道一起定义了每个 token 在 Transformer 中的职责。

课程主线 Main Line

本章主线是：推理输入不是“把 prompt 喂给模型”这么简单，而是把文本特征、图像 latent token 以及角色与位置轨道一起打包成一条可供 Transformer 处理的序列。

推理阶段会先放入文本 token 对应的特征槽位，再接上图像 latent token。与此同时，位置、分段和角色三条辅助轨道一起告诉 Transformer：哪些位置是文本，哪些位置是图像，以及它们应当如何被读取。

这为什么重要。 只有看清这个打包方式，后面的 MRoPE、indicator、采样更新和输出契约才会连成同一个机制。

为什么可以相信这一点。 课程里的变量表、2D 车道图和 3D 示意图都在解释同一个实现事实：模型读到的是一条混合序列，而不是两套彼此隔离的输入。

学习边界。 这里说明的是推理时的张量组织方式，不据此推断内部训练批次如何构造。

关键对象

对象形状或取值作用
`token_ids``(B, total_seq_len)`Qwen 文本 token id；图像槽位保持为零。
`position_ids``(B, total_seq_len, 3)`文本和图像 token 共用的 MRoPE 坐标。
`segment_ids``(B, total_seq_len)`用于 block-diagonal attention 分组和 padding 隔离。
`indicator``(B, total_seq_len)`区分 LLM token 与输出图像 token。
`num_image_tokens``height/16 * width/16`VAE 压缩并经过 2x2 patch 后得到的图像 latent token 数。

这条序列先放文本，再放图像。文本位置沿着 token 顺序递增；图像位置则把二维网格坐标编码进去。角色轨道再额外标明当前槽位对应的是文本条件还是输出图像 token。

3D 示意图

这个 3D 视图是示意图，不是模型内部真的存在的几何空间。蓝色高轨表示文本位置，红色平面表示图像 token 网格；旋转它的目的，是把序列角色、网格结构和位置元数据分开看清。

文字说明

蓝色高轨表示承载文本条件特征的位置，红色平面表示带有二维网格坐标的图像 latent token。这只是为了帮助理解的示意投影，不要把它读成真实的嵌入空间。

机制结果

  • 提高分辨率会让图像 token 数量按面积增长，而不是线性增长。
  • 拉长 prompt 会增加文本序列长度，但不会改变图像网格本身。
  • 文本和图像的角色不能只靠位置猜出来，系统还会显式带上角色轨道。
  • padding 会被段轨道隔离，不会混入真实样本序列。
