Evidence Boundaries

本页定义课程中的证据分类，防止把研究主张、代码事实和工程建议混为一谈。

分类定义示例
`code_fact`本地仓库可直接阅读或静态验证。runner 中 T2I 强制 `num_frames=1`。
`paper_fact`arXiv source 明确陈述。论文称训练数据包含 70,000+ hours embodied-oriented footage。
`lineage_context`上游框架或相关研究背景。FSDP、Diffusers、SGLang 的一般生态含义。
`course_reconstruction`课程为帮助理解而重组的说明。端到端 ASCII 数据流图。
`engineering_transfer`维护、调试或测试建议。先跑 Dense T2I smoke test 再跑 MoE/refiner。
`unknown`缺少权重、GPU、训练代码或原始实验数据，无法确认。当前环境不能验证生成质量和 benchmark 分数。
`conflicting_surface`代码或文档存在可解释为不同结论的表面信号，需要继续实机或作者级证据裁决。TI2V pipeline 含 `cfg_parallel_group` 分支，但公开文档写到 TI2V runner 未启用 CFG parallel。

课程红线

学生页不得把 `paper_fact` 写成“我们已在本地验证”；不得把 `course_reconstruction` 写成“源码中的单一模块”；不得把 `engineering_transfer` 写成项目作者承诺；不得隐藏 `unknown` 或 `conflicting_surface`。

当公开文档和静态代码给出不同信号时，课程表述应优先说明冲突本身，并把结论降级为“需要实机验证”，而不是选择更乐观的一侧作为事实。
