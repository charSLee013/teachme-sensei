04 Runner Orchestration

本章把 runner 看作一个"意图 → 契约"的编译器：它把 CLI 参数解析、规范化，解决 backend/engine 的分派与回退，并把多卡并行分解成一个满足 $\text{world\_size}=D_{\text{cfg}}\times D_{\text{cp}}$ 的进程网格,最后凝结成一次 pipe(**call_kwargs) 调用。

Remember Now

  • scripts/inference.py 是薄入口(把仓库根加入 sys.path + 默认注入 --backend diffusers);真正的编排在 runner.main。
  • backend/engine 解析是一棵带回退分支的决策树:--backend sglang 不可用时 warning + 回退 diffusers;但显式 --engine sglang-native 不回退(用户明确要求 native,失败即失败)。
  • 并行拓扑是一个乘法分解:$\text{world\_size}=D_{\text{cfg}}\times D_{\text{cp}}$(CFG 分支 lane × context 序列 lane)。
  • 两组互斥约束:$D_{\text{cfg}}>1$ 与 --batch_cfg 互斥;$D_{\text{cfg}}>1$ 与 --refiner_batch_cfg 互斥——同一维度不能既切进程又切 batch。
  • 所有策略(prompt/negative/geometry/parallel/refiner)最终收敛成一个 call_kwargs 字典,传给 pipe(**call_kwargs)。

数学基础:并行拓扑的乘法分解

多卡推理的核心问题是:给定 $N$ 张 GPU(world_size),如何把它们分配到不同的并行维度?LingBot-Video 用两个正交维度:

$$ \text{world\_size} = D_{\text{cfg}} \times D_{\text{cp}} $$

其中 $D_{\text{cfg}}$ 是 CFG 并行度(把 classifier-free guidance 的 conditional/unconditional 两条分支拆到不同进程组),$D_{\text{cp}}$ 是 context parallel 并行度(把注意力的序列维切到不同进程,即 Ulysses,见 Lesson 10)。这个乘法关系意味着进程被组织成一个 $D_{\text{cfg}}\times D_{\text{cp}}$ 的二维网格:同一 CFG lane 内的进程协作完成一条 guidance 分支的序列并行注意力,不同 CFG lane 并行处理两条 guidance 分支。

每个 rank 的坐标可以写成:

$$ \text{rank} = r_{\text{cfg}}\cdot D_{\text{cp}} + r_{\text{cp}},\qquad r_{\text{cfg}}\in[0,D_{\text{cfg}}),\; r_{\text{cp}}\in[0,D_{\text{cp}}) $$

这正是 DeviceMesh(runner.py:347)构造的二维网格索引方式:先按 context lane 展开,再按 cfg lane 分组。

为什么 $D_{\text{cfg}}>1$ 与 batch_cfg 互斥?

CFG 需要同时算 conditional 和 unconditional 两条前向。有两种把它们"并行化"的方式,它们在物理上争抢同一份工作,不能叠加:

$$ \underbrace{D_{\text{cfg}}=2}_{\text{进程并行:两条分支放到两个进程组}} \quad\text{vs}\quad \underbrace{\texttt{batch\_cfg}}_{\text{batch 并行:两条分支拼进同一次前向的 batch 维}} $$

如果 $D_{\text{cfg}}=2$ 已经把两条分支分给了两个进程,再要求 batch_cfg 把它们拼进同一个 batch 就自相矛盾——同一份分支工作不能既在进程 A 又在进程 B 的 batch 里。所以 runner.py:1100-1101 直接报错。--refiner_batch_cfg 同理(1102-1104)。

数据流:参数到调用

设计选择

为什么 sglang 回退但 sglang-native 不回退? 这是"意图强度"的区分。public --backend sglang 表达的是"我想要快,用 SGLang 如果能用"——这是一个软偏好,不可用时回退 diffusers 保证任务能跑完,只打 warning。而显式 --engine sglang-native 表达的是"我明确要走 native 路径"——这是一个硬要求,通常出现在性能基准测试或调试 native adapter 时;此时无提示回退反而会掩盖问题,让用户误以为在测 native 其实在测 diffusers。所以 runner 让它失败即失败。这是一个把 CLI 表层词汇映射到"回退策略"的经典设计。
为什么把并行做成乘法分解而不是单一维度? 视频 DiT 的序列极长(见 Lesson 07,$T'\times H'\times W'$ 个 patch token),单靠一种并行难以同时利用大量 GPU。CFG 并行天然是 2 路(两条 guidance 分支),超过 2 就没意义;context parallel 可扩到任意路但通信开销随之上升。把二者相乘 $\text{world\_size}=D_{\text{cfg}}\times D_{\text{cp}}$,就能用 2 路 CFG × $k$ 路序列并行组合出 $2k$ 卡的拓扑,兼顾两种并行的收益。runner 的职责是校验这个分解合法(乘积等于 world_size、互斥约束不冲突)并构造对应的 DeviceMesh。
