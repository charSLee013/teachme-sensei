05 Diffusion Pipeline Loop

本章进入生成核心：LingBotVideoPipeline 如何把 prompt condition 和 latent diffusion 过程连接起来，以及为何选择 Rectified Flow 而非 DDPM。

Remember Now

  • Pipeline 的核心工作是把文本条件编码、latent 初始化、scheduler timesteps、Transformer 预测和 VAE decode 串成采样循环。
  • 普通 CFG 需要 conditional 与 unconditional 两次 Transformer forward；batch CFG 将两支拼成 batch。
  • 视频 latent 的时间维由 $T_{\text{latent}} = \frac{T_{\text{frames}} - 1}{4} + 1$ 决定，空间维由 height/width 除以 VAE spatial scale 决定。
  • Rectified Flow 的直线轨迹使得推理时所需的 NFE（Number of Function Evaluations）远少于 DDPM 的布朗曲线路径。
  • Pipeline 本身不理解用户自然语言；它接收 runner 传入的 prompt 字符串或预缓存 prompt embeddings。

Flow Matching 的数学基础

LingBot-Video 的 scheduler 基于 Rectified Flow（Liu et al., 2022）框架，而非传统 DDPM。其核心思想是在数据分布 $x_0$ 与噪声分布 $x_1 \sim \mathcal{N}(0,I)$ 之间定义一条直线插值路径，并学习沿该路径的速度场。

Rectified Flow 插值路径：

$$x_t = (1-t)\,x_0 + t\,x_1, \quad t \in [0,1]$$

最优速度场（直线方向）：

$$v^* = x_1 - x_0$$

训练目标（匹配直线速度）：

$$\mathcal{L}(\theta) = \mathbb{E}_{t,\,x_0,\,x_1}\,\bigl\|v_\theta(x_t,\,t) - (x_1 - x_0)\bigr\|^2$$

推理 ODE（从噪声走向数据）：

$$\frac{dx_t}{dt} = v_\theta(x_t,\,t)$$

与此对比，DDPM 的正向过程是布朗运动驱动的随机微分方程（SDE），其轨迹是随机弯曲的。DDPM 的反向过程需要小步长离散化才能稳定积分，导致典型 NFE 在 50—1000 步之间。Rectified Flow 的直线轨迹在几何上是最短路径，理论上可以用更大步长积分而不引入过大离散化误差，典型 NFE 可降至 4—20 步。

Classifier-Free Guidance（速度场线性外推）：

$$v_{\text{guided}} = v_{\text{uncond}} + w \cdot \bigl(v_{\text{cond}} - v_{\text{uncond}}\bigr)$$

Video latent 时间压缩：

$$T_{\text{latent}} = \frac{T_{\text{frames}} - 1}{4} + 1$$

注意 CFG 在 Rectified Flow 中依然是速度场的线性外推，逻辑与 DDPM 的 $\epsilon$-prediction CFG 完全平行。$w=0$ 退化为无条件生成，$w=7.5$ 是常用的强引导值。

设计选择：为什么 Flow Matching？

更少的 NFE（推理加速）

DDPM 的布朗运动轨迹曲率大，ODE 求解器必须用小步长以保证精度，典型需要 250—1000 步。Rectified Flow 的直线轨迹曲率接近零，相同精度只需 4—20 步，视频生成的推理时间直接减少一个数量级。

更稳定的训练（几何优势）

训练目标 $v^* = x_1 - x_0$ 在整个时间轴上是一个常数目标（与 $t$ 无关），不像 DDPM 的 $\epsilon$-prediction 目标在不同噪声水平下尺度差异大。这使得模型训练更稳定，梯度方差更低。

LingBot-Video 选择 FlowUniPC 作为 scheduler，是在 Rectified Flow 框架上叠加了 UniPC（Unified Predictor-Corrector）多步求解器，进一步减少步数同时提升生成质量。

采样循环结构

伪代码中 $v_{\text{cond}}$ 和 $v_{\text{uncond}}$ 是 Transformer 对速度场的预测。scheduler.step 执行一步 ODE 积分 $x_{t-\Delta t} = x_t - \Delta t \cdot v_{\text{guided}}$（简化表示，实际用 UniPC 多步预测）。

输入检查和 latent 形状

check_inputs 规定视频帧数必须是 1 或 $4n+1$（$n \geq 1$），原因是 VAE temporal stride 为 4：$T_{\text{latent}} = (T_{\text{frames}} - 1) / 4 + 1$ 需要整除。height 和 width 必须是 16 的倍数（patch size 2 $\times$ VAE spatial scale 8）。

prepare_latents 生成形状为 $[B, C, T_L, H_L, W_L]$ 的随机高斯噪声张量，其中：

$$T_L = \frac{T_{\text{frames}}-1}{4}+1, \quad H_L = \frac{H}{8}, \quad W_L = \frac{W}{8}$$

Prompt Encoding

encode_prompt 使用 processor 构造 Qwen 风格输入，再调用 text encoder 取得 hidden states。随后它裁掉 prompt template 前缀，只保留用户内容对应的 token 表示。对于 batch=1 的情况，右侧 padding 会被移除，以减少后续 DiT 处理长度——这是 LingBot-Video 针对推理效率的工程决策。

CFG 的三种执行方式

方式机制公式适用场景
普通 CFG conditional 与 unconditional 分别 forward，再线性组合。 $v_g = v_u + w(v_c - v_u)$ 最直接，易于审计和调试。
Batch CFG 将两支 prompt embeddings 和 latents 拼成 batch，一次 forward。 同上，拆分后合并 减少调度开销，需要 padding 对齐。
CFG Parallel 不同 rank 分别计算 conditional/unconditional，AllReduce 合并。 分布式实现同上 多卡布局，要求初始化分布式组。
