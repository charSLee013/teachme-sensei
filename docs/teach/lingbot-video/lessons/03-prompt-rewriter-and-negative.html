03 Prompt Rewriter And Auto Negative

本章从数学上说明 rewriter 的两阶段设计:为什么同一份 base 权重靠开关 LoRA adapter 就能扮演两个角色;以及 auto-negative 的 delete-only 剪枝为何在集合论上必须是"严格子集"约束。

Remember Now

  • rewriter 两阶段共用同一份 base VLM 权重:step1 EXPAND 关掉 LoRA(use_lora=False),step2 MAP 开启 LoRA(use_lora=True)。
  • LoRA 是低秩增量:$W' = W + \frac{\alpha}{r}BA$,其中 $B\in\mathbb{R}^{d\times r}$、$A\in\mathbb{R}^{r\times k}$、$r\ll\min(d,k)$。
  • 关 adapter 等价于令增量为零:$\Delta W=0 \Rightarrow W'=W$——这就是 disable_adapter() 的数学含义,不是加载第二个模型。
  • auto-negative 是 delete-only:输出 negative 必须满足 $N_{\text{out}}\subseteq N_{\text{default}}$(默认集合的子集),不允许新增或改写。
  • 最终 negative = 默认集合减去有正当理由删除的项:$N_{\text{final}} = N_{\text{default}}\setminus D$,且 $D\subseteq N_{\text{default}}$。

数学基础:一份权重,两个角色

初学者最容易误解的是"两阶段=两个模型"。实际上两阶段共享同一份冻结的 base VLM,差别只在于是否叠加一个低秩增量。理解 LoRA 的代数结构就理解了整个 rewriter。

LoRA 低秩分解

一个线性层的权重 $W\in\mathbb{R}^{d\times k}$ 全参数微调需要更新 $d\times k$ 个参数。LoRA 的观察是:微调引起的权重变化 $\Delta W$ 往往是低秩的,于是把它分解成两个瘦矩阵的乘积:

$$ W' = W + \Delta W = W + \frac{\alpha}{r}\,B\,A,\qquad B\in\mathbb{R}^{d\times r},\ A\in\mathbb{R}^{r\times k},\ r\ll\min(d,k) $$

其中 $r$ 是秩(rank),$\alpha$ 是缩放系数。可训练参数从 $d\cdot k$ 降到 $r(d+k)$。例如 $d=k=4096$、$r=16$:全参数是 $1.6\times10^7$,LoRA 只有 $1.3\times10^5$——约 1% 。前向传播时:

$$ h = W'x = Wx + \frac{\alpha}{r}B(Ax) $$

注意 $Ax$ 先把输入压到 $r$ 维,再由 $B$ 升回 $d$ 维,增量分支的计算量极小。

disable_adapter 的代数含义

rewriter 的 TransformersBackend.generate 里有一行决定角色切换(inference.py:66):

$$ \texttt{adapter\_ctx} = \begin{cases} \texttt{nullcontext()} & \text{if } \texttt{use\_lora=True}\\[4pt] \texttt{model.disable\_adapter()} & \text{if } \texttt{use\_lora=False} \end{cases} $$

当 disable_adapter() 生效,增量分支被旁路,等价于令 $\alpha=0$:

$$ \alpha=0 \ \Rightarrow\ \Delta W = \tfrac{0}{r}BA = 0 \ \Rightarrow\ W' = W $$

所以 step1 EXPAND 用的是纯 base VLM(通用改写能力,把短 prompt 扩成动作中心的 caption);step2 MAP 打开 LoRA 增量(专门学过"映射到 structured JSON 格式"这一窄任务)。一份权重、一次加载、靠 context manager 开关,省掉了维护两个 checkpoint 的成本。

为什么 EXPAND 不用 LoRA、MAP 才用? EXPAND 需要的是宽泛的语言理解与世界知识(把"一只猫跳上桌子"扩写成富含动作、外观、镜头的描述)——这正是 base VLM 预训练时最强的能力,叠加窄任务 LoRA 反而可能损害泛化。MAP 需要的是把自由文本严格塞进训练时的 structured schema(固定字段、JSON 合法性),这是一个格式对齐的窄任务,恰好适合 LoRA 这种小增量专门化。用一个开关切换,本质是"通才模式 vs 专才模式"的切换。

数学基础:delete-only 剪枝的集合约束

auto-negative 要解决的问题:系统有一份默认 negative prompt 集合 $N_{\text{default}}$(比如"模糊、低质量、畸变的手"等),但某些用户意图会与默认项冲突——用户想要"暗光电影感",而默认 negative 里有"黑暗、欠曝"。这时需要删掉冲突项,但绝不能让模型自由发挥新增内容,否则会引入不可控的生成偏置。

形式化:设默认集合 $N_{\text{default}}$,模型给出的剪枝建议产生删除集合 $D$。约束是:

$$ N_{\text{final}} = N_{\text{default}}\setminus D,\qquad \text{约束}\ D\subseteq N_{\text{default}} \ \Longleftrightarrow\ N_{\text{final}}\subseteq N_{\text{default}} $$

换句话说,输出永远是默认集合的子集——只减不增、只减不改。代码用三道防线强制这个不变量:

函数作用数学对应
_enforce_delete_only(auto_negative.py:153)强制 pruned ⊆ default,任何新增/改写项被剔除投影到 $N_{\text{default}}$ 的幂集 $2^{N_{\text{default}}}$
_restore_unjustified_block_deletions(211)把"无正当理由"的整块删除恢复回去收缩 $D$:删掉 $D$ 中无依据的元素
_apply_caption_pruning_hints(223)基于 caption 的确定性提示决定哪些该删用 caption 构造合法的 $D$

另外,edit() 阶段用 use_lora=False(auto_negative.py:270)——即 auto-negative 的判断跑在 zero-shot base 模型上,不叠加 rewriter 的 MAP LoRA,因为这里的任务是"判断冲突",不是"格式映射"。

两阶段重写数据流

设计选择

为什么用 structured JSON 而不是自由文本 prompt? 论文 data 章节说明训练侧用 dense structured caption(含 camera tags、prominent elements、时间戳动作等字段)。训练与推理的 prompt 分布若不一致,会造成 train-inference gap。rewriter 的两阶段本质是一个分布对齐器:把用户的 out-of-distribution 短 prompt 拉回训练时的 caption 分布。这是 caption-based 生成模型(DALL-E 3 的 recaptioning、PixArt 的 dense caption)共享的思路。
为什么 auto-negative 要 delete-only 而不是自由生成? 让 LLM 自由生成 negative prompt 看似更灵活,但会引入两个风险:(1)不可复现——同一输入产生不同 negative,破坏实验对照;(2)分布漂移——模型可能生成训练时从未见过的 negative 概念,导致 CFG 引导方向失控。把操作限制成"从一个受控默认集合里删除",就把自由度压到最小可控范围:$N_{\text{final}}$ 的搜索空间从"所有可能字符串"缩小到 $N_{\text{default}}$ 的 $2^{|N_{\text{default}}|}$ 个子集,且每个子集都由人工审核过的项组成。这是安全性与灵活性的工程折中。

证据边界与工程风险

  • LoRA 公式 $W'=W+\frac{\alpha}{r}BA$ 是 lineage_context(Hu et al. 2021 的通用 LoRA 定义);本仓库代码事实是 PeftModel.from_pretrained + disable_adapter() 的开关逻辑(inference.py:53,66),PEFT 内部确实以低秩 adapter 实现。
  • "EXPAND 用通才、MAP 用专才"的角色解读是 course_reconstruction(教学重构);代码事实只是两次 generate 调用的 use_lora 取值不同。
  • 工程风险(code_fact): parse_json 失败返回 None,而 save_caption 仍可能写出 {"caption": null}——这会把无效 prompt 传给 DiT runner。生产应 fail-fast。
  • 当前环境缺 transformers/peft,无法实跑 rewriter;剪枝纯函数 _enforce_delete_only 可离线单测。
