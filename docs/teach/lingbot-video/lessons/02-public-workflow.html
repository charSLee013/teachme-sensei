02 Public Workflow

本章从用户运行路径出发，把三种模式、prompt JSON、帧数公式和分辨率桶之间的契约讲成可计算的规则：给定 mode / duration / fps / resolution，你要能手推出送进 DiT 的张量形状 $[1, C, T', H/8, W/8]$。

Remember Now

  • 公开工作流不是把自然语言直接喂给 DiT，而是先经 rewriter 得到 structured JSON prompt。
  • 帧数不是任取的：num_frames_from_duration 把它对齐到 $4n{+}1$ 结构 —— $N=\big(\lfloor(\lfloor d\cdot\text{fps}\rfloor-1)/4\rfloor+1\big)\cdot 4+1$。
  • latent 时间长度 $T'=(N-1)/4+1$（VAE temporal scale = 4）；空间 $H'=H/8,\ W'=W/8$（spatial scale = 8）。
  • check_inputs 是硬约束：$N\in\{1\}\cup\{4n{+}1\}$，且 $H,W$ 必须是 16 的倍数。
  • --resolution 与 --ratio 必须同时给出，二者在 RESOLUTION_BUCKETS 里查表得到 $(W,H)$；prompt JSON 的 duration 覆盖命令行 --num_frames。

数学基础：从时长到张量形状

用户给的是"5 秒、1080p、16:9"这种人类语言。DiT 要的是一个精确形状的张量。中间的换算不是模糊的——它由两条硬性对齐规则决定：时间维必须 $4n{+}1$（因 VAE 在时间上 4 倍压缩），空间维必须 16 的倍数（patch + VAE 对齐）。

帧数：4n+1 对齐

num_frames_from_duration(duration, fps)（utils.py:16-18）先把时长换算成原始帧数 $\lfloor d\cdot\text{fps}\rfloor$，再向上对齐到 $4n{+}1$：

$$ N = \Big(\big\lfloor \tfrac{\lfloor d\cdot\text{fps}\rfloor-1}{4}\big\rfloor + 1\Big)\cdot 4 + 1 $$

为什么是 $4n{+}1$ 而不是 $4n$？因为 VAE 的时间压缩是"首帧独立 + 其余 4 帧一组"的因果结构：1 个 latent 帧解码回 1 个像素帧（首帧），之后每 1 个 latent 帧解码回 4 个像素帧。于是像素帧数 $N=4(T'-1)+1=4T'-3$，反解得 latent 帧数：

$$ T' = \frac{N-1}{4} + 1 $$

举例：$d=5,\ \text{fps}=24 \Rightarrow \lfloor120\rfloor$，$\lfloor(120-1)/4\rfloor+1=30$，$N=30\cdot4+1=121$ 像素帧；对应 $T'=(121-1)/4+1=31$ 个 latent 帧。T2I 是这条链的退化情形：$N=1 \Rightarrow T'=1$。

空间维与最终张量

空间上 VAE 做 8 倍压缩，同时 check_inputs（pipeline_lingbot_video.py:132-134）要求 $H,W$ 是 16 的倍数（保证 patch 与 VAE 双重对齐无余数）。合起来，送进 DiT 的 latent 张量形状是：

$$ x \in \mathbb{R}^{[\,1,\ C,\ T',\ H/8,\ W/8\,]},\qquad T'=\frac{N-1}{4}+1,\quad N\in\{1\}\cup\{4n{+}1\},\quad 16\mid H,\ 16\mid W $$

分辨率桶（RESOLUTION_BUCKETS，runner.py:235-269）

分辨率不是任意像素数，而是查表：--resolution（如 720）和 --ratio（如 16:9）共同索引出确切的 $(W,H)$。二者必须同时给（runner.py:1069-1072），否则报错。所有桶值都满足 $16\mid H,\ 16\mid W$，所以查表结果天然通过 check_inputs。注意 1080p 的高取 1088 而非 1080（$1080$ 不是 16 的倍数，$1088=16\cdot68$ 才是）。

代码数据流

设计选择

为什么把 T2I 做成"单帧视频"而不是独立图片管线？ T2I 令 $N=1 \Rightarrow T'=1$，走的仍是同一条 $[1,C,T',H/8,W/8]$ 张量与同一个 DiT。这避免维护两套代码路径：图片只是视频在时间维退化到长度 1 的特例。代价是 refiner 无法用于 T2I（refiner 面向视频时序细化训练），所以代码显式禁用。
为什么 duration（JSON）要覆盖 --num_frames（CLI）？ rewriter 产出的 structured prompt 里已经根据内容语义决定了合适时长，它比用户在命令行随手写的 --num_frames 更"懂"这条 prompt。让 JSON 优先（runner.py:1063-1064, 1077-1080）保证"内容驱动的时长"不被命令行默认值意外覆盖。这也是文档与代码曾出现表述差异之处，见 Lesson 11。

证据边界

  • 帧公式、check_inputs 约束、RESOLUTION_BUCKETS、mode 分支均为 code_fact（utils.py / pipeline_lingbot_video.py / runner.py 行号见 Evidence）。
  • "VAE 首帧独立 + 4 帧一组"的因果压缩解释属 course_reconstruction：由 $T'=(N-1)/4+1$ 与 temporal scale=4 反推的教学说明，非逐行代码断言。
  • 当前环境缺 diffusers/transformers/peft/decord/json_repair/sglang 且无权重，可验证静态结构与纯函数（如 num_frames_from_duration），不能声称端到端生成。

Retrieval Prompts

  1. 手推：$d=3.2$ 秒、$\text{fps}=30$，$N$ 是多少？对应 latent 帧数 $T'$ 是多少？写出每一步取整。
  2. 为什么像素帧数必须是 $4n{+}1$ 而不能是 $4n$？从 $T'=(N-1)/4+1$ 与 VAE 时间压缩结构解释。
  3. 1080p 的高为什么代码里是 1088 而不是 1080？用 check_inputs 的 16 倍数约束回答。
  4. 若命令行传 --num_frames 49 而 prompt JSON 里 duration=5、fps=24，最终帧数是多少？为什么？
  5. 给定 mode=ti2v、720p、9:16、$d=4$、fps=24，写出送进 DiT 的完整张量形状 $[1,C,T',H/8,W/8]$（$C$ 用符号）。
