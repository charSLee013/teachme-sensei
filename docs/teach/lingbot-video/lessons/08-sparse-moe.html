08 Sparse MoE

本章解释 LingBot-Video 的 sparse MoE 分支：从 capacity-compute decoupling 的 FLOPs 数学，到 router 的选择-门控非对称性，再到 token dispatch 的 3D 可视化。

Remember Now

  • MoE 的核心目标是解耦总参数容量与每 token 活跃计算：总参数 $\propto E$，激活计算 $\propto k$，其中 $k \ll E$。
  • router 先在 fp32 下算 affinity $\to$ 选 top-$k$ experts；选择用加 bias 的分数，门控权重用不加 bias 的原始分数——这个非对称性是代码显式保留的。
  • group-limited top-k 先按组打分（每组 top-2 之和）选出若干组，再在组内选 expert，减少通信与负载倾斜。
  • 推理实现包含 loop fallback、grouped_mm、Triton pack/restore、SGLang fused 和 FP8 五条 expert path。
  • load balancing、训练 loss 和 scaling 结果是论文事实；本地仓库主要可验证推理结构。

为什么要 MoE？Capacity-Compute Decoupling 的数学

一个 dense FFN 层对每个 token 都要过完整的前馈网络。若隐藏维 $d$、中间维 $d_{ff}$，则每 token 的 FFN FLOPs 约为：

Dense FFN 每 token 计算量：

$$C_{\text{dense}} \approx 2 \cdot 2 \cdot d \cdot d_{ff} = 4\,d\,d_{ff}$$

（两个线性层 gate/up 与 down，每个矩阵乘约 $2\,d\,d_{ff}$ FLOPs。）

MoE 用 $E$ 个独立 expert（每个是一个 FFN）替换单个 FFN，但每个 token 只激活其中 $k$ 个。于是：

MoE 参数量 vs 激活计算量：

$$P_{\text{MoE}} \approx E \cdot P_{\text{expert}}, \qquad C_{\text{MoE}} \approx k \cdot C_{\text{expert}}$$

参数容量随 $E$ 线性增长，而每 token 计算只随 $k$ 增长。当 $E=64, k=8$ 时，容量是 dense 的 $64\times$，计算却只有 $8\times$（相对单 expert）。

为什么这对视频生成尤其重要

视频 DiT 的序列长度 $N = N_{\text{video}} + N_{\text{text}}$ 极长（一段 5 秒 720p 视频可达数万 token）。attention 已是 $O(N^2)$ 瓶颈，若再堆 dense FFN 容量，计算量不可承受。MoE 让模型在几乎不增加每 token FLOPs 的前提下扩大知识容量——这正是论文 "capacity-compute decoupling" 的论点。总激活参数（例如 30B-A3B 中的 A3B）决定推理成本，总参数（30B）决定模型能力上限。

Router 的数学：选择与门控的非对称性

router 的 forward（LingBotVideoRouter.forward）严格保留了一个容易被忽略但至关重要的非对称设计。以下每一步都对应源码：

1. fp32 affinity logits（禁用 autocast 保证数值稳定）：

$$\ell = W_r\, x, \quad W_r \in \mathbb{R}^{E \times d}$$

2. 归一化为分数（sigmoid 或 softmax）：

$$s = \sigma(\ell) \quad \text{或} \quad s = \text{softmax}(\ell)$$

3. 选择用「加 bias」的分数：

$$s_{\text{choice}} = s + b_e \quad (b_e = \text{e\_score\_correction\_bias})$$

4. 选出 top-$k$ 索引：

$$\mathcal{I} = \text{TopK}(s_{\text{choice}},\, k)$$

5. 门控权重从「不加 bias」的原始分数 gather：

$$g = s[\mathcal{I}] \quad \text{(注意：是 } s \text{ 而非 } s_{\text{choice}}\text{)}$$

6. 可选归一化并缩放：

$$g \leftarrow \frac{g}{\sum g + \varepsilon} \cdot \text{route\_scale}$$
为什么 bias 只进选择、不进门控？

correction bias $b_e$ 的唯一职责是均衡负载：训练中若某 expert 被过度选择，就调低它的 $b_e$，让它在下一批中不易被选。但这是一个「路由策略」的调节量，不代表该 expert 对当前 token 的真实亲和度。如果把 $b_e$ 也乘进最终输出的门控权重 $g$，就会用「负载均衡的人为偏置」污染「专家贡献的加权和」，破坏梯度估计的无偏性。因此代码从 $s$（无 bias）而非 $s_{\text{choice}}$（有 bias）gather 门控权重。这一设计与 DeepSeek-V3 的 auxiliary-loss-free load balancing 思路一致。

Group-Limited Top-K

朴素 top-$k$ 在全部 $E$ 个 expert 上选择。group-limited 版本先把 expert 分成 $n_{\text{group}}$ 组，做两级选择：

组打分（每组取 top-2 分数之和）：

$$\text{group\_score}_j = \sum_{\text{top-2}} s_{\text{choice}}[\text{group}_j]$$

选出 topk_group 个组，组外 expert 置 $-\infty$，再在存活 expert 上取 top-$k$：

$$\mathcal{I} = \text{TopK}\bigl(\text{mask}_{-\infty}(s_{\text{choice}}),\, k\bigr)$$
为什么分组？

在 expert 并行（EP）部署中，不同组的 expert 往往放在不同设备上。限制每个 token 只路由到少数几个组，就把 all-to-all 通信限制在少数设备之间，减少跨设备通信量。同时两级选择降低了「一个 token 的 top-k 分散到所有组」的概率，让负载更集中可控。这是 DeepSeek-MoE 系列引入的技术。

证据边界

论文讨论 MoE recipe、expert scale、fine-grained routing 和 speed benchmark；这些是论文事实。当前本地环境缺 SGLang 和 diffusers，且没有权重，所以课程不能声称已经验证 MoE 速度或生成质量。上文 FLOPs 估算是数量级分析，非实测。
