10 Serving, FSDP, And SGLang

本章从通信复杂度的角度解释三种正交并行维度——FSDP 切参数、Context Parallel 切序列、CFG Parallel 切 guidance 分支——它们分别付出什么通信代价，以及为什么视频 DiT 需要把它们组合起来。

Remember Now

  • 三个正交维度切的对象不同：FSDP 切参数张量（每卡存 $\approx P/N_{\text{fsdp}}$），Context Parallel 切序列（每卡算 $\approx N_{\text{seq}}/N_{\text{cp}}$ token），CFG Parallel 切 guidance 两条分支。
  • FSDP 用 all-gather 在前向时临时拼回完整权重，通信量随参数量走；Ulysses（CP）用 all-to-all 交换 head/序列，通信量 $O(N_{\text{seq}}\cdot d/N_{\text{cp}})$。
  • world_size = cfg_parallel_degree × context_parallel_degree（见 Lesson 04）；FSDP 与它们复合，切的是另一个维度（参数）而非进程网格。
  • FSDP 在模型构造之后才包装（fully_shard），因此不能消除加载阶段的 host memory 峰值。
  • --backend sglang 不是外部 OpenAI-style server 客户端，而是本地 native adapter + MoE fused kernel；不可用时静默回退 diffusers。

数学基础：三种切分与通信代价

大模型多卡的核心矛盾：单卡放不下（内存墙）或算不快（算力墙）。不同并行维度针对不同的墙，通信代价也不同。理解"切什么、换来什么、代价是什么"是部署决策的基础。

维度一：FSDP —— 切参数

Fully Sharded Data Parallel 把每个参数张量按 $N_{\text{fsdp}}$ 卡分片，每卡常驻只有一份分片：

$$ \text{每卡常驻参数} \approx \frac{P}{N_{\text{fsdp}}},\qquad \text{前向/反向时 all-gather 临时拼回完整层参数} $$

代价是每进入一层就要 all-gather 该层的完整权重（用完即弃），通信量正比于参数量。LingBot-Video 的 apply_fsdp_inference 遍历 transformer block，用 PyTorch composable FSDP 的 fully_shard 逐 block 包装。关键细节：它按参数 dtype 分组，把主 dtype作为 sharded 主体，dtype 不同的参数（如保留 FP32 的 norm/router/scale-shift，见 Lesson 08）放进 ignored_params 不参与分片。

维度二：Context Parallel（Ulysses）—— 切序列

视频 DiT 的序列极长：$N_{\text{seq}}=T'\times H'\times W'$ 个 patch token（见 Lesson 07），单卡注意力算不动。Context Parallel 把序列切到 $N_{\text{cp}}$ 卡，每卡只算一段：

$$ \text{每卡序列长度} \approx \frac{N_{\text{seq}}}{N_{\text{cp}}} $$

但注意力需要每个 query 看到所有 key，序列切开后就必须通信。DeepSpeed-Ulysses 的做法是在注意力前后各做一次 all-to-all：把"按序列切"转成"按 head 切"，算完再转回来。其通信量为：

$$ \text{Ulysses all-to-all} \sim O\!\left(\frac{N_{\text{seq}}\cdot d}{N_{\text{cp}}}\right) $$

相比朴素的 all-gather 全序列（$O(N_{\text{seq}}\cdot d)$），all-to-all 的通信量随 $N_{\text{cp}}$ 下降，这是 Ulysses 对长序列友好的关键。runner.py 在 913–960 行构造 context parallel mesh 并在 transformer 上挂 context parallel plan。

维度三：CFG Parallel —— 切 guidance 分支

Classifier-free guidance 每步要算 conditional 与 unconditional 两条前向。CFG Parallel 把这两条分到不同进程组并行算，$D_{\text{cfg}}\le 2$（分支只有两条，超过 2 无意义）。它与 CP 正交相乘构成进程网格：

$$ \text{world\_size} = D_{\text{cfg}}\times D_{\text{cp}},\qquad D_{\text{cfg}}\in\{1,2\} $$

FSDP 则是第三个正交轴：它切的是参数，可以叠加在上面这张网格的每个进程上（每个进程只存参数分片）。所以完整的部署拓扑是"参数分片 × guidance 分支 × 序列切片"三重复合。

后端选择与数据流

设计选择

为什么要三维复合，而不是单靠一种并行？ 每种并行都有天花板：CFG 只有 2 路（分支数固定）；FSDP 降内存但每层 all-gather 通信随参数量涨，扩太多卡通信占比过高；CP 可扩到很多卡但 all-to-all 通信随之上升。视频 DiT 同时面临"参数大 + 序列长 + 要 CFG"三重压力，单一维度都顶不住。三者正交复合，才能在给定 GPU 预算下把每一维都控制在高效区间——这是大规模视频生成部署的典型组合拳。
为什么 FSDP inference 不能省加载阶段的内存峰值？ fully_shard 是在模型已经构造完成之后才包装、切分参数的。也就是说，模型对象必须先在 host/device 上被完整实例化（峰值内存 = 完整模型），然后才分片。真正的 meta-init / 流式加载才能避免这个峰值，而 FSDP inference 不做这件事。教学结论：FSDP 降的是稳态驻留内存，不是加载瞬时峰值。

证据边界

  • TI2V + CFG parallel 是 conflicting_surface：i2v pipeline 代码里有 cfg_parallel_group 分支，但 docs/en/dit_inference.md 明确写 TI2V runner 未启用 CFG parallel。课程不把它列为已确认组合，需实机拓扑测试。
  • 通信量公式（$O(N_{\text{seq}}d/N_{\text{cp}})$ 等）是数量级分析，非本环境实测——当前环境缺 sglang/多 GPU，无法跑分布式验证。
  • --backend sglang 不是外部 server 客户端路径；这是 paper_fact（serving infrastructure）+ code_fact（native adapter）的组合，非"已端到端部署验证"。
  • 竞品方案（Megatron TP / Ring Attention / ZeRO）为 lineage_context，用于定位谱系。
