09 Refiner

本章从数学上解释 LingBot-Video 的二阶段细化：为什么 refiner 不从纯噪声 $t=1$ 出发，而是从低噪声阈值 $t_{\text{thresh}}$ 起步；sigma 调度如何用 flow-shift 把采样步数集中到细节修复区间。

Remember Now

  • Refiner 只支持视频模式（t2v/ti2v），T2I 禁用；且 refiner 内部始终以 mode="t2v" 运行（runner.py:840）。
  • 起点不是纯噪声，而是 degraded latent 加少量噪声：$x_{t_{\text{thresh}}}=(1-t_{\text{thresh}})\,x_{\text{up}}+t_{\text{thresh}}\,\varepsilon$。
  • sigma 调度用 flow-shift 变换 $\sigma'=\dfrac{s\,\sigma}{1+(s-1)\sigma}$，$s>1$ 把更多步数推向高噪声端。
  • refiner 只取 $\sigma \le t_{\text{thresh}}$ 的那一段调度，且强制首元素恰为 $t_{\text{thresh}}$，把整段容量投入到 $[t_{\text{thresh}}\to 0]$ 的细节恢复。
  • validate_refiner_sigmas 是硬校验：1D、非空、有限、$\in[0,1]$、严格降序、首元素 $==t_{\text{thresh}}$。

数学基础：低噪声起点

标准视频生成从纯噪声 $t=1$（即 $x_1=\varepsilon$）一路去噪到 $t=0$（干净 latent $x_0$）。refiner 的任务不同：它已经有一段 base 输出 $x_{\text{up}}$（低分辨率、被上采样、细节退化的 latent），要做的是局部细节恢复，不是从零生成内容。所以它没必要走完整噪声区间。

在 Rectified Flow 的直线插值框架下（见 Lesson 05），任意时刻 $t$ 的带噪 latent 是干净信号与噪声的线性组合：

$$ x_t = (1-t)\,x_0 + t\,\varepsilon,\qquad \varepsilon\sim\mathcal{N}(0,I) $$

refiner 把 degraded base latent $x_{\text{up}}$ 当作"近似干净信号"，注入到 $t=t_{\text{thresh}}$ 这个低噪声点上，这正是 prepare_refiner_latent 做的：

$$ x_{t_{\text{thresh}}} = (1-t_{\text{thresh}})\,x_{\text{up}} + t_{\text{thresh}}\,\varepsilon $$

直观理解：$t_{\text{thresh}}$ 越小，起点越接近 $x_{\text{up}}$（保留更多 base 结构，改动更保守）；$t_{\text{thresh}}$ 越大，注入的噪声越多，refiner 有更大自由度重写细节（但可能偏离 base）。这是一个"忠实度 vs 修复力度"的旋钮。

sigma 调度：flow-shift 变换

compute_refiner_sigmas 先在 $[\sigma_{\max},\sigma_{\min}]$ 上均匀取 $N{+}1$ 个点并丢掉末点，再对每个 $\sigma$ 施加 flow-shift：

$$ \sigma_{\text{base}} = \text{linspace}(\sigma_{\max},\sigma_{\min},N{+}1)[:-1],\qquad \sigma' = \frac{s\cdot\sigma_{\text{base}}}{1+(s-1)\cdot\sigma_{\text{base}}} $$

其中 $s$ 是 shift 参数。当 $s=1$ 时 $\sigma'=\sigma$（恒等）；当 $s>1$ 时，函数把中间值往上抬，等价于把更多采样步数分配到高噪声区间——这对视频这种高维信号有利，因为高噪声段的结构决策更需要密集步数。

$$ \text{保留 } \sigma' \le t_{\text{thresh}}+\epsilon,\qquad \text{强制 } \sigma'_0 = t_{\text{thresh}},\qquad \text{(可选) tail\_steps: 线性延伸至 } \sigma_{\min} $$

过滤掉大于阈值的部分后，refiner 的整条调度都落在 $[t_{\text{thresh}}\to 0]$ 窗口内。首元素被钉成恰好 $t_{\text{thresh}}$，与 prepare_refiner_latent 的起点严格对齐——这就是 validate_refiner_sigmas 检查"首元素 $==t_{\text{thresh}}$"的原因。

Refiner 运行路径

设计选择

为什么 refiner 不从纯噪声开始？ 从 $t=1$ 重新生成会丢掉 base 阶段已经确定的全局结构（构图、运动、主体），既浪费算力又可能引入不一致。从 $t_{\text{thresh}}$ 出发意味着"信任 base 的低频结构，只重做高频细节"。数学上，起点 $x_{t_{\text{thresh}}}$ 里 $(1-t_{\text{thresh}})$ 比例的信号来自 $x_{\text{up}}$，天然携带 base 结构。这是 cascaded / SDEdit 类方法的核心思想：用噪声强度控制"改多少"。
flow-shift 为什么偏向高噪声端？ 在 Rectified Flow / EDM 类调度里，高噪声段决定大尺度结构，低噪声段只做精修。视频的 latent 维度极高，结构决策一旦错了后面救不回来，所以把步数密度往高噪声端倾斜（$s>1$）能提升结构稳定性。$\sigma'=s\sigma/(1{+}(s{-}1)\sigma)$ 是一个单调、把 $[0,1]$ 映回 $[0,1]$、在中段上凸的变换——$s$ 越大，弯曲越强。
