01 System Map

本章建立 LingBot-Video 的系统边界：DiT 消费的 prompt 格式、rewriter 存在的原因、三大子系统如何通过明确的数据契约衔接，并在此确立整本课程贯穿使用的张量符号体系。

Remember Now

  • DiT 消费的是 structured JSON caption（含镜头参数与逐元素时间戳动作），不是自然语言短句；rewriter 是训练-推理分布对齐的桥梁。
  • 三大子系统与数据契约：Prompt 处理 →(structured JSON)→ DiT 推理 →(latents)→ 部署与细化。边上流动的是有明确类型和 shape 的张量，不是模糊的"数据"。
  • 贯穿全书的符号：像素张量 $[B,3,F,H,W]$，latent 张量 $[B,C,T',H',W']$，其中时间维压缩 $T'=\frac{F-1}{4}+1$、空间维压缩 $H'=H/8,\ W'=W/8$。
  • 三种模式 t2i/t2v/ti2v 共享同一套 DiT 与 VAE，仅输入契约不同；T2I 强制 $F=1$。
  • 本仓库是公开推理运行时；完整训练管线、post-training、benchmark 评测是论文事实，不在仓库内——不要把二者混为一谈。

一个问题：训练时的 prompt 是什么格式？

训练视频生成模型时，需要为每段视频配一条 prompt 作为条件信号。这里有一个根本性的设计选择：

  • 选项 A：自然语言描述 — 例如 "an ice cube falling into whiskey"，简洁，人类易写
  • 选项 B：结构化 JSON caption — 包含摄像机参数、元素列表、逐帧时间戳动作，精确但复杂

LingBot-Video 选择了选项 B。原因在于 embodied 应用对时空控制有精确需求：镜头语言（焦段、构图、光线）和逐元素的时间戳动作（"[0.0s–0.8s] falls from above"）共同约束 DiT 学习准确的时空关联，而这种信息在普通自然语言短句中无法表达。

这带来了推理时的挑战：用户不会手写 140 行 JSON，但 DiT 在训练时从未见过普通短句。必须有一个桥梁将用户输入转换为 DiT 期望的格式。

Structured JSON Prompt 的真实结构

以下是 assets/cases/ti2v/example_1/prompt.json 的缩略版（141行完整文件，此处省略部分元素字段）：

{
  "caption": {
    "camera_info": {
      "color": "Warm",
      "frame_size": "Close Up",
      "shot_type_angle": "Eye level",
      "lens_size": "Macro",
      "composition": "Center",
      "lighting": "Soft light",
      "lighting_type": "Studio lighting"
    },
    "prominent_elements": [
      {
        "name": "transparent ice cube",
        "actions": [
          { "timestamp": "[0.0s - 0.8s]",
            "action": "falls from above the rim, rotating slightly as it descends" },
          { "timestamp": "[0.8s - 1.6s]",
            "action": "strikes the surface of the liquid, creating a circular depression and initial splash" },
          { "timestamp": "[1.6s - 2.8s]",
            "action": "partially submerges, generating splashes and bubbles" },
          { "timestamp": "[2.8s - 4.0s]",
            "action": "bobs slightly up and down and rotates within the liquid" },
          { "timestamp": "[4.0s - 5.0s]",
            "action": "floats gently in the liquid, slowly coming to a stop" }
        ]
      }
      /* ... 还有 3 个 prominent_elements（whiskey glass、amber liquid、dark wooden table）
             以及每个元素的 location、shape_and_color、texture、appearance_details 等字段 */
    ]
  },
  "duration": 5
}

注意 prominent_elements[0].actions：每个条目包含一个时间戳区间和一条动作描述。这让 DiT 在训练时学会"在 [0.0s–0.8s] 时冰块下落"而不只是"冰块在某处落下"。时间戳是时空对齐信息的核心载体。

问题重述

有了上面的背景，推理时的问题可以精确描述：

  • 用户输入："an ice cube falling into whiskey"（一句话）
  • DiT 需要：上面所示的 141 行结构化 JSON（含时间戳级 actions 数组）

中间的桥梁就是接下来要介绍的 rewriter 系统。

两阶段 Rewriter

为什么是两阶段，而不是一步直接生成 JSON？有两个核心原因：

  1. JSON 是 domain-specific 格式：将用户短句直接映射到 141 行 JSON 需要大量领域专用知识（时间戳分割逻辑、摄像机参数词表），这是高度特化的技能，不适合用通用 VLM 一步完成。
  2. 扩展和结构化是两种不同能力：先将 "an ice cube falling into whiskey" 扩展为富有细节的散文描述（利用 VLM 的通用视觉知识），再将散文映射到 JSON schema（利用 LoRA 学到的结构化输出能力），比一步跳转更稳定。

两阶段的分工：

  • Step 1（EXPAND）：使用 base VLM，不加载 LoRA。输入用户短句，输出富含细节的散文描述，例如 "A transparent ice cube suspended above a glass of amber whiskey, catching studio light..."
  • Step 2（MAP）：使用 base VLM + LoRA adapter。输入 step1 的散文，输出符合训练 schema 的结构化 JSON caption。LoRA 只在此阶段激活，专门负责格式映射。

这种设计使 LoRA 的学习目标单一（只需学格式映射），base VLM 的通用知识在 step1 得到充分利用。第 03 章会给出 LoRA 的低秩分解数学与 disable_adapter 的等价含义。

数学基础：贯穿全书的张量符号

在读任何一章之前，先建立整本课程共用的符号体系。视频在两个空间中流动：像素空间（人能看的 RGB 帧）与 latent 空间（VAE 压缩后、DiT 真正操作的低维表示）。二者由一个 3D VAE 相连，压缩比是固定的。

$$ \underbrace{x_{\text{pixel}} \in \mathbb{R}^{B\times 3\times F\times H\times W}}_{\text{像素视频}} \;\xrightarrow{\ \text{VAE.encode}\ }\; \underbrace{z \in \mathbb{R}^{B\times C\times T'\times H'\times W'}}_{\text{latent 视频}} $$

其中 $F$ 是像素帧数、$H\times W$ 是像素分辨率。VAE 在时间维压缩 $4\times$、在空间两维各压缩 $8\times$，因此：

$$ T' = \frac{F-1}{4}+1,\qquad H' = \frac{H}{8},\qquad W' = \frac{W}{8},\qquad C = \text{latent 通道数} $$

时间维的 $\frac{F-1}{4}+1$ 不是普通除法：因果式的 3D VAE 单独保留第一帧、再对其余帧按 4 分组，所以像素帧数必须满足 $F=4n+1$ 才能整除。这正是第 02 章帧公式 num_frames_from_duration 强制对齐 $4n+1$ 的根本原因。举例：$F=121$（5 秒 @ 24fps 对齐后）$\Rightarrow T'=\frac{120}{4}+1=31$ 个 latent 帧。

为什么 DiT 在 latent 空间而非像素空间工作？ 一段 $121\times 720\times 1280$ 的 RGB 视频有约 $3.3\times 10^8$ 个像素值；经 VAE 压到 latent 后是 $C\times 31\times 90\times 160$，元素数下降约两个数量级。DiT 的自注意力对序列长度是 $O(N^2)$（$N=T'H'W'$ 个 patch token，见 Lesson 07），在 latent 空间做能把 $N$ 压小、让长视频的注意力可承受。这就是"latent diffusion"范式的核心动机——本课程后续所有 shape 都以 latent 张量为准。

关键边界

当前仓库没有完整训练循环、数据引擎或公开 benchmark 复现实验。它包含的是可读、可运行的推理和部署路径。因此，本课程先讲公开 runtime，再把论文中的训练与评测内容作为机制背景和研究上下文。上文张量压缩比（temporal 4×、spatial 8×）来自 pipeline 的 vae_scale_factor_temporal=4 / vae_scale_factor_spatial=8（code_fact）；论文的训练与 scaling 描述属 paper_fact，不等于本地已验证。
