# Teachme Sensei Agent Guide

这份指南适用于仓库内的所有 agent。目标是让每次修改都从根因出发，落在正确的责任层，形成读者可用、可验证、可持续生成的结果。

## 项目使命

- 维护一个面向工程师、模型研究者和贡献者的静态中文教学站。
- 将事实、证据、实验条件和结论边界表达得清楚、直接、可追溯。
- 让提交到 `docs/` 的产物可以稳定阅读、解析、部署和复现。
- 让 adapter、生成器、同步规则和确定性配置承担长期正确性。

## 开始任务

每次编辑前完成下面的顺序，并在 commentary 中用一句话说明当前的根因检查：

1. 说明用户目标、受影响的读者和可观察的成功标准。
2. 找到目标内容或产物的 canonical owner：手写页面、adapter、sanitizer、共享 CSS、manifest 或 workflow。
3. 读取最小必要的调用链、数据流和相邻测试，确认现象与原因之间的关系。
4. 提出至少两个可能解释，使用代码阅读、静态检查或小型实验排除错误解释。
5. 思考输入漂移、重复生成、窄屏布局、缺失资源、脚本状态和失败恢复等边界。
6. 选择最小的持久改动，并写下每个改动对应的验证方式。

## 责任层

按下面的层次放置修复：

- 内容层负责事实、引用、证据范围、实验条件和读者措辞。
- 生成层负责 HTML 结构、链接、资源路径、metadata、可访问性和安全属性。
- 样式层负责布局约束、可读性、窄屏适配、滚动区域和稳定尺寸。
- 交互层负责明确的用户动作、状态变化、键盘操作和渐进增强。
- 开发验证层负责 schema、静态结构、链接、资源、adapter、单元测试和人工页面探索。
- 发布层负责使用已验证的 `docs/`，上传 Pages artifact，并完成部署。

生成或同步页面的持久修复进入 adapter、sanitizer、共享样式或确定性配置，然后重新生成提交产物。手写页面保留在页面自身的语义结构中。每个改动都与 canonical owner 保持一致。

## 页面措辞

公开页面采用直接、积极、证据一致的表达：

- 使用“主体 + 行为 + 条件/结果”的句式。
- 用独立句子说明比较对象各自测量的内容、适用条件和差异。
- 用明确范围描述结论边界：分辨率、硬件、版本、指标、seed、数据来源和运行命令。
- 将信息分成三类：已验证事实、基于证据的推断、待执行的实验计划。
- 让标题、段落、表格和链接共同服务于学习路径。
- 把限制写成可执行条件和下一步记录项。
- 只把读者需要的课程信息放进公开页面；工程治理、审查记录、source matrix 和内部 proof 进入仓库控制面。

合格表达示例：

- “本课程解释 Krea 2 的数据流、实验协议和证据边界。”
- “当前证据支持以下硬件与软件组合。”
- “X 衡量文本对齐，Y 衡量人工偏好。”
- “该结论适用于以下分辨率、版本和采样设置。”
- “下一步实验将记录环境、命令、日志和测量结果。”
- “本页记录代码入口、数据流和可复现实验步骤。”

## HTML、CSS 和 JavaScript

- 使用结构化 HTML 语义，让页面在浏览器和解析器中都保持稳定。
- 让每个页面拥有清晰的标题层级、`main`、唯一 `h1`、可访问链接和稳定的本地资源路径。
- 为固定格式组件设置稳定尺寸、可滚动区域、键盘焦点和可读名称。
- 让 CSS 在桌面和窄屏下保持内容可读，长 URL、代码、表格和公式拥有明确的换行或滚动策略。
- 让 JavaScript 只承担明确的交互；阅读、导航和重要信息保持原生 HTML 可用。
- 在开发阶段直接操作受影响页面，检查状态变化、键盘路径和失败反馈。
- 按页面模板和高风险内容选择人工探索样本，并把发现的结构问题修入 canonical owner。

## 测试分层

先运行最接近改动责任层的测试，再扩大范围：

1. 纯内容或配置：JSON/schema/文案计数、可见文本和固定字段检查。
2. 生成器或 adapter：输入布局、输出集合、路径、hash、HTML 结构和恶意输入 fixture。
3. 静态站点：全量 route、链接、fragment、资源、metadata、脚本策略和解析器错误。
4. 人工页面探索：模板代表页、窄屏布局、真实交互、CSS 加载和可访问性。
5. 发布契约：单一 workflow、单一 deploy job、单一 Pages artifact 和固定部署顺序。

日常开发使用快速静态检查、单元测试和模板级人工探索，让反馈尽早返回给作者。

## 发布与同步

- 已提交的 `docs/` 是公开发布真相。
- 外部教学源包通过显式 source root、source lock、staging 和事务流程进入生成链。
- CI 直接验证提交产物；源包同步属于明确的本地操作。
- Pages 发布链保持为“checkout -> configure Pages -> upload docs -> deploy”。
- workflow 验证器检查发布安全属性、权限、artifact 唯一性和四步顺序。
- 发布 workflow 只部署已提交的 `docs/`；开发验证通过 `npm test` 在提交前完成。

## 修改纪律

- 只修改完成用户目标所需的文件和行。
- 保留现有风格、命名和模块边界。
- 看到无关问题时记录位置和风险，保持当前任务边界清晰。
- 让每个新抽象消除真实重复或表达稳定契约。
- 先写失败测试或可复现样例，再写实现；随后运行同一验证确认修复。
- 生成文件发生变化时，同时检查生成规则、输入锁和提交产物的对应关系。
- 记录假设、取舍、命令、结果和剩余风险。

## 完成标准

交付前确认：

- 每个改动都能追溯到用户目标和 canonical owner。
- 内容事实、证据范围和页面措辞经过人工阅读。
- HTML 可由结构化解析器稳定读取。
- CSS 在目标模板和窄屏下保持布局稳定。
- JavaScript 交互经过开发期人工操作和状态检查。
- 相关静态检查与单元测试全部通过，目标页面完成人工阅读和交互检查。
- `git diff --check` 通过，生成产物和配置文件保持同步。
- 最终报告准确列出完成项、证据、运行命令和剩余风险。
